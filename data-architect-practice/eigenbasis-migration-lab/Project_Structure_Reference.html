01Is there actually a "standard"?

Short answer: there's a strong de facto convention, not a single canonical one.

Unlike a Python package (where the layout is essentially fixed), data projects don't have one official structure. But most modern teams converge on the same shape anyway, driven by two forces: dbt's own opinions about where models, tests, macros, and seeds go, and the medallion pattern (bronze → silver → gold layering). Learn this convention once and you'll recognize ~80% of real repos on sight.

Honest framing
What you built in the lab is one reasonable version of this convention — the dbt portion especially is exactly right. It's recognizable to anyone who's worked in a modern stack. The rest of this file shows the full shape so you know what the lab is a subset of.

02The de facto layout

The structure is organized around the medallion layering — bronze → silver → gold, which in dbt terms is staging → intermediate → marts. That layering is the spine; everything else (orchestration, ingestion, data quality) wraps around it. The tree below is annotated with which tier each folder produces. Read the layering as: ingestion → bronze, staging/intermediate → silver, marts → gold. The models/ folder holds the silver and gold transformations; bronze lands just before it; orchestration and data quality sit outside the layering because they act on all of it.

The relationship in one line
Ingestion lands raw data; dbt transforms it through the layers; orchestration decides when it all runs and in what order; the data-quality layer proves the result is correct and gates go-live. Your harness is the validation box — a legitimately distinct piece many teams under-build.

04The bronze / silver / gold vocabulary

You'll hear "medallion architecture" constantly. It's just named tiers of refinement — the same three zones from the concepts companion, with industry labels:

TierWhat's in itYour lab equivalent

Bronze/silver/gold and staging/intermediate/marts describe the same idea from two angles — the medallion terms name the data tiers, the dbt folder names organize the code that produces them. Knowing both vocabularies means you can talk to a Databricks shop (medallion-speak) and a dbt shop (folder-speak) without missing a beat.

05Choosing the gold-layer shape

The medallion tiers tell you how refined the data is, not what shape the gold tables take. fct_/dim_ is one shape — the star schema — and it's the right default, but it isn't the only option. Three shapes show up in practice, and they're really points on a denormalization spectrum: how much you pre-join so the consumer doesn't have to.

Normalized · 3NFmany narrow tables
→
Star schemafact + denormalized dims
→
One Big Tablesingle wide table

Left is write-optimized and join-heavy; right is read-optimized and join-free. You move rightward as you go from the operational source toward the dashboard.

1 · Normalized (3NF) — the source's shape, not the warehouse's

Every fact is stored exactly once, split across many narrow tables linked by keys (an order references a customer_id, not the customer's name and city). This kills update anomalies and is ideal for transactional writes — which is why your Postgres source is almost certainly 3NF. It's poor for analytics: a single business question fans out across a dozen joins, and BI users can't navigate it.

Use when
The system is transactional (the OLTP source). In the warehouse, 3NF is the thing you're migrating away from — reshaping it is the gold layer's job.

2 · Star schema — the default gold layer

One central fact table at a fixed grain (one row per order), surrounded by dimension tables that hold descriptive context (dim_customers, dim_products, dim_date). Dimensions are denormalized — each collapses its own sub-tables so a query joins fact→dim once, not through a chain — but the fact and dims stay separate so wide text isn't duplicated on every fact row. It's the Kimball standard because it balances three things at once: simple joins, BI-tool friendliness, and one model that many dashboards read.

Use when
It's the default — pick it unless you have a specific reason not to. Especially when several facts share the same dimensions (conformed dims), when self-serve BI tools query the model, or when you want one governed layer many reports sit on.

3 · One Big Table (OBT) — fully flattened, no joins

Take the star and pre-join every dimension attribute into the fact, producing a single wide table the consumer scans with zero joins. On columnar engines (DuckDB, BigQuery, Snowflake) a wide scan is cheap and the join is the real cost, so this can be faster and simpler for one consumer. The price: data is duplicated (every order row carries the full customer name/city), it's harder to keep consistent, and a dimension change has to be rewritten across all history.

Use when
One well-understood consumer — a single dashboard, an ML feature table, or analysts who can't/won't write joins. Build it on top of the marts as a convenience layer, not instead of them.

One more you'll hear named
The snowflake schema is a star whose dimensions are themselves normalized (a dim_product that points out to a separate dim_category rather than carrying the category name). It saves a little storage at the cost of more joins, and on cheap columnar storage that trade rarely pays — most teams keep dims flat and stay with a plain star. Data Vault exists for highly-audited, many-source enterprise warehouses; it's almost never your call on a single migration.
For this migration specifically
Your source is 3NF; the gold layer's job is to reshape 3NF → star in marts/. That's the work, and it's exactly what dbt staging→marts produces. Reach for an OBT only when a named consumer asks for one — build it downstream of the star, never as a replacement for it.

07Three things production wraps around the core

These are the honest gaps between the lab and a full production repo. None of them undercut the lab — they're the next layers out — but knowing them is the difference between "I built a thing" and "I built a thing a team can run safely."

Gap 1 · Orchestration

In the lab you run load_source.py, then dbt run, then reconcile by hand. Production wires those into a scheduled, dependency-aware pipeline (Airflow or Dagster) with retries, alerting, and logging. You already know these tools — the lab just doesn't exercise them. Expect the interview question: "how does this run on a schedule?"

Gap 2 · Ingestion is usually bought, not built

Your hand-written migration script is correct for a one-time cutover migration — which is exactly your use case, so this is appropriate. But for ongoing pipelines most shops use managed connectors (Fivetran/Airbyte) instead of custom Python. The distinction to hold: migration (one-time, custom, where your harness shines) vs. ingestion (continuous, off-the-shelf).

Gap 3 · Environments & CI

Production separates dev / staging / prod, runs dbt build + tests + reconciliation on every pull request via CI, and lints SQL (sqlfluff). The lab is single-environment. This is the gap between a working artifact and one a team can change without fear.

Tools that fill these gaps, for reference: AirflowDagsterFivetranAirbyteGitHub Actionssqlfluffpre-commit — most are already in your stack familiarity.

08The bottom line

Your layout is a correct and recognizable version of the standard — the dbt portion especially is exactly right.

The convention itself is real and worth internalizing as a template. A full production repo wraps that core in three things the lab omits: orchestration, environment/CI discipline, and (for ongoing work) managed ingestion.

For a migration engagement specifically
One-time move, prove it, cut over — your setup is close to complete, because migration leans on the validation harness far more than on continuous orchestration. The gaps above matter most for ongoing data-platform work, which is your Phase 3 (fractional/advisory) territory, not your entry point.
