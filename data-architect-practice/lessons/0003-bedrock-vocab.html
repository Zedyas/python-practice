The win

One sentence each for OLTP vs OLAP, row vs columnar, normalization, ACID, the grain, surrogate key — plus the single thread that links them: a database built to record the business is the wrong shape to analyze it.

01OLTP vs OLAP Source ↔ Store

OLTP (Online Transaction Processing) = the database running the business: many tiny reads/writes — place an order, update a balance. OLAP (Online Analytical Processing) = the system answering questions about the business: few huge scans — "revenue by state this quarter."

This is the master distinction. Almost every other word here exists because these two have opposite needs:

OLTP (your Postgres source)OLAP (your warehouse)
Optimized forfast writes, single-row lookupsfast scans, big aggregations
Layoutrow-orientedcolumnar
Shapenormalized (many thin tables)denormalized (few wide tables)
Example"insert this order""sum revenue by month"
Why it mattersThe whole pipeline exists to move data from the OLTP shape (good for recording) to the OLAP shape (good for analyzing). Ingest + Transform are that reshaping.

02Row-oriented vs columnar Store

This is the physical reason OLTP and OLAP differ. It's just how bytes sit on disk:

  • Row-oriented: all of one row's columns stored together. To read/update a whole record, grab one place. Perfect for OLTP.
  • Columnar: all of one column's values stored together. To sum(revenue) over 100M rows, read only the revenue column — and since neighbouring values are similar, it compresses brutally well. Perfect for OLAP.
-- "sum revenue" touches ONE column. Columnar reads just that column's
-- bytes and skips the other 40. Row storage would read every full row.
SELECT sum(qty * unit_price) FROM orders JOIN products USING(product_id);

What breaks if you get it backwards: run analytics on a row store and every aggregate drags the full table off disk — slow and expensive. This is why warehouses (BigQuery, Snowflake) and Parquet files are columnar.

03Normalization vs denormalization Source ↔ Transform

Normalized = each fact stored once, in its own thin table, joined by keys. Your shop is normalized: orders holds customer_id, not the customer's name — the name lives once in customers. Great for OLTP (change Amy's email in one place).

Denormalized = pre-join the pieces into one wide row so analytics needs no joins. Transform's job is to do exactly this:

-- three normalized tables -> one wide analytic row
SELECT o.order_id, c.name AS customer, p.name AS product,
       o.qty, p.unit_price, o.qty*p.unit_price AS line_revenue
FROM orders o JOIN customers c USING(customer_id)
                JOIN products  p USING(product_id);
-- 1000 │ Amy  │ Mug      │ 2 │ 12.00 │ 24.00
-- 1001 │ Amy  │ Notebook │ 1 │  6.50 │  6.50
-- 1003 │ Cara │ Mug      │ 4 │ 12.00 │ 48.00   …one row, no joins needed downstream

Closely related: schema-on-write (enforce structure when loading — the warehouse way) vs schema-on-read (store raw, structure it at query time — the lake way). [Glossary · Bedrock]

04ACID Source

Definition: the four guarantees an OLTP database makes about a transaction — Atomicity (all-or-nothing), Consistency (valid state to valid state), Isolation (concurrent transactions don't trip over each other), Durability (once committed, it survives a crash).

The canonical example: a bank transfer debits one account and credits another. Atomicity guarantees you can't get the debit without the credit — even if the server dies between them.

Why it matters here: it's why the source can be trusted as ground truth — and why, when you migrate, you're copying out of a system with strong guarantees into an analytics world that trades some of them away for scale (that's the CAP/consistency story, in the Architecture vocab lesson).

05The grain Transform · the most important word here

Definition: what one row represents. You state it as a sentence — "one row per ___" — before you build any table. For your orders: one row per order line.

Grain governs how you aggregate. Because each orders row is one line, "total paid revenue" sums qty × unit_price at that line grain:

SELECT sum(o.qty * p.unit_price) AS paid_revenue
FROM orders o JOIN products p USING(product_id)
WHERE o.status = 'paid';
-- paid_revenue = 78.50

What breaks without it: mix grains and you double-count. If you joined orders to a table that had two rows per customer (say two addresses), every order would silently count twice and revenue would inflate — the classic join explosion / fan-out. Declaring the grain is what stops it.

The interview tell"What's the grain?" is the first question a good designer asks about any fact table. Answer it out loud before designing anything. It returns in Lesson 0004 as its own deep dive — it's that load-bearing.

06Surrogate key vs natural key Transform

Natural key = the real-world business id (email, order_id, customer_id from the source). Surrogate key = a meaningless system-generated id (1, 2, 3…) you mint in the warehouse to join dimensions.

SELECT row_number() OVER (ORDER BY customer_id) AS customer_sk,
       customer_id AS natural_key, name
FROM customers;
-- customer_sk │ natural_key │ name
--      1       │      1      │ Amy
--      2       │      2      │ Ben
--      3       │      3      │ Cara

Why bother when the natural key looks fine? Two reasons that matter downstream: (1) source keys can change or be reused — surrogates stay stable; (2) SCD2 needs them — when Cara's address changes you get a second history row for the same customer_id, so the natural key is no longer unique; the surrogate key is what stays one-per-row. That's the bridge from this lesson back to SCD2 in 0002.

07Practice — say it before you flip

OLTP vs OLAP — one line each, and which layout each uses.

OLTP = runs the business, many small writes, row-oriented, normalized. OLAP = analyzes the business, big scans, columnar, denormalized.

Why are analytics warehouses columnar?

Aggregations touch few columns over many rows; columnar reads only those columns and compresses them hard — far less data off disk than reading whole rows.

State the grain of the shop's orders table, and why grain matters.

One row per order line. Grain governs aggregation; mixing grains (a fan-out join) double-counts and inflates every measure.

Why use a surrogate key when the natural key looks fine?

Source keys can change/be reused; and SCD2 makes the natural key non-unique (multiple history rows per customer), so the surrogate is what stays one-per-row.
