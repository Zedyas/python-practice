01Materializations Transform · the dbt build strategy

Definition: a dbt materialization is the strategy for turning a model's SELECT into something in the warehouse. The SQL stays the same; the materialization decides what gets built and how often. Four to know:

MaterializationWhat dbt buildsClaims example
viewstores only the SQL; recomputed on every read. No stored data.stg_claim_line — light renaming/casting over raw. Cheap, always fresh.
tablefully rebuilt and stored each run. Fast to read, full recompute every run.dim_member — small, rebuilt nightly from source.
incrementalprocesses only new/changed rows since last run, then appends/merges into the existing table.fct_claim_line — billions of lines; load only claims with updated_at > last load, merge on (claim_id, line_no).
ephemeralnot built at all; inlined as a CTE into the models that reference it.int_claim_dedupe — a reused helper you don't want as its own object.
When eachview for light staging where freshness matters and compute is cheap · table for small/medium models read often · incremental for large append-mostly facts where rebuilding history every run is wasteful · ephemeral for small reused logic you don't want cluttering the warehouse. The decision is a cost/freshness trade, not a correctness one — the answer is identical.
Interview line"Staging models I leave as views; large facts I make incremental and merge on the grain's key so I never reprocess history; small dimensions are tables. The materialization is a cost lever — same result, fewer bytes rebuilt." Incremental ties straight to the watermark + idempotency ideas from 0002.

02Star vs snowflake schema Transform · the shape

Star schema: a central fact at a declared grain, surrounded by denormalized dimension tables — each dimension holds its descriptive attributes directly, so a query is one join hop fact→dim.

Snowflake schema: the dimensions are normalized into sub-tables, so attributes sit several hops away (fact → dim_member → dim_plan → plan attributes).

-- STAR: plan name lives right on the member dimension (one hop)
fct_claim_line → dim_member(plan='GOLD', plan_tier='premium', …)

-- SNOWFLAKE: member points to a plan dimension (two hops)
fct_claim_line → dim_member(plan_key) → dim_plan(plan='GOLD', plan_tier='premium')
Which, and whyAnalytics defaults to star: fewer joins, simpler SQL, and columnar storage makes the redundant repeated text essentially free. Snowflaking saves space that's already cheap and adds join hops that aren't — so you only do it for genuinely large or volatile sub-dimensions. "Normalize for writes (OLTP), denormalize for reads (OLAP)" — star is that principle applied.

03Dimension types Transform · Kimball vocabulary

Three dimension words interviewers reach for, all illustrated on the claims star:

TypeDefinitionClaims example
Conformedone dimension shared across multiple facts with identical keys & meaningdim_member & dim_date used by both fct_claim_line and fct_pharmacy_claim → lets you compare medical vs pharmacy spend per member/date
Role-playingone physical dimension joined multiple times in different rolesdim_date joined as service_date, paid_date, submitted_date — same table, three foreign keys
Degeneratean identifier stored on the fact that you group/filter by but that has no dimension table of its ownclaim_id, line_no — they describe the row but carry no attributes worth a table

Conformed dimensions are the load-bearing one: they're what make drill-across possible — asking one question of two business processes and lining the answers up by the shared dimension. A claims fact and a pharmacy fact that both conform to dim_member can be compared member-by-member; if each had its own private member table, they couldn't.

-- ROLE-PLAYING: dim_date joined twice, two roles on the same fact
SELECT sd.date_key AS service_day, pd.date_key AS paid_day, sum(line_amount)
FROM fct_claim_line f
JOIN dim_date sd ON f.service_date = sd.date_key   -- role 1
JOIN dim_date pd ON f.paid_date    = pd.date_key;  -- role 2  → 2026-03-10 / 2026-03-20 / 300.00
Also worth a nameJunk dimension — a small table bundling several low-cardinality flags (is_emergency, is_inpatient, is_outofnetwork) so the fact carries one key instead of a scatter of boolean columns.

04Slowly changing dimensions (SCD 1 / 2 / 3) Transform · history

Definition: SCD strategies decide what happens to a dimension row when an attribute changes — e.g. a member switches plan BRONZE → GOLD. The choice is whether you keep history, and how much.

  • Type 1 — overwrite. Update the value in place; no history. Reports always show the current plan, even for old claims.
  • Type 2 — new row per change. Add a row with valid_from / valid_to + an is_current flag; the old row is closed off. Full history; you join point-in-time. (The pattern from 0002.)
  • Type 3 — add a column. Keep prior_plan alongside plan — exactly one previous value, no full timeline.

Why it matters, on real numbers. M-1 was BRONZE through Jan, GOLD from Feb. With a Type-2 dimension you can ask "what plan was the member on when the claim was incurred?" — and the answer splits correctly:

-- SCD2 POINT-IN-TIME: plan as-of the service date
SELECT m.plan, sum(f.line_amount)
FROM fct_claim_line f
JOIN dim_member m ON f.member_id = m.member_id
 AND f.service_date >= m.valid_from AND f.service_date < m.valid_to
GROUP BY m.plan;
--  BRONZE  300.00     GOLD  300.00      ← history preserved

-- join to the CURRENT row only (is_current) — history is lost
--  GOLD    600.00                       ← every old claim mislabeled GOLD
Interview line"If the business ever asks 'what was true at the time' — the plan when the claim was incurred, the address at time of shipment — the dimension has to be SCD2 and you join point-in-time on valid_from/valid_to. Type 1 silently rewrites history; that $600-all-GOLD number is the bug."

05Practice — say it before you flip

view / table / incremental / ephemeral — one line each.

view = stored SQL, recomputed on read · table = fully rebuilt & stored each run · incremental = only new/changed rows merged in · ephemeral = inlined as a CTE, never built.

Why does analytics prefer star over snowflake?

Fewer join hops + simpler SQL, and columnar storage makes the denormalized repetition cheap. Normalize for writes (OLTP), denormalize for reads (OLAP).

Conformed vs role-playing vs degenerate?

Conformed = one dim shared across facts (enables drill-across) · role-playing = one dim joined in several roles (dim_date as service/paid) · degenerate = an id on the fact with no dim table (claim_id).

When must a dimension be SCD2?

Whenever the business asks "what was true at the time" — you need valid_from/valid_to history and a point-in-time join. Type 1 overwrites and mislabels old facts with current values.
