01CDC — Change Data Capture Stage · Ingest

Definition: a way to move only the rows that changed in a source since the last sync, instead of re-copying the whole table every time.

Two mechanisms you should be able to name:

  • Query-based — keep a watermark (the last value you saw of an ever-increasing column like updated_at) and select rows past it.
  • Log-based — read the database's write-ahead log directly (tools like Debezium), so you catch deletes too and don't hammer the source. This is the "real" CDC.

Query-based CDC on your orders, using order_date as the watermark — "give me everything after the last load (2026-06-01)":

SELECT order_id, order_date FROM orders
WHERE order_date > DATE '2026-06-01';
-- ┌──────────┬────────────┐
-- │ order_id │ order_date │
-- ├──────────┼────────────┤
-- │ 1002     │ 2026-06-02 │   ← only the 2 new rows move, not all 4
-- │ 1003     │ 2026-06-02 │
-- └──────────┴────────────┘

What breaks without it: you re-copy the entire source every run. Fine for 4 rows, fatal for 400M — slow, expensive, and it pounds the live production database.

Interview line"CDC moves only what changed. Query-based uses a watermark column; log-based reads the DB's write-ahead log and also captures deletes." [Concepts 02 · CDC]

02Reconciliation Stage · Ingest (the proving job)

Definition: proving the data that landed matches the data at the source. Moving data is job one; reconciliation is job two — and it's the hard, billable one (your differentiator).

You climb a ladder, cheapest check first, only going deeper if a rung disagrees:

RungCheckCatches
1Row counts matchwhole batches dropped
2Column aggregates match (sum, min, max, checksum)truncation, rounding, type coercion
3Row-level diff on the primary keythe exact rows that differ
 
-- Rung 1 + 2 on the source side (run the same on the target, compare)
SELECT count(*) AS row_count, sum(qty) AS total_qty FROM orders;
-- row_count = 4 , total_qty = 8   →  target must show 4 and 8, or you have a bug

What breaks without it: silent data loss. The pipeline goes green, a dashboard quietly under-reports revenue, and nobody notices until a client does. You'll build this ladder the hard way — rough Python first — in a later lesson.

Interview line"Counts, then aggregates/checksums, then a PK-level row diff — escalate only when a rung disagrees." [Migration Lab · the seven checks]

03SCD2 — Slowly Changing Dimension, Type 2 Stage · Transform

Definition: a way to keep history when a descriptive attribute changes. Instead of overwriting the old value, you close the old row and open a new one, each stamped with a validity window.

Contrast the types: Type 1 overwrites (history lost). Type 2 keeps every version. Say Cara (customer 3) moves from TX to CA on June 10:

-- dim_customer as SCD2: two rows for Cara, each with a validity window
SELECT * FROM dim_customer_scd2 WHERE customer_id = 3;
-- ┌─────────────┬───────┬────────────┬────────────┬────────────┐
-- │ customer_id │ state │ valid_from │  valid_to  │ is_current │
-- ├─────────────┼───────┼────────────┼────────────┼────────────┤
-- │ 3           │ TX    │ 2026-01-01 │ 2026-06-10 │ false      │  ← closed
-- │ 3           │ CA    │ 2026-06-10 │ 9999-12-31 │ true       │  ← open
-- └─────────────┴───────┴────────────┴────────────┴────────────┘

Now "what did TX customers buy at the time of the order?" is answerable — you join each order to the dimension row whose window contains the order date. dbt does this for you with snapshots.

What breaks without it: overwrite Cara to CA and every historical TX order retroactively becomes a CA order. Your past reports silently rewrite themselves.

Interview line"SCD2 preserves history by closing the old row and inserting a new one with valid_from/valid_to — so point-in-time questions stay correct." [Concepts 03 · SCD2 worked example]

04Idempotency Undercurrent · Orchestration

Definition: running a step twice leaves the system in the same state as running it once. No duplicates, no double-counting — so a retry or a backfill is always safe.

The difference is entirely in how you write. A blind INSERT is not idempotent — re-run it and rows double. Load on a key (anti-join, or MERGE) and it is:

-- idempotent load: insert only keys not already present
INSERT INTO tgt
SELECT * FROM orders o
WHERE NOT EXISTS (SELECT 1 FROM tgt t WHERE t.order_id = o.order_id);

-- run it once  → tgt has 4 rows
-- run it AGAIN → tgt STILL has 4 rows   (a blind INSERT would show 8)

What breaks without it: a job fails halfway, the orchestrator retries it, and now half your rows are duplicated. Idempotency is what makes retries and backfills survivable — which is why it lives in the Orchestration band.

Interview line"Idempotent = re-running is a no-op. I get it by loading on a unique key (merge / delete+insert), so retries and backfills can't double-count." [Concepts 05 · idempotency]

05Masking Undercurrent · Governance

Definition: showing a sensitive column in an obscured form to roles that aren't allowed to see the real value — while the true value stays underneath for those who are.

Your customers.email is PII. A masked view shows enough to be useful (you can tell two customers apart) but not the actual address:

SELECT email AS real,
       regexp_replace(email, '(^.).*(@.*)$', '\1***\2') AS masked
FROM customers;
-- ┌────────────┬────────────┐
-- │    real    │   masked   │
-- ├────────────┼────────────┤
-- │ amy@x.com  │ a***@x.com │   ← analysts see this column
-- │ ben@x.com  │ b***@x.com │
-- └────────────┴────────────┘

In production you don't hand-write this per query — a privileged role sees email, everyone else is routed to the masked view (or a column-masking policy). What breaks without it: every analyst, contractor, and BI dashboard has raw PII — a direct GDPR/HIPAA violation and a breach waiting to happen.

Interview line"Masking obscures a sensitive value for unprivileged roles while keeping the real value for authorized ones — enforced by a policy, not by trusting each query." [Concepts G · masking worked example]

06Practice — say it before you flip

Cover the lesson. For each card, answer out loud, then reveal.

CDC — one sentence, plus its two mechanisms.

Move only the rows that changed since last sync. Mechanisms: query-based (watermark column) and log-based (read the write-ahead log; catches deletes).

The reconciliation ladder — three rungs, cheapest first.

1) row counts, 2) column aggregates/checksums, 3) row-level diff on the primary key. Escalate only when a rung disagrees.

Why does SCD Type 2 beat Type 1 for history?

Type 1 overwrites, erasing the past; Type 2 closes the old row and opens a new one with valid_from/valid_to, so point-in-time questions stay correct.

What makes an INSERT load idempotent?

Loading on a unique key — anti-join / MERGE / delete+insert — so re-running inserts nothing new. A blind INSERT duplicates on every re-run.
