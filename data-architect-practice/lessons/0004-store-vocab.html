The win

One sentence each for storage÷compute, MPP, lake / warehouse / lakehouse, table format, partitioning, clustering, bytes scanned — and the thread: modern platforms decouple where data sits from what reads it, so you tune cost by reading less.

01Storage ÷ compute (separation) Store · the core idea

Definition: the cloud warehouse keeps your data in cheap object storage and runs queries on compute it spins up and down separately. The two are decoupled — you pay for each on its own.

Old on-prem warehouses fused them: one box held the disks and did the querying, so to store more you bought more compute you didn't need (and vice-versa). Separation is the modern-platform move:

  • Storage is cheap and effectively unlimited — keep all the raw data.
  • Compute scales on demand — ten analysts at 9am spin up ten warehouses; at night it's off and costs nothing.
Interview line"Cloud warehouses separate storage from compute, so you scale and pay for each independently — that's why they're elastic and cheap at rest." [Glossary · Store]

02MPP — Massively Parallel Processing Store

Definition: a query is split across many compute nodes, each scanning a slice of the data at the same time, then the partial results are combined.

It's how compute scales. "Sum revenue over 2 billion rows" doesn't run on one machine reading 2B rows; it runs on, say, 200 nodes each reading 10M, in parallel. Double the nodes ≈ halve the time. This is why warehouses (BigQuery "slots", Snowflake virtual warehouses) feel instant on huge tables — and why columnar storage matters so much: each node reads only the needed columns of its slice.

03Lake vs warehouse vs lakehouse Store · know the trade

The three shapes the store can take. Define each precisely:

  • Lake — raw files (Parquet/JSON/CSV) sitting in your cheap object storage, with no table layer. You read whatever files are there (schema-on-read). Flexible and cheap; untrustworthy on its own — the "data swamp."
  • Lakehouse — those same open files in your object storage, plus an open table-format metadata layer (Iceberg/Delta) that tracks which files make up a table, its schema, and its versions. Result: ACID, schema enforcement, and time-travel on files you still own.
  • Warehouse — you load data into the vendor's system, which stores it in its own proprietary, internal format you never see. Only the warehouse's engine reads it. Most polished and governed; proprietary lock-in and pricier (BigQuery, Snowflake).
The distinction people get wrongBoth the warehouse and the lakehouse have strict tables, schemas, and ACID — so "strict tables" is not what separates them. Two different jumps:
• Lake → lakehouse: add the metadata/table layer over the files.
• Lakehouse → warehouse: open files you own (any engine reads them) vs a proprietary store the vendor fully manages (only their engine reads it).

04Table format (Iceberg / Delta / Hudi) Store

Definition: a metadata layer over your Parquet files that tracks which files make up a table, so you get database features — ACID transactions, schema evolution, time-travel — on top of plain files.

Without it, "a table" is just a folder of Parquet; two writers collide, a schema change breaks readers, there's no "as of last Tuesday." A table format (Apache Iceberg, Delta Lake, Apache Hudi) adds a transaction log + manifest so the folder behaves like a real, safe table. This is what turns a lake into a lakehouse.

Parquet itself is the layer below: the open, compressed, columnar file format — the physical bytes. Table format = the rules over the files; Parquet = the files.

The standup question, answered"What is an Iceberg table?" → "Parquet files plus a metadata/transaction layer that gives them ACID, schema evolution and time-travel — so a pile of files acts like a governed table." [Concepts 01 · table formats]

05Partitioning Store · the #1 cost lever

Definition: partitioning physically splits a table into separate folders by the value of a column (usually a date or category). The value lives in the folder path, not just inside the data — so a filtered query can read the folder names, throw away every folder it doesn't need, and never open those files at all. That skip is called partition pruning.

It's a coarse, higher-order split: the division happens at the directory level, and a whole folder is either in or out. Partition your orders by status and the storage literally becomes separate folders:

COPY (SELECT * FROM orders)
  TO 'orders_parq' (FORMAT parquet, PARTITION_BY (status));

-- on disk:
--   orders_parq/status=paid/data_0.parquet
--   orders_parq/status=refunded/data_0.parquet

-- a query filtered on status reads ONLY the paid folder — the
-- refunded files are never opened (that's partition pruning):
SELECT count(*) FROM read_parquet('orders_parq/*/*.parquet', hive_partitioning=true)
WHERE status = 'paid';   -- 3

Pick the partition column to match how you filter — almost always date ("last 7 days" reads 7 folders, not 5 years). Partition on the wrong / too-fine a column (e.g. a unique id) and you get millions of tiny files — worse, not better.

06Clustering Store

Definition: clustering sorts / co-locates rows within the data by a column, so similar values sit together in the same files. Each file then carries min/max statistics for its columns, and the engine skips any file whose range can't contain your filter value — without reading it. That skip is called data skipping (or file skipping).

The efficiency comes entirely from tight ranges. Sort orders by customer_id and file A holds ids 1–4000, file B holds 4001–8000; a query for customer_id = 5000 reads only B. Unsorted, every file's range would be ~1–8000 and skipping would do nothing.

The mental split: partitioning chops the table into separate folders — coarse, visible on disk, prune whole folders. Clustering orders the rows inside the files so min/max ranges stay tight — fine, invisible on disk, skip individual files. You often do both: partition by date, cluster by customer.

When eachPartition on a low-cardinality column you filter on a lot (date, region) — you want few, healthy-sized folders. Cluster on a high-cardinality column you also filter/join on (customer_id, product_id) — too many distinct values to partition by without exploding into tiny files.
…vs an indexNeither is a database index. A B-tree index is a separate structure of pointers to individual rows, built for OLTP point lookups ("fetch the one row where id = 5000"). Partitioning and clustering store no pointers — they organise the data so an OLAP engine can skip scanning what it doesn't need. The clean line: OLTP indexes data to find rows; OLAP partitions and clusters data to skip scanning it. Analytics warehouses generally don't maintain row-level indexes at all.
How fine / how many
  • Partitions — governed by size, not count. Aim for each partition to hold a healthy chunk (≈ a few hundred MB–1 GB). Use one partition column (date), and pick the grain to hit that size: month for low volume, day as default, hour for a firehose. A second level (date + region) rarely pays — counts multiply into the small-file problem. Keep total partitions in the low thousands. (BigQuery hard-caps a table at 4,000 partitions, one partition column.)
  • Clusters — about 3–4 columns, order matters. Clustering takes an ordered list; put the most-filtered / most-selective column first, because a query only benefits from a prefix of that list. (BigQuery max 4 clustering columns; Snowflake recommends ≤ 3–4.) Past ~3 keys the benefit flattens and maintenance cost rises.
  • Partitions are structural; clustering decays. New writes land unsorted, so clustering co-location degrades and needs periodic re-clustering / compaction (Snowflake auto-reclusters · Delta OPTIMIZE ZORDER · Iceberg compaction). Partitioning doesn't drift.

07Bytes scanned Cost / FinOps

Definition: how much data a query actually reads off storage — and on a serverless warehouse like BigQuery, literally what you're billed for. The cost conversation is conducted in bytes scanned.

Everything above is a lever on this one number:

  • Columnar → read only the columns you select, not whole rows.
  • Partitioning → read only the folders your filter needs.
  • Clustering → skip blocks within those that can't match.

The senior framing: "reduce cost without hurting SLAs" = "scan fewer bytes" — select fewer columns, partition + cluster to match query patterns, and don't SELECT * on a wide table. Cost is a designed property, which is why it's a cross-cutting band, not an afterthought.

Interview line"On a serverless warehouse you pay per byte scanned, so I tune cost by scanning less — column pruning, partition pruning, clustering — without changing the answer or the freshness." [Concepts 01 · the cost lever]

08Practice — say it before you flip

What does "separation of storage and compute" buy you?

Scale and pay for each independently — unlimited cheap storage, on-demand compute that costs nothing at rest. The core modern-platform property.

What turns a data lake into a lakehouse?

A table format (Iceberg/Delta/Hudi) — a metadata + transaction layer over the Parquet files giving ACID, schema evolution, and time-travel.

Partitioning vs clustering — one line each.

Partitioning splits the table into separate folders by a column (coarse, visible on disk); clustering sorts rows within them so filters skip blocks (fine, invisible). Both reduce bytes scanned.

"Reduce warehouse cost without hurting SLAs" — translate to one metric and three levers.

Metric: bytes scanned. Levers: column pruning (don't SELECT *), partition pruning (filter the partition key), clustering (skip blocks). Answer doesn't change; cost drops.
