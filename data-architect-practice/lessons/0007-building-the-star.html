01Where Transform sits — ELT, not ETL Transform

Definition: in the modern stack the order is ELT — Extract and Load land raw data in the warehouse first (the Ingest/Store stages), then Transform runs as SQL inside the warehouse. dbt is that T: it takes a folder of SELECT statements ("models") and builds them into tables/views in dependency order.

The old ETL transformed data before loading, on a separate box. ELT won because storage is cheap (keep the raw), and the warehouse's MPP compute is the best engine you have to transform with — so push the work to where the data already is. This is why "Transform" on the spine is in-warehouse SQL, not a separate processing tier.

02The three model layers Transform · project structure

A dbt project is organised into three layers, each with a job it is and isn't allowed to do. This is the single most-asked structural question in interviews.

LayerJob — and the ruleClaims modelsMedallion
staging stg_one model per source table; only rename, recast, light cleanup. No joins, no business logic. Usually views.stg_claims, stg_members, stg_providersbronze→silver
intermediate int_the messy middle — joins, dedupe, reshaping, business logic. Not exposed to BI.int_claim_enriched (lines + header + member)silver
marts dim_/fct_the business-ready star — what BI and analysts actually query.fct_claim_line, dim_member, dim_date, dim_cptgold
Why the disciplineEach model does one thing, so logic lives in exactly one place and is testable. Staging being 1:1 with sources means there's a single spot where each raw table gets cleaned — every downstream model reads the clean version, never the raw. Skip the layers and you get the same join logic copy-pasted into five marts, drifting out of sync — the thing the structure exists to prevent.

03ref() / source() and the DAG Transform · the engine

You never hardcode a table name in dbt. You write two functions, and dbt reads them to build the dependency graph:

  • source('raw', 'claims') — "this model depends on a declared raw input." Marks the edge of your project where external data enters.
  • ref('stg_claims') — "this model depends on another model." dbt substitutes the real (environment-correct) table name.
-- models/intermediate/int_claim_enriched.sql
SELECT l.claim_id, l.line_no, l.cpt, l.line_amount, c.member_id, c.service_date
FROM {{ ref('stg_claim_line') }} l
JOIN {{ ref('stg_claims')    }} c USING (claim_id)

From those calls dbt derives the whole DAG (directed acyclic graph) and gets three things for free: build order (a model runs only after its refs), parallelism (independent branches run together), and lineage (it knows every model's upstream sources and downstream consumers — so "what breaks if this column changes?" is answerable). Hardcoding a table name would sever that edge and you'd lose all three.

04Assembling the claims star Transform · putting it together

Here is the whole project as a graph — raw on the left, the star on the right, each box a single model file:

raw.claims  ──▶ stg_claims   ─┐
raw.members ──▶ stg_members  ─┤                              ┌─▶ dim_member  ─┐
raw.cpt     ──▶ stg_cpt      ─┼─▶ int_claim_enriched ──────▶ ┤  dim_date     ─┼─▶ fct_claim_line ──▶ BI
raw.lines   ──▶ stg_claim_line┘                              └─▶ dim_cpt      ─┘
                staging (views)        intermediate              marts (the star)
  • staging cleans each source once (cast service_date to a date, trim ids, standardise plan codes).
  • int_claim_enriched joins lines to their header and member, dedupes resubmitted claims — all the logic that needs more than one table.
  • dims are built from staging (dim_member as an SCD2 snapshot — see §05); fct_claim_line is the atomic-grain fact (incremental, merged on (claim_id, line_no)) carrying foreign keys to each dim plus the additive measures.
Read the grain off the graphThe fact sits at the line grain (0005); every dimension feeding it is at one-row-per-entity grain. The layers are just the path from raw to that declared grain — staging makes raw trustworthy, intermediate does the joins, the mart is the grain made into a table.

05The surrogate-key / SCD2 fact wiring Transform · the trap

The deferred question from 0006: does the fact store member_id (natural) or member_key (surrogate)? With an SCD2 dimension the answer matters, and getting it wrong silently corrupts history.

An SCD2 dim_member has one row per version, each with its own surrogate member_key (1, 2, 3…) — so M-1's BRONZE era is key 1 and its GOLD era is key 2. The load step resolves which version was valid at the event date and stamps that surrogate onto the fact:

-- LOAD: resolve the SCD2 version valid AT the service date, freeze its surrogate key
INSERT INTO fct_claim_line
SELECT r.claim_id, r.line_no, m.member_key, r.service_date, r.line_amount
FROM raw_claim_line r
JOIN dim_member m
  ON r.member_id = m.member_id
 AND r.service_date >= m.valid_from AND r.service_date < m.valid_to;

-- result: Jan lines get member_key 1 (BRONZE), Mar line gets member_key 2 (GOLD)

Now the point-in-time logic is frozen into the fact at load, so every query is a plain surrogate join — no date ranges, and history stays correct:

SELECT m.plan, sum(f.line_amount)
FROM fct_claim_line f JOIN dim_member m USING (member_key)
GROUP BY m.plan;
--  BRONZE 300.00   GOLD 300.00     ✓  (vs. natural-key+is_current = 600 all GOLD)
The rule + interview lineStore the surrogate member_key on the fact, resolved to the version valid at event time. Storing the natural member_id forces a fragile point-in-time join on every query — and joining to is_current instead silently relabels old claims with today's plan. "Facts carry the surrogate key of the dimension version that was true when the event happened — that's what makes SCD2 history queryable with a normal join." 06Practice — say it before you flip

Name the three model layers and the one-line rule for each.

staging = 1:1 per source, clean only, no joins · intermediate = joins/dedupe/business logic, not for BI · marts = the business-ready star (dim_/fct_) that BI queries.

What do ref() and source() buy you?

dbt reads them to build the DAG → automatic build order, parallelism, and end-to-end lineage. source() = declared raw input; ref() = dependency on another model.

Why ELT instead of ETL?

Cheap storage means you keep the raw; the warehouse's MPP compute is the best transform engine you have, so push the T to where the data already sits. dbt is the T.

Natural or surrogate key on a fact pointing at an SCD2 dim — and why?

Surrogate (member_key), resolved at load to the version valid at event time. It freezes point-in-time history so query joins are plain and correct; natural key + is_current relabels old facts with current values.
