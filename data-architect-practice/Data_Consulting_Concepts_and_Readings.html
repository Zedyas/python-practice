A · How data is stored and queried

OLTP vs OLAPThe two jobs a database can have

Plain version. A database is doing one of two very different jobs. OLTP (Online Transaction Processing) runs the business minute to minute — place an order, update a balance — millions of tiny reads and writes, each touching one or a few rows. OLAP (Online Analytical Processing) analyzes the business — "total sales by region last quarter" — a few big queries that scan millions of rows.

Analogy. OLTP is the cash register: fast, one customer at a time. OLAP is the accountant who reads every receipt at night to find trends.

Example OLTP: UPDATE accounts SET balance = balance - 20 WHERE id = 7; — touches one row, runs in a millisecond.
OLAP: SELECT region, SUM(amount) FROM sales GROUP BY region; — scans the whole sales history.

Going deeper

They want opposite physical designs. OLTP wants row storage, indexes, and normalization so a single record is cheap to fetch and update. OLAP wants columnar storage and denormalization so scanning one column across millions of rows is cheap. Crucially, you don't run heavy analytics on the live OLTP database — a big report would compete with real customer traffic and slow the app. So we copy data out of the OLTP systems into a separate analytics store (the warehouse). That copy is the reason the entire data-engineering field exists.

Where each belongs
  • OLTP system (Postgres, MySQL, Oracle): anything customer-facing that reads/writes single records live.
  • OLAP store (a warehouse): analyst-facing aggregation and reporting — fed by a copy of OLTP data.
  • Rule: never run a heavy report directly on the OLTP database; copy it out first.

Why we care: the very first question on any engagement — "is this transactional or analytical?" — decides every later choice: storage layout, data model, and tool. See Part 7.

Row vs ColumnarWhy analytics storage looks different

Plain version. The same table can be physically laid out on disk two ways: row by row (all of Amy's fields together, then all of Ben's), or column by column (every name together, then every age together).

Analogy. A row store keeps each person's full file in one folder — perfect for pulling up one person. A column store keeps all the ages in one drawer, all the cities in another — perfect for "what's the average age of everyone?" (grab one drawer, ignore the rest).

Example A table with 50 columns and 100M rows. SELECT AVG(salary) on a column store reads only the salary column — about 1/50th of the data. A row store must read every full row to get at that one field.

Going deeper

Columnar wins analytics for three reasons: compression (a column of similar values squashes far better than mixed-type rows), predicate pushdown (skip whole blocks that can't match a filter), and column pruning (read only the columns the query names). The cost is that writing or updating a single record is expensive — which is exactly why columnar = analytics and row = transactions. Parquet and ORC are columnar file formats; cloud warehouses store columnar internally.

When to use which
  • Columnar (Parquet/ORC, warehouse internals): analytics — scans and aggregations, "a few columns over many rows."
  • Row (OLTP engines): point lookups and frequent single-record inserts/updates.
  • Rule: let the workload pick the layout — OLAP→columnar, OLTP→row — not the reverse.

Why we care: this is why Parquet exists, why a warehouse is fast and cheap for reporting, and why you never point a BI dashboard at the production app database.

NormalizationStore each fact once — or copy it for speed?

Plain version. Normalize = store every fact exactly once, split across related tables, no repetition. Denormalize = deliberately repeat data, pre-joined into wide tables, so reads are fast.

Analogy. Normalized is a tidy filing cabinet: a customer's address lives in one place, so you change it once. Denormalized is a printed report where the address is copied onto every line — redundant, but readable with no cross-referencing.

Example Normalized: orders(order_id, customer_id) + customers(customer_id, name, city) — to show the city you must join.
Denormalized: orders(order_id, customer_id, customer_name, city) — city is stored on every order row.

Going deeper

Normalization (the textbook target is "3rd normal form") prevents update anomalies — if the address lived in 10 places you might update 9 and create a contradiction — and is ideal for OLTP writes. Denormalization trades storage and update-complexity for read speed, ideal for OLAP. Star schemas and "One Big Table" are deliberate denormalization. The one-line rule: normalize for writes, denormalize for reads.

When to use which
  • Normalize (3NF) for OLTP source systems and a Data Vault raw layer — write-heavy, change-once.
  • Denormalize (star schema, One Big Table) for the BI-facing layer — read-heavy, join-light.
  • In dbt: staging stays close to source shape; marts denormalize for consumption.

Why we care: every modeling decision you make is a point on this spectrum; naming where you sit and why is half of dimensional modeling (Part 3).

Schema-on-read vs -writeWhen do you enforce the shape?

Plain version. Do you enforce the structure of data when you write it in (a warehouse rejects anything that doesn't fit the table) or when you read it out (a lake stores anything; you impose structure only at query time)?

Analogy. Schema-on-write is a form with required fields — nothing is saved unless it fits. Schema-on-read is a shoebox of receipts — throw anything in now, make sense of it later.

Example Warehouse: you must CREATE TABLE … (id INT, ts TIMESTAMP) before loading, and a malformed row is rejected. Lake: dump raw JSON files into S3 today; impose a schema only when a query finally reads them.

Going deeper

Schema-on-write buys governance and reliability up front at the cost of rigidity. Schema-on-read buys flexibility and cheap raw capture at the risk of a "data swamp" nobody can trust. The lakehouse (Group B) exists to get both: lake-style cheap files plus a table format that adds schema enforcement and evolution back on top.

When to use which
  • Schema-on-write when consumers need reliability and governance (finance, regulated reporting, defined KPIs).
  • Schema-on-read for raw landing, exploration, ML, and data whose shape is unknown or still changing.
  • Lakehouse = read-cheap landing (Bronze) + write-strict curated layers (Silver/Gold) — you don't pick globally.

Why we care: this single trade-off is the warehouse-vs-lake tension that Part 1 is built around.

B · The modern cloud platform

Storage ÷ ComputeThe defining cloud-warehouse shift

Plain version. Old databases bolted the disks (storage) and the processors (compute) together — to get more query power you were forced to buy more storage too. Cloud warehouses split them: data sits in cheap object storage; query engines spin up separately, read from it, and shut off.

In short. Old systems welded storage and compute into one box, so more query power meant buying more storage you didn't need. Cloud warehouses rent them separately — cheap storage always on, query engines you spin up only while querying and suspend when idle.

Example In Snowflake the data lives in the storage layer; you turn on a "virtual warehouse" (the compute) to query it, scale it up for a heavy job, and suspend it after. The finance team and the data-science team can each run their own compute over the same data without slowing each other down.

Going deeper

This unlocks independent scaling, workload isolation (your nightly ETL can't throttle the BI users), pay-for-use pricing, and many engines over one copy of the data — which is the technical seed of the lakehouse. Snowflake popularized it; BigQuery does it with serverless "slots," Databricks with clusters over lake storage.

How to use it in practice
  • Run separate compute per workload so ETL can't throttle BI and data science can't throttle either.
  • Size to the job and auto-suspend idle compute — the main cost lever.
  • Scale compute up for a one-off heavy load, then back down; storage is already cheap and independent.

Why we care: it's the foundation of every cloud-cost conversation — "size the compute, isolate the workloads, suspend when idle" is consulting bread and butter.

MPPHow warehouses scan billions of rows fast

Plain version. Massively Parallel Processing: split one big query across many machines that each handle a slice of the data, then combine the partial answers.

Analogy. Counting votes. One person counting a million ballots is slow; 100 people each counting 10,000 and then summing their totals is fast. Handing out the ballots evenly is the whole trick.

Example SUM(sales) over 1 billion rows: 100 nodes each sum ~10M rows, then one final step adds the 100 partial sums.

Going deeper

The hidden cost is moving data between machines. A shuffle — redistributing rows across nodes so matching keys land together, e.g. for a join — is expensive. A broadcast join avoids it by copying a small table to every node. Data skew — one "hot" key holding most of the rows — means one node does all the work while the rest idle, and everyone waits for that straggler. Tuning MPP is mostly minimizing shuffle and skew.

How to use it in practice
  • Cut shuffle: pick a distribution/cluster key on the column you join on most.
  • Broadcast small dimension tables instead of shuffling them.
  • Watch skew: one hot key (a giant customer, a NULL bucket) makes a single node the bottleneck — salt or filter it.

Why we care: it explains why some queries are mysteriously slow and gives you the exact vocabulary — shuffle, broadcast, skew — that system-design rounds probe.

Warehouse / Lake / LakehouseThe three storage architectures

Plain version. Three ways to hold analytics data. Warehouse = structured tables, strict and governed, great for SQL/BI. Lake = raw files on cheap object storage, holds anything, weak guarantees. Lakehouse = lake-style cheap storage with warehouse-grade guarantees added by a table format.

Analogy. Warehouse = a library with a strict catalog: everything shelved and indexed. Lake = a giant storage unit: cheap, holds anything, easy to lose track of. Lakehouse = that storage unit with a librarian and catalog bolted on.

Example Land raw clickstream JSON cheaply in the lake (Bronze), clean it into governed tables (Silver/Gold), and serve BI from those — all on one storage system, no second copy in a separate warehouse. That single-copy unification is the lakehouse pitch.

Going deeper

A pure warehouse struggles with cheap raw/unstructured/ML data; a pure lake becomes a "swamp" with no ACID, no reliable schema, weak governance. The lakehouse closes the gap by adding a table format (next card) over lake files. Trade-offs: newer tooling, and some lock-in to the chosen table format.

When to use which
  • Pure warehouse when the work is SQL/BI on structured data and you value simplicity + governance (classic finance/ops shop).
  • Lakehouse when you also have ML, semi-structured/streaming data, or large raw volumes and want one governed copy for both BI and ML.
  • Pure lake alone is rarely the end state now — add a table format and it becomes a lakehouse.

Why we care: "warehouse vs lakehouse — how do you choose for this client?" is one of the most common prompts; the honest answer turns on data variety, ML needs, and existing tooling.

Table formats"ACID on a pile of files"

Plain version. A table format (Apache Iceberg, Delta Lake, Apache Hudi) is a layer of metadata that turns a loose pile of Parquet files in object storage into something that behaves like a real database table — transactions, safe schema changes, and "time travel" to past versions.

Analogy. The Parquet files are loose pages. The table format is the binding and table-of-contents that declares "version 5 of this table = exactly these pages," so a reader never catches a half-written update.

Example Two jobs append to the table at the same time. The format uses snapshots so each reader sees one consistent version, and you can ask "show me this table as it was yesterday at noon."

Going deeper

Metadata/manifest files track which data files belong to which snapshot. That gives ACID transactions (see Group C), schema evolution, time travel, and partition evolution — and it decouples the query engine from storage, so Spark, Trino, and a warehouse can all read the same tables. Copy-on-write (rewrite files on update, fast reads) vs merge-on-read (write deltas, merge at read time, fast writes) is the main tuning knob.

When to use which
  • Delta Lake if you're Databricks-centric (tightest integration).
  • Apache Iceberg for engine-neutral, multi-tool, open-ecosystem setups — the emerging default.
  • Apache Hudi for heavy streaming upserts / record-level CDC ingestion.
  • Any of them over raw Parquet the moment you need ACID, concurrent writers, schema evolution, or time travel.

Why we care: "why a table format instead of raw Parquet?" is a stock question — and the whole lakehouse depends on the answer. You can't give it without first understanding ACID, below.

Partitioning & clusteringReading less of the table

Plain version. Ways to physically organize data so a query reads only the slice it needs. Partition = physically split a table into separate folders by a column's value (e.g. one folder per day); the value lives in the folder path, so a filtered query prunes whole folders without opening them — coarse, directory-level. Cluster / sort = order rows within the files so similar values sit together; each file then carries min/max stats and the engine skips any file whose range can't match — fine, invisible on disk. Shard = split data across separate machines.

Analogy. A filing cabinet split into drawers by year (partition); within a drawer, files alphabetized (clustering); and several cabinets in different rooms (sharding).

Example On a table partitioned by day, WHERE order_date = '2026-06-01' reads one day's folder instead of the whole table. That skip is called partition pruning.

Going deeper

A good partition column is one you filter on often and is low-cardinality (few distinct values) — you want few, healthy-sized folders. Over-partitioning on a high-cardinality column creates millions of tiny files (the "small-file problem") that slow scans. Clustering / Z-ordering handles the high-cardinality columns you also filter/join on. Watch for hot partitions (one partition holding most of the data — the skew problem again). Sharding is the OLTP cousin: horizontal scaling by splitting rows across databases.

Not the same as a database index. A B-tree index is a separate structure of pointers to individual rows, built for OLTP point lookups ("fetch the one row where id = 5000"). Partitioning and clustering store no pointers — they organize the data so an OLAP engine can skip scanning what it can't match. The line: OLTP indexes data to find rows; OLAP partitions and clusters data to read less. The same logical action (e.g. "make date queries fast") is therefore implemented differently under the hood depending on the engine — an index on OLTP, a partition key on OLAP.

When to use which
  • Partition on a low-cardinality column you filter on most often — usually a date.
  • Cluster / Z-order for a common secondary filter within partitions.
  • Don't partition on a high-cardinality key (user_id, UUID) — it explodes into tiny files and slows scans.

Why we care: partitioning is the single biggest lever on both query speed and warehouse cost — "reduce spend without hurting SLAs" usually starts here.

C · Correctness & distributed-systems guarantees (the concepts you asked about)

ACIDThe four promises of a transaction

Plain version. A transaction is a unit of work that should happen all-or-nothing. ACID is the four guarantees a database makes about it: Atomicity, Consistency, Isolation, Durability.

Analogy. A bank transfer of $20 from account A to account B. It must fully happen or not at all — never "left A but never arrived at B."

The four letters, on that transfer Atomicity — the debit and the credit are one indivisible unit; if the credit fails, the debit is rolled back. No half-transfers.
Consistency — the transaction moves the database from one valid state to another, respecting all rules (balances stay non-negative, the books still balance).
Isolation — if two transfers run at once, each behaves as if it ran alone; neither sees the other's half-finished state.
Durability — once the database says "done," it survives a crash or power loss, because it's been written to permanent storage.

Going deeper

Isolation comes in levels (read committed → repeatable read → serializable) that trade strictness for speed; weak levels allow anomalies like dirty reads (seeing another transaction's uncommitted data). The data-engineering punchline: classic data lakes had no ACID — a Spark job that died mid-write could leave half-written files and a corrupt table. Table formats (Delta/Iceberg/Hudi) brought ACID to the lake. So the claim "Delta gives you ACID" only means something once you know what ACID is.

What to expect in practice
  • Assume ACID from OLTP databases and from lakehouse table formats (Delta/Iceberg/Hudi).
  • Don't assume it from plain lake files, raw CSV/JSON dumps, or naive Spark writes.
  • Matters most with concurrent writers, partial-failure recovery, and anything financial.

Why we care: it's the baseline reliability guarantee — the lakehouse pitch, CDC correctness, and migration safety all reference it.

CAP theoremWhy distributed databases must choose

Plain version. When data is copied across multiple machines and the network between them breaks (a "partition"), you can keep only one of two things: every node still answers (Availability) or every node agrees on the latest value (Consistency). The third letter, Partition tolerance, isn't really optional — networks fail in the real world — so the genuine choice is C or A, but only during a partition.

Concrete picture. Two clerks share one ledger by phone; the line drops. Clerk B must either refuse to answer until it's back (chose Consistency — never a stale number) or answer from what they last knew (chose Availability — possibly out of date). While the line is down, they can't do both.

Example A CP store (traditional SQL DB, HBase, ZooKeeper) blocks or errors during a partition to avoid serving wrong data. An AP store (Cassandra, DynamoDB by default) keeps answering and reconciles the copies afterward — that "afterward" is eventual consistency.

Going deeper

The most common misunderstanding is "pick 2 of 3." In practice P is mandatory, so CAP only bites during a network failure, and it reduces to C-vs-A. When the network is healthy, CAP says nothing — which is exactly the gap PACELC fills next.

When to choose which
  • CP (consistency first) for money, inventory, bookings — serving a wrong value is worse than being briefly unavailable.
  • AP (availability first) for high-volume feeds, telemetry, carts, social — staleness is tolerable, uptime matters more.
  • Remember: the choice only bites during a network partition; healthy-state behavior is PACELC's job.

Why we care: it's the foundational reason different databases behave differently, and the setup interviewers use to check that you really understand eventual consistency.

PACELCCAP, plus what happens when nothing's broken

Plain version. CAP only describes the failure case. PACELC adds the normal case. Read it as a sentence: if Partition (P) → choose Availability or Consistency (A/C) — that's CAP — else (E), in normal operation → choose Latency or Consistency (L/C).

Concrete picture. Same clerks, line working: do you make the customer wait while you phone the other city to confirm the latest balance (latency sacrificed for consistency), or answer instantly from your own copy (fast, but maybe seconds stale)?

Example DynamoDB / Cassandra are PA/EL: favor availability under partition and low latency normally, accepting eventual consistency. A strongly-consistent store is PC/EC: always correct, accepting blocking and higher latency.

Going deeper

The real insight: even with zero failures, keeping copies in sync costs latency — every confirm-with-the-other-replica is a round trip. The A/C trade only shows up during incidents; the L/C trade is the one you feel every single day. That's why PACELC is the more honest, complete framing.

How to use the framing
  • Describe a store as two letters: the P→A/C (failure) choice and the E→L/C (normal) choice.
  • Favor EL (latency) for user-facing reads that tolerate slight staleness; favor EC (consistency) when every read must be current.
  • Use it to explain why a "fast, globally distributed" store is usually eventually consistent.

Why we care: naming the "else, latency-vs-consistency" half unprompted is a clear seniority signal — it shows you think about the steady state, not just outages.

Consistency modelsStrong vs eventual, concretely

Plain version. Strong consistency: the instant a write succeeds, every reader sees the new value. Eventual consistency: readers may see the old value for a while, but if writes stop, everyone converges to the same value.

Analogy. Strong = a group chat where nobody can speak until everyone has received the last message. Eventual = email — your message reaches people at slightly different times, but everyone gets it.

Example You change your profile photo. Strong: it's the new photo for everyone immediately. Eventual: some friends still see the old one for a few seconds until the replicas catch up.

Going deeper

Eventual consistency is the price you pay for the Availability and Latency you bought in CAP/PACELC. It has useful in-between guarantees: read-your-writes (you always see your own changes) and monotonic reads (you never see time go backwards). In analytics this shows up as replica lag — your dashboard reads a copy that's a few minutes behind the source, so "today's number" wobbles.

When to use which
  • Strong for transactional correctness — balances, inventory counts, "did my write take?"
  • Eventual when you need scale and low latency and brief staleness is harmless — most analytics tolerates it via replica lag.
  • Tell the client: a dashboard a few minutes behind is usually fine; a wrong bank balance never is.

Why we care: it turns the abstract CAP/PACELC trade into the concrete "why is my dashboard slightly stale?" conversation you'll have with clients constantly.

IdempotencyWhy retries don't double-count

Plain version. An operation is idempotent if running it twice has the same effect as running it once. This matters enormously because pipelines retry on failure, and a retry must not create duplicate data.

Analogy. Pressing the "floor 3" elevator button five times still just takes you to floor 3 (idempotent). A vending machine that drops a snack on every press is not idempotent.

Example INSERT a row on every run → not idempotent (a re-run creates duplicates).
MERGE/upsert on a key, or "delete the day's partition then re-insert it" → idempotent (a re-run lands on the same result).

Going deeper

You achieve it with unique keys, upserts, deterministic partition overwrites, or dedup on a natural key. It's the practical answer to at-least-once delivery (next card): if the system might hand you the same record twice, idempotent processing makes the duplicate harmless.

How to use it in practice
  • Default every task to idempotent: upsert on a key, or delete-then-reinsert a partition.
  • Append-only only for immutable event logs where duplicates are impossible by design.
  • Test it: a task you can safely re-run is a task you can safely retry and backfill.

Why we care: it's what makes retries, backfills, and re-runs safe — one of the strongest reliability signals you can give, and central to incremental dbt models and CDC.

Delivery guaranteesAt-least-once vs exactly-once

Plain version. How a system promises to deliver each message/record. At-most-once: might lose some, never duplicates. At-least-once: never loses, might duplicate. Exactly-once: each record takes effect once — no loss, no duplicates (the hard, expensive one).

Analogy. Mailing checks. At-most-once: mail it and hope. At-least-once: keep resending until confirmed (it might arrive twice). Exactly-once: resend, but the recipient tracks check numbers and ignores a duplicate — so the effect happens once.

Example A Kafka consumer crashes after processing a message but before recording that it did. On restart it reprocesses — that's at-least-once. Make the processing idempotent and the reprocess is harmless: effectively exactly-once.

Going deeper

True end-to-end exactly-once is usually at-least-once delivery + idempotent processing (or a transactional sink). The lesson: "exactly-once" is almost always achieved at the effect level, not by the network magically delivering once. That's why the previous card matters so much.

When to use which
  • At-most-once: only for lossy-OK signals (best-effort metrics, sampling).
  • At-least-once + idempotent sink: the default for data pipelines — never lose, dedupe on landing.
  • True exactly-once: reserve for cases where a duplicate is genuinely unacceptable and you'll pay the extra complexity.

Why we care: it lets you cut through vendor "exactly-once!" claims and explain what actually guarantees correctness — idempotency at the sink.

D · Moving and transforming data

ETL vs ELTWhere the transforming happens

Plain version. ETL = Extract, Transform, then Load — clean and reshape data before it lands in the warehouse. ELT = Extract, Load, then Transform — dump raw data into the warehouse first, then transform it there with SQL.

Analogy. ETL = prep your vegetables before putting them in the fridge. ELT = put the groceries in the fridge as-is and prep them when you cook.

Example ETL (old): a separate server reshapes data and only the clean result enters the warehouse. ELT (modern): Fivetran loads raw source tables; dbt runs SQL inside the warehouse to build clean models from them.

Going deeper

ELT won because cloud compute and storage are cheap and elastic: keep the raw data forever, and you can re-transform anytime, audit the source, or fix transformation logic without re-extracting from the source system. dbt is the "T" in ELT — that's its entire reason for existing.

When to use which
  • ELT by default on a cloud warehouse — cheap elastic compute, keep raw, transform in SQL/dbt.
  • ETL when you must transform before landing: masking PII you can't store raw, compliance, or heavy non-SQL processing.
  • Hybrid is common: light pre-load masking, then ELT for everything else.

Why we care: ELT is the shape of the entire modern data stack (Part 3); understanding why the T moved last explains every tool choice after it.

CDCCopying only what changed

Plain version. Change Data Capture continuously copies only the rows that changed in a source database — inserts, updates, deletes — into your warehouse, instead of re-copying the whole table every time.

Analogy. Instead of re-photocopying an entire book every night, you only photocopy the pages that were edited that day.

Example A customer updates one order in the source app. CDC captures just that single change and applies it to the warehouse copy within seconds to minutes — the rest of the table is untouched.

Going deeper

Three mechanisms: log-based (read the database's transaction log, e.g. Debezium — lowest load on the source, and it catches hard deletes), timestamp/query (poll an updated_at column — simple, but misses deletes and needs reliable timestamps), and trigger-based (DB triggers record changes — complete, but adds write overhead). It leans on idempotency, watermarks (how far you've consumed), and tombstones (delete markers).

When to use which mechanism
  • Log-based (Debezium): the default for serious CDC — low source impact, catches hard deletes, low latency.
  • Timestamp/query (updated_at): simple sources you control, where soft-deletes are acceptable.
  • Trigger-based: only when you can't read the transaction log and can absorb the write overhead.

Why we care: CDC is how you keep a warehouse fresh and how a migration keeps the target in sync with the live source during the cutover window (Part 2).

Batch vs streamingChunks on a schedule vs event-by-event

Plain version. Batch processes data in scheduled chunks (hourly, nightly). Streaming processes each event continuously as it arrives (sub-second to seconds).

Analogy. Batch = doing all the laundry on Sunday. Streaming = washing each item the moment you take it off.

Example A nightly sales roll-up is fine as batch. A fraud check that must score a card swipe before it's approved must be streaming.

Going deeper

Micro-batch is the middle ground (tiny, frequent batches). Streaming introduces windowing (tumbling/sliding/session), event-time vs processing-time (when it happened vs when you saw it), and watermarks (how long to wait for stragglers before closing a window). Most analytics is perfectly fine as batch.

When to use which
  • Batch by default when the SLA is minutes-to-hours — cheaper, simpler, easier to reason about.
  • Micro-batch for near-real-time needs (every few minutes).
  • Streaming only when a decision must happen per-event within seconds: fraud, alerting, real-time personalization.

Why we care: "do you actually need streaming?" is a maturity test — the senior answer is usually "no, batch meets the SLA at a fraction of the complexity."

E · Modeling data for analytics

GrainThe single most important modeling word

Plain version. The grain of a table is the precise definition of what one row means. You decide it and state it out loud before building anything.

Analogy. Before drawing a chart you must answer "what is one dot here?" — one sale? one customer? one day? Everything downstream depends on that answer.

Example Grain = "one row per order line item." Then an order-level total is an aggregation of several rows — and you would never mix per-line rows and per-order rows in the same table.

Going deeper

Declaring grain up front prevents the classic disaster: fan-out, where a careless join multiplies rows and silently doubles your revenue number. A fact table has exactly one stated grain and you never mix grains within it. "Declare the grain first" is the phrase that signals you think like an analytics engineer.

How to choose the grain
  • Pick the finest grain the source supports and the business needs — you can always aggregate up, never down.
  • One grain per fact table; never mix (e.g. order-line rows and order-header rows together).
  • State it in one sentence before writing SQL: "one row per ___."

Why we care: it's the most repeated interview signal there is, and the root cause of most real-world metric bugs. Over-invest here (Part 3).

Star schemaFacts in the middle, dimensions around

Plain version. Organize analytics data into facts (the measurements/events — sales, clicks) and dimensions (the descriptive context — customer, product, date). Put the fact in the center with dimensions radiating out and you get a star.

Analogy. A sentence. The fact is the verb + number ("sold 5"); the dimensions are the who/what/when/where ("customer Amy bought product X on June 1 in Texas").

Example fct_sales(date_key, product_key, customer_key, amount) joins out to dim_date, dim_product, and dim_customer.

Going deeper

Facts are numeric and very long (many rows); dimensions are wide and descriptive (many columns, fewer rows). Conformed dimensions are shared across multiple facts — the same dim_customer for both sales and returns — so metrics line up across the business. A star keeps dimensions denormalized (fast, simple); a snowflake normalizes them (less redundancy, more joins). BI tools are built to love stars.

When to use which shape
  • Star (denormalized dims) by default — best BI performance and simplicity.
  • Snowflake (normalized dims) only when a dimension is huge or shared and normalizing meaningfully cuts redundancy/maintenance.
  • One Big Table when a single BI tool performs best on one flat, wide table.

Why we care: it's the default analytics model and a stock prompt: "design a star schema for X — what's the grain?"

SCD Type 2Keeping history when a value changes

Plain version. A Slowly Changing Dimension, Type 2 keeps history: when a descriptive attribute changes, instead of overwriting it you add a new row and keep the old one, so you can still report "as it was at the time."

Analogy. A customer moves from Texas to New York. Type 1 overwrites the address — history erased, as if she always lived in NY. Type 2 keeps both rows with date ranges, so a 2024 sale still correctly shows Texas.

Example dim_customer gains valid_from, valid_to, and is_current. On a change, the old row's valid_to is closed and is_current set to false; a new current row is inserted.

Going deeper

Type 2 needs a surrogate key (next card) so one customer can have several rows, plus effective/expiry dates and a current flag. In dbt this is exactly what a snapshot does. Contrast Type 0 (never changes), Type 1 (overwrite, no history), Type 3 (keep one prior value in a column).

When to use which SCD type
  • Type 1 (overwrite) when only the current value matters — no history needed.
  • Type 2 (history rows) when you must report point-in-time / "as it was then."
  • Type 3 (prior-value column) when you only ever need the single most recent previous value.

Why we care: it's the most-tested advanced modeling mechanic and maps straight onto the business question "do you need history or just the current value?"

Surrogate vs natural keysThe warehouse's own IDs

Plain version. A natural key is a business identifier (email, SSN, order number). A surrogate key is a meaningless, system-generated id (1, 2, 3…) the warehouse assigns instead.

Analogy. Your name is a natural key — it can change and can collide with someone else's. Your customer number is a surrogate — stable, unique, and deliberately meaningless.

Example dim_customer gets customer_sk = 1001 even though the business id is cust_id = 'ACME-TX'. Fact tables reference 1001, not the messy business code.

Going deeper

Surrogates decouple the warehouse from source-system quirks (re-keying, format changes), are compact and fast to join on, and are what make SCD Type 2 possible — the same natural key can map to several surrogate-keyed rows across time.

When to use which
  • Surrogate keys on every dimension in a Kimball model — always; keep the natural key as an attribute for lineage.
  • Natural key alone only for tiny, static lookups where a surrogate adds no value.
  • Required whenever you do SCD Type 2 (one natural key → many rows over time).

Why we care: it's foundational plumbing for dimensional modeling; not using surrogates is a common junior tell.

Semantic layerDefining "revenue" exactly once

Plain version. A semantic (metrics) layer is one central place where metrics like "revenue" or "active user" are defined once, so every dashboard and report computes them identically.

Analogy. A shared dictionary. Instead of each team inventing its own definition of "revenue," everyone looks it up in one agreed place.

Example Define revenue = SUM(amount) WHERE status = 'paid' once. Tableau, Power BI, and an analyst's notebook all ask the layer and get the identical number.

Going deeper

Sometimes called "headless BI": the BI tools query through the layer rather than each re-implementing the math. It kills metric sprawl — the situation where finance and ops quote different revenue figures and nobody can say which is right. Implementations: dbt Semantic Layer / MetricFlow, Cube, LookML.

When it's worth it
  • Build one once more than one BI tool or team computes the same metrics and they disagree — or metrics need versioning/certification.
  • Skip it for a single small team on a single BI tool — the dbt marts are enough.
  • Sweet spot: mid-size orgs with metric sprawl across finance, ops, and product.

Why we care: it's the direct answer to the very common prompt "make finance and ops agree on revenue" (Part 3).

Learn it — the modernization story

Almost every platform-modernization engagement is the same arc. A client runs analytics on something old — an on-prem warehouse like Teradata or Oracle, or a sprawl of SQL Server and Excel. The three architectures — and how you actually choose

Foundations B taught these by analogy (a library, a storage unit, a storage-unit-with-a-librarian). Here is the consultant's version: how the choice is actually made in a room with a client.

  • Data warehouse — structured tables, strict schema enforced on write, tuned for SQL and BI. Choose it when the work is governed reporting on structured data and the client values simplicity over flexibility. Classic fit: finance, regulated reporting. (Snowflake, BigQuery, Redshift.)
  • Data lake — raw files (Parquet/JSON/CSV) on cheap object storage (S3/GCS/ADLS), schema imposed only when read. Choose it when you need to land huge or messy or semi-structured data cheaply, or feed ML. The risk is a "data swamp" nobody trusts. Rarely the whole answer by itself anymore.
  • Lakehouse — lake-style cheap storage plus a table format that re-adds warehouse guarantees (ACID, schema, time travel). Choose it when the client has both BI and ML, or large/varied data, and wants one governed copy serving both. (Databricks; or BigQuery/Snowflake reading open table formats.)
Decision heuristic you can say out loud"Structured data, SQL/BI only, governance-first → warehouse. Also have ML, semi-structured, or large raw volumes → lakehouse so there's one copy for both. Pure lake on its own is mostly a landing zone, not a destination." Then name the two org patterns that are orthogonal to this: data mesh (domains own their data products) and data fabric (a metadata layer over scattered sources) — neither is a thing you install, both are operating models.

What's physically underneath — and why cloud warehouses are different

The defining shift (Foundations B) is separating storage from compute. In an old warehouse the disks and processors were one box; in the cloud the data sits in object storage and you rent query engines separately. Two consequences you'll exploit constantly: you can query files in place without loading them first, and you can scale or suspend compute independently of the data.

Worked example · query a file in place, then see columnar pruning

Write the orders out as a Parquet file (a columnar format), then query it directly — no load step. This is the lake/lakehouse pattern in miniature.

-- export to columnar Parquet, then query the FILE directly
COPY orders TO 'orders.parquet' (FORMAT parquet);

SELECT status, count(*) FROM 'orders.parquet' GROUP BY status;
-- DuckDB reads only the `status` column from the file (column pruning),
-- not order_id/customer_id/etc. On a 50-column table that's ~1/50th the I/O.

That "read only the columns you asked for" behavior is why columnar storage (Part 1's Parquet/ORC) makes analytics cheap, and why you never run these scans on the row-oriented OLTP database.

Table formats — turning files into real tables

A pile of Parquet files isn't a table: two writers can corrupt each other, there's no transaction, no safe schema change, no history. A table format — Iceberg, Delta, or Hudi — adds a metadata layer over the files that provides exactly those things. It's the single technology that makes a lakehouse possible.

Worked example · what "time travel" and ACID buy you

Conceptually, a table format keeps a log of snapshots. Every write creates a new immutable version; readers always see one complete version, and you can query the past:

-- Delta/Iceberg-style operations (syntax varies by engine):
UPDATE orders SET status = 'paid' WHERE order_id = 1002;   -- creates snapshot v2

SELECT * FROM orders VERSION AS OF 1;   -- read yesterday's v1 — the refund is still there
SELECT * FROM orders;                  -- current v2 — now 'paid'

Because every reader sees a whole snapshot, a job that dies halfway never exposes a half-written table (atomicity), two pipelines can write without clobbering each other (isolation), and you can audit or roll back to any past version (time travel). That's ACID — the thing raw lake files lacked.

The trade you'll be asked about

Copy-on-write rewrites whole files on update (fast reads, slow writes — good for read-heavy BI). Merge-on-read writes small delta files and merges them at query time (fast writes, slightly slower reads — good for streaming/CDC ingestion). Picking between them is a real lakehouse tuning decision.

"So what is an Iceberg table?" — the question from your standup

This trips up almost everyone, because the name makes it sound like a new kind of database. It isn't. The clearest way in is to compare it to a "normal" table in Snowflake or BigQuery and ask one question: who owns the data, and who's allowed to read it?

The one ideaA native Snowflake table (or native BigQuery table) stores your data inside the warehouse, in that vendor's secret internal format. Only Snowflake can read a Snowflake table; only BigQuery can read a BigQuery table. An Iceberg table stores the exact same data as plain Parquet files plus open metadata, in your own cloud storage bucket (S3/GCS/ADLS) — in a published, non-secret layout that any engine can read. So Iceberg isn't a database; it's an agreed standard for laying a table out as open files so many engines can share it.

Analogy. A native warehouse table is a document saved in a proprietary app's format — only that app opens it, and if you ever leave the app, getting your document out is painful. An Iceberg table is the same document saved as a PDF in a shared Dropbox folder you own: it sits in your folder, in an open format, and Word, Preview, Chrome — any program — can open it. Snowflake, BigQuery, Spark, Trino, and Databricks are just different "programs" all opening the same PDF.

Your work situation, decoded

You said: the data "lives in Snowflake," you hear about "Iceberg tables," and there's also "a BigQuery copy." Here's what's almost certainly going on:

The old world (why there's a "BQ copy")

Because a native Snowflake table can only be read by Snowflake, when the BigQuery users needed that same data, someone built a pipeline to copy it into BigQuery's native format. Now you have two copies of the same data, in two secret formats, in two warehouses — double the storage bill, plus a sync job that can lag or drift, so the two "copies" sometimes disagree. That copy exists purely because neither warehouse can read the other's tables.

What Iceberg changes

Store the data once as an Iceberg table in object storage. Point both Snowflake and BigQuery at it. Now there is one table, no copy, no sync job, no drift — each engine reads the same Parquet files and the same metadata. That is exactly the problem Iceberg was invented to kill, which is why you hear about it constantly: companies are migrating their "Snowflake table + BigQuery copy" pattern to "one Iceberg table, both engines read it."

The cost lever — partitioning and bytes scanned

On a cloud warehouse you often pay per byte scanned (BigQuery) or per second of compute (Snowflake). The biggest lever on both is making queries read less data. Partitioning splits a table into folders by a column — usually a date — so a filter on that column skips everything else (partition pruning).

Worked example · partition pruning = money saved
-- write orders partitioned into one folder per day
COPY orders TO 'orders_part'
  (FORMAT parquet, PARTITION_BY (order_date));
-- creates orders_part/order_date=2026-06-01/...  and  .../2026-06-02/...

SELECT sum(qty) FROM read_parquet('orders_part/**/*.parquet', hive_partitioning=true)
WHERE order_date = DATE '2026-06-02';
-- only the order_date=2026-06-02 folder is opened. On a billion-row,
-- 3-year table, filtering one day reads ~1/1000th of the data = ~1/1000th the cost.

Don't over-partition. Partition on a low-cardinality column you filter on (date), not a high-cardinality one (customer_id) — the latter creates millions of tiny files and makes scans slower (the "small-file problem"). For a second common filter, use clustering / Z-ordering (sorting data so similar values sit together) instead of another partition level.

02Data Migration & Validation

Your stated specialty. Go deeper here than anywhere else — it's both an interview edge and a sellable Eigenbasis methodology.

Learn it — a migration is two jobs, and the second is the hard one

A data migration is (1) move the data from an old system to a new one, and (2) prove it arrived correctly. Job 1 is mostly tooling. Job 2 — reconciliation — is where projects actually succeed or fail, where clients are most afraid, and where you'll differentiate. Anyone can copy a table; the consultant who can demonstrate, row by row, that 40 million records moved without a single silent corruption is the one who gets trusted with the cutover. This chapter teaches that proof.

The lifecycle — what you'd whiteboard

Every migration walks the same eight steps. Be able to draw and narrate them:

  1. Discovery & assessment — inventory the sources, profile the data, map dependencies and lineage, get volumetrics. You can't move what you haven't catalogued.
  2. Schema mapping & conversion — map source types to target types (this is where silent corruption is born), decide surrogate keys, handle source quirks.
  3. Initial (historical) load — the big bulk backfill of all existing data.
  4. CDC / incremental sync — keep the target current as the source keeps changing during the project (Foundations D).
  5. Validation & reconciliation — the proof, below.
  6. Cutover — freeze the source, apply the final delta, switch consumers to the target.
  7. Rollback plan — always have the "if it goes wrong, here's how we revert" written before cutover.
  8. Decommission — retire the legacy system once the new one is proven in production.

The reconciliation ladder — with real SQL

You don't reconcile a 40-million-row table by eyeballing it. You climb a tiered ladder, cheapest and broadest first, narrowing to expensive precise checks only where needed. Here is each rung as runnable SQL, against the shop dataset.

-- setup (run once): load the dataset, then make a "legacy source" to compare
-- against. Load shop_seed.sql first (or paste the shop tables), then:
CREATE TABLE orders_src AS SELECT * FROM orders;
-- orders = the migrated TARGET, orders_src = the legacy SOURCE. Identical for
-- now, so every check below reports "match"; further down we inject a bug.
Rung 1 · row counts (control totals) — catch gross loss/duplication
SELECT
  (SELECT count(*) FROM orders_src) AS src_rows,
  (SELECT count(*) FROM orders)     AS tgt_rows,
  (SELECT count(*) FROM orders) -
  (SELECT count(*) FROM orders_src)   AS diff;   -- must be 0

Fast, runs on any size, catches whole batches dropped or a bad join that doubled rows. But equal counts prove nothing about values — two tables can have the same count and totally different data.

Rung 2 · column aggregates & checksums — catch shifted values cheaply
-- compare per-column summaries; any mismatch flags a column to investigate
SELECT count(*) n, sum(qty) sum_qty, min(order_date) min_d,
       max(order_date) max_d, count(DISTINCT customer_id) n_cust
FROM orders_src
UNION ALL
SELECT count(*), sum(qty), min(order_date),
       max(order_date), count(DISTINCT customer_id)
FROM orders;
-- sum_qty differs? a value was truncated/rounded. max_d off by hours? a timezone shift.

Still cheap (one scan each), and now sensitive to value drift. A differing sum_qty means numbers changed; a differing max(order_date) often means a timezone bug. This rung is where most real corruption first shows up.

Rung 3 · row-level hash diff on the primary key — the decisive proof

This is the technique that wins trust. Give every row a one-value fingerprint (a hash of all its columns), then join the two tables on the primary key and compare the fingerprints. Any row whose hash differs — or is missing on one side — is flagged, and the PK tells you exactly which row and how it differs.

-- per-row fingerprint: cast every column to text, concatenate, hash.
-- COALESCE so a NULL doesn't make the whole hash NULL; a delimiter so
-- '1'+'23' can't collide with '12'+'3'.
WITH src AS (
  SELECT order_id,
    md5(concat_ws('|', customer_id, product_id, qty,
                   order_date, coalesce(status,'∅'))) AS h
  FROM orders_src),
tgt AS (
  SELECT order_id,
    md5(concat_ws('|', customer_id, product_id, qty,
                   order_date, coalesce(status,'∅'))) AS h
  FROM orders)
SELECT
  coalesce(s.order_id, t.order_id) AS order_id,
  CASE WHEN s.order_id IS NULL THEN 'extra_in_target'      -- in target only
       WHEN t.order_id IS NULL THEN 'missing_in_target'    -- in source only
       ELSE 'changed' END AS issue                       -- same PK, different content
FROM src s
FULL OUTER JOIN tgt t ON s.order_id = t.order_id
WHERE s.h IS DISTINCT FROM t.h;          -- IS DISTINCT FROM = NULL-safe inequality
-- empty result = every row matches exactly. That's your proof.

The FULL OUTER JOIN aligns rows by primary key; IS DISTINCT FROM returns the rows whose fingerprints disagree (it treats NULLs safely, so a missing side counts as different). Note why comparing only the order_id wouldn't work: a row whose values changed still has the same PK on both sides, so you must compare the hash, not the key.

On huge tables you don't hash all of it every night — you tier: row counts + checksums on everything (cheap), full row-hash diff on a sampled or recent slice (precise). That cost/coverage trade is exactly what a senior reconciliation answer names.

Rung 4 · schema & type validation — catch the corruption at its source
-- compare column names, types, nullability between source and target catalogs
SELECT column_name, data_type, is_nullable
FROM information_schema.columns WHERE table_name = 'orders_src'
EXCEPT
SELECT column_name, data_type, is_nullable
FROM information_schema.columns WHERE table_name = 'orders';
-- a VARCHAR(50)→VARCHAR(20) mapping, or NUMERIC(10,2)→NUMERIC(10,0), shows up here
-- BEFORE it silently truncates data during the load.

This is the demo that sells the methodology. With orders_src already created in setup (the correct "legacy" copy), corrupt one value in the target, then re-run Rung 3 — and watch it point at the exact row:

Check yourself

  2. Your nightly SUM(amount) matches but SUM(amount_cents) built from it is off by a few cents across the table. Which failure class, and why did the dollar sum hide it?
  3. Why COALESCE every column before hashing in Rung 3? What breaks without it?
  4. A 5-billion-row table can't be fully hash-diffed nightly. Describe the tiered strategy you'd propose.
  5. Name three failure classes you'd deliberately test for on a migration out of an old Oracle system handling international customer names and money.
Answers

1) No — equal counts only rule out gross loss/duplication. Values could be entirely wrong with identical counts. You still need checksums and a row-level hash diff.

2) Rounding/precision — each row rounded by a fraction of a cent; summing dollars averages the error away, but the per-row cents reveal it. Aggregate checks can mask per-row drift, which is why Rung 3 exists.

3) NULLs make concat return NULL, so the whole row-hash becomes NULL and every NULL-containing row looks "equal" to every other — corruption hides. COALESCE to a sentinel keeps each row's fingerprint meaningful. (And a delimiter prevents '1'+'23' colliding with '12'+'3'.)

4) Run cheap rungs (counts + column checksums) on the whole table every cycle; run the full row-hash diff on a recent partition (e.g. last 7 days) plus a random sample of history; full historical hash diff once at cutover. Cost where it's cheap, precision where it matters.

5) Charset/collation (the names), rounding/precision and truncation (the money and names), timezone (order timestamps). Each maps to a concrete check you'd write.

03Analytics Engineering & the Modern Data Stack

The dbt-centered core of most billable work — and the densest interview territory.

Learn it — why "analytics engineering" exists

Once cloud warehouses made compute cheap and elastic, the old habit of transforming data before loading it (ETL) flipped: load the raw data first, then transform it inside the warehouse with SQL (ELT). That one change created a new role — the analytics engineer — and a tool that owns the "T": dbt. Most of what you'll bill for is building and modeling that transformation layer well. This chapter takes the running shop tables and turns them into the clean, modeled, business-facing layer a company actually runs on.

The dbt mental model

dbt is deceptively simple: a dbt model is just a SELECT statement in a .sql file. dbt wraps it in CREATE TABLE/VIEW, runs it against the warehouse, and — crucially — lets models reference each other with {{ ref('other_model') }}. Those refs build a dependency graph (a DAG) so dbt knows what to build in what order, and gives you free lineage, tests, and docs.

-- models/staging/stg_orders.sql — one staging model per source, cleaned & typed
SELECT
  order_id,
  customer_id,
  product_id,
  qty::int            AS quantity,
  order_date::date     AS order_date,
  lower(status)       AS status
FROM {{ source('shop', 'orders') }}    -- source() = declared raw input

You then layer models in three tiers — the standard answer to "how do you structure a dbt project?":

staging (stg_) 1:1 with each source · rename, cast, light clean — never join here → intermediate (int_) business logic · joins · reusable building blocks → marts (fct_/dim_) dimensional, business-facing tables, organized by domain
Materializations — the one knob to knowEach model is built as a view (default; cheap, always fresh), a table (expensive query or many readers — compute once), incremental (huge fact tables — process only new rows), ephemeral (inlined as a CTE), or a snapshot (captures history → SCD2). Choosing among them is the "view vs table vs incremental, when?" interview question.

Worked example — build the star schema

Before any SQL, the most important sentence in analytics engineering: declare the grain. "One row of fct_orders = one product line on one order." Everything follows from that. Then facts (the numbers) go in the center, dimensions (the descriptive context) around them.

Worked example · fct_orders + dim_customer + dim_product
-- marts/dim_customer.sql — a surrogate key + descriptive attributes
SELECT
  row_number() OVER (ORDER BY customer_id) AS customer_sk,  -- surrogate key
  customer_id AS customer_nk,                                -- natural key (kept for lineage)
  name, state
FROM {{ ref('stg_customers') }};

-- marts/fct_orders.sql — grain: one row per order line; numbers + foreign keys to dims
SELECT
  o.order_id,
  c.customer_sk,                          -- join to the dim's surrogate key
  p.product_sk,
  o.order_date,
  o.quantity,
  o.quantity * p.unit_price AS line_revenue,
  o.status
FROM {{ ref('stg_orders') }} o
JOIN {{ ref('dim_customer') }} c ON o.customer_id = c.customer_nk
JOIN {{ ref('dim_product') }}  p ON o.product_id  = p.product_nk;

Why the surrogate keys (customer_sk)? They decouple the warehouse from messy source IDs, make joins compact, and — next section — let one customer have multiple rows over time. Why declare grain? Because if you accidentally joined something one-to-many here, row counts would inflate and SUM(line_revenue) would silently double. That bug is fan-out, and grain discipline is how you prevent it.

Worked example — SCD Type 2 (keeping history)

Amy moves from TX to NY. If you just UPDATE her row (SCD Type 1), every past sale now looks like it happened in NY — history is falsified. SCD Type 2 instead keeps both rows with date ranges, so a June sale still correctly shows TX.

Worked example · a Type-2 dimension by hand (what a dbt snapshot automates)
-- dim_customer_history: one row per (customer, period-of-validity)
CREATE TABLE dim_customer_history (
  customer_sk   int,      -- unique per VERSION, not per customer
  customer_nk   int,      -- the stable business id
  state         text,
  valid_from    date,
  valid_to      date,     -- NULL/9999 = still current
  is_current    boolean
);
-- Amy starts in TX:
--  (1, 1, 'TX', 2026-01-01, 9999-12-31, true)
-- Amy moves to NY on 2026-06-15 → close the old row, open a new one:
--  (1, 1, 'TX', 2026-01-01, 2026-06-15, false)
--  (7, 1, 'NY', 2026-06-15, 9999-12-31, true)
-- A June-1 order joined on the date range still maps to the TX row. History preserved.

In dbt you don't hand-code this — a snapshot does it for you:

-- snapshots/customer_snapshot.sql
{% snapshot customer_snapshot %}
{{ config(unique_key='customer_id', strategy='check',
          check_cols=['state']) }}
SELECT * FROM {{ source('shop','customers') }}
{% endsnapshot %}
-- dbt adds dbt_valid_from / dbt_valid_to automatically on each run.

-- an incremental fact: on the first run build everything; after that, only new days

And the semantic layer idea — define the metric one time so finance and ops can never disagree:

-- revenue = paid line revenue only. Refunds excluded, on purpose, in one place.

FROM {{ ref('fct_orders') }}

-- In MetricFlow/Cube this becomes a named metric with that filter baked in,
-- so nobody re-implements "revenue" and gets a different number.
Check yourself
  1. Why is "declare the grain" the first thing you do, and what bug does skipping it cause?
  2. A model is queried by 40 dashboards and takes 90 seconds to run. view, table, or incremental?
  3. The business asks: "what % of our revenue last year came from customers who were in Texas at the time of purchase?" Which modeling technique makes this answerable, and why would SCD Type 1 give the wrong answer?

Answers

1) It fixes what one row means, so joins and aggregations stay correct. Skip it and a one-to-many join inflates rows (fan-out), silently multiplying your metrics.

2) table — the query is expensive and has many readers, so pay the compute once and let dashboards read the stored result. (Incremental only if it's also large and append-mostly.)

3) SCD Type 2 on the customer dimension — it preserves the state-at-time-of-purchase. Type 1 overwrites to the current state, so every historical order would be re-attributed to where the customer lives now, not then.

4) It builds the dependency DAG (correct build order), and gives you lineage, environment-aware table names (dev vs prod), and the ability to rebuild only what changed.

04Data Quality, Reliability & Observability

Trust is the product. This is where reliability-minded consultants get repeat business.

Learn it — why quality is a deliverable, not an afterthought

A pipeline that produces wrong numbers confidently is worse than no pipeline. The moment a client catches one bad figure on a dashboard, they distrust all of them — and the project. So "data quality" isn't a nice-to-have; it's the thing that makes everything else worth paying for. The craft has two halves: knowing what can be wrong (the dimensions and pillars) and catching it automatically (tests and monitors that fail loudly).

Two vocabularies you'll be asked to recite

The 7 quality dimensions describe ways a value can be wrong: accuracy, completeness, consistency, timeliness, validity, uniqueness, integrity. The 5 observability pillars describe what you monitor about a table to notice trouble — and you should be able to give a concrete alert for each:

PillarA concrete monitor
FreshnessAlert if orders hasn't updated in 25h against a 24h SLA.
VolumeAlert if today's row count is <50% or >200% of the trailing-7-day average.
SchemaAlert if a column is dropped, renamed, or changes type upstream.
DistributionAlert if the null-rate of email jumps, or amount's mean shifts beyond a baseline.
LineageWhen something breaks, trace which upstream model/source caused it and what's downstream.

Worked example — catch problems with dbt tests

The cheapest, highest-leverage quality work is declarative tests on your models. In dbt you attach them in YAML and they run as part of the build:

Worked example · schema tests + a custom business rule
# models/marts/_marts.yml — generic tests, declared once
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests: [unique, not_null]              # uniqueness + completeness
      - name: customer_sk
        tests:
          - not_null
          - relationships:                      # integrity: every FK exists in the dim
              to: ref('dim_customer')
              field: customer_sk
      - name: status
        tests:
          - accepted_values: { values: ['paid','refunded','pending'] }  # validity
-- tests/assert_no_negative_revenue.sql — a SINGULAR test = any SELECT that
-- should return ZERO rows. If it returns rows, the build fails.
SELECT order_id, line_revenue
FROM {{ ref('fct_orders') }}
WHERE line_revenue < 0;   -- revenue should never be negative

This is assertion-based testing: you state the rule explicitly. Its opposite, anomaly detection (tools like Monte Carlo), learns a statistical baseline and alerts on deviations — better coverage of the unknown, but more tuning and false positives. Real stacks use both: assertions for known rules, anomaly detection for the rest.

The pattern that stops bad data spreading

Circuit breakerPut quality gates between stages and halt the pipeline when one fails, so corrupt data never reaches the dashboards. In dbt this is literally dbt build: it runs a model, then its tests, and stops dependents if the tests fail. Bad rows that do slip through go to a quarantine / dead-letter table for inspection rather than into the marts. Pair this with data contracts (Governance chapter) — schema+semantics agreed and enforced at the producer boundary — and most quality incidents never happen.
Check yourself
  1. Give a monitor for each of the 5 pillars on a daily payments table.
  2. Which dbt generic test enforces "every customer_sk in the fact exists in the customer dim," and which quality dimension is that?
  3. When would you reach for anomaly detection instead of a fixed assertion?
  4. Where in an ELT flow do you put the circuit breaker, and what happens to the bad rows?
Answers

1) Freshness: updated within SLA. Volume: row count within expected band. Schema: columns/types unchanged. Distribution: null-rate and amount distribution stable. Lineage: upstream sources all fresh/passing.

2) relationships — that's the integrity (referential) dimension.

3) When you can't enumerate the rule — e.g. "is today's traffic weird?" — a learned baseline catches drifts you didn't anticipate, at the cost of tuning.

4) Between transform stages (run tests after each model, before its dependents build) — dbt build halts the branch; failing/quarantined rows go to a dead-letter table for triage instead of into the marts.

05Orchestration & Pipeline Design

How the pieces run, retry, and recover.

Learn it — the conductor of the pipeline

You've got ingestion, dbt, and quality tests. Something has to run them in the right order, on a schedule, retry the flaky bits, and recover when a step dies at 3am. That's an orchestrator. It models work as a DAG (directed acyclic graph) — tasks with dependencies — and handles scheduling, retries, backfills, and alerting. Airflow and Dagster are the two you'll meet.

Two philosophies — task-centric vs asset-centric

Airflow is task-centric: you define steps ("run this script, then that one") and their order. Dagster is asset-centric: you define the data products you want to exist ("this table, built from those tables") and it derives the steps. The asset view maps naturally onto a dbt project — your DAG is your tables.

Worked example · the same pipeline, both ways
# Dagster (asset-centric): declare WHAT should exist; deps are inferred from args
@asset
def raw_orders(): return extract_from_postgres("orders")

@asset
def fct_orders(raw_orders):          # depends on raw_orders automatically
    return run_dbt_model("fct_orders")

# Airflow (task-centric): declare the STEPS and wire the order yourself
extract = PythonOperator(task_id="extract", python_callable=extract_orders)
build   = BashOperator(task_id="build", bash_command="dbt run -s fct_orders")
extract >> build                          # extract THEN build

The two properties that make pipelines survivable

Idempotency + backfillsThe single most important pipeline property (Foundations C): running a task twice produces the same result as once. Achieve it by overwriting a partition or upserting on a key, never blind-appending. Why it matters: when a run fails and retries, or when you backfill (re-run history after fixing a bug), idempotent tasks just re-run safely — non-idempotent ones double your data. A good mental test: "if this 3am job fired twice, would I notice?" If yes, it's not idempotent yet.

Related vocabulary you should hold: retries (auto re-attempt transient failures), catchup (run all missed schedules after downtime), watermarks (track how far you've processed so you resume, not restart), and at-least-once delivery (Foundations C) — which is exactly why idempotency is non-negotiable.

Check yourself
  1. Contrast Airflow and Dagster in one sentence each. Which fits a greenfield dbt-centric platform, and why?
  2. Your daily job does INSERT INTO fct_orders SELECT … WHERE order_date = today. It got retried after a network blip. What went wrong, and how do you fix it?
  3. You fixed a transformation bug and need last quarter's numbers recomputed. What's the name for that, and what property must your tasks have for it to be safe?
Answers

1) Airflow = task-centric (define steps + order); Dagster = asset-centric (define the data products, steps inferred). Greenfield dbt platform → Dagster: the asset model mirrors your tables and gives data-aware lineage/partitions out of the box.

2) The retry ran the INSERT twice → today's rows are duplicated (not idempotent). Fix: make it an upsert/MERGE on order_id, or "delete today's partition then insert," so a re-run lands on the same result.

3) A backfill; tasks must be idempotent so re-running historical windows overwrites cleanly instead of duplicating.

06Reporting / BI Modernization

Where the modernization work becomes visible to the business.

Learn it — the layer the client actually sees

Everything upstream is invisible to executives; the dashboard is not. BI modernization is where trust is won or lost in public, and it has a few decisions that recur on every engagement.

Live query vs import/extract

A BI tool can either query the warehouse live on each interaction (always fresh, but every click costs a query and can be slow) or work off an imported extract / in-memory model (Power BI's VertiPaq, Tableau extracts — blazing fast, but as stale as the last refresh). The trade is freshness vs speed/cost. Rule of thumb: extracts for heavy executive dashboards refreshed a few times a day; live for operational dashboards that must be current.

Why BI tools love a star schema

The dimensional model from Part 3 isn't just tidy — BI tools are built for it. A clean star lets the tool auto-generate joins, makes drag-and-drop filtering intuitive (filter by any dimension), and keeps queries fast. Hand a BI tool a tangle of normalized tables and self-serve falls apart.

Worked example — speed and security

Worked example · a pre-aggregate for speed, and row-level security
-- PRE-AGGREGATION: dashboards rarely need row-level detail. Build a small
-- summary table so the dashboard scans thousands of rows, not billions.
CREATE TABLE agg_daily_revenue AS
SELECT order_date, sum(line_revenue) AS revenue, count(DISTINCT order_id) AS orders
FROM fct_orders WHERE status = 'paid' GROUP BY order_date;

-- ROW-LEVEL SECURITY: a regional manager should see only their state.
-- A secured view filters by the logged-in user's mapped region.
CREATE VIEW fct_orders_secure AS
SELECT f.* FROM fct_orders f
JOIN dim_customer c ON f.customer_sk = c.customer_sk
JOIN user_region_map u ON c.state = u.state
WHERE u.username = current_user();   -- each user sees only their rows

Report rationalization — the modernization play

The "modernize 400 legacy reports" answerYou never rebuild 400 reports one-for-one. You rationalize: pull usage analytics to find the ~20% actually opened, consolidate overlapping ones onto governed marts + a semantic layer (so "revenue" means one thing), certify the survivors as trusted, and deprecate the dead ones by usage. The deliverable is fewer, trusted, consistent dashboards — not a faithful copy of the old mess.
Check yourself
  1. An exec dashboard is slow and only needs daily totals. Extract or live? Pre-aggregate or not?
  2. Why does self-serve BI work better on a star than on the raw normalized source?
  3. A client has 400 SSRS reports and wants them "all in Power BI by Q3." What do you propose instead, in four moves?
Answers

1) Extract (refreshed a few times daily — freshness isn't critical) off a pre-aggregated daily table so each interaction is instant and cheap.

2) The tool can auto-join fact→dims, filtering by any dimension is intuitive, and queries stay fast; normalized tables force users to know join paths and bog down.

3) Rationalize → consolidate onto governed marts + semantic layer → certify the survivors → deprecate by usage. Rebuild what's used and trusted, not all 400.

07Architecture & System Design

The cross-cutting reasoning that separates an implementer from an architect.

Learn it — architecture is defensible trade-offs, not tool-picking

An implementer is handed a design and builds it. An architect produces the design — and, crucially, can defend every choice against the alternatives. There is almost never one right answer; there's a choice you can justify given the constraints. So "system design" is really a thinking procedure you run out loud. This chapter teaches that procedure and walks one real prompt through it end to end.

The answer-shape — run every design prompt through this

Whether on a whiteboard or a client call, move through these seven beats in order. Skipping straight to "I'd use Snowflake and dbt" is the classic junior mistake — you haven't earned that answer yet.

  1. Requirements — ask before you design: data volume, latency/freshness need, who the consumers are, SLAs, budget, sources, compliance.
  2. Assumptions & constraints — state them out loud so the design is anchored ("assume ~5 TB, daily freshness is fine, team knows SQL not Spark").
  3. Grain & source of truth — declare what one row means and which system is authoritative. (The analytics-engineering signal.)
  4. The design — propose the pipeline end to end.
  5. Trade-offs & alternatives — name what you didn't pick and why.
  6. Failure & recovery — what breaks, how you detect it, how you recover. (The reliability signal.)
  7. Cost — name the spend and the levers, as a first-class constraint.
Worked design · "Migrate a legacy on-prem EDW to a cloud lakehouse"

Here is the same prompt from the reference below, run through the loop the way you'd actually say it.

1 · Requirements (ask, don't assume)

"Before I design — how much data and growth? What's the freshness requirement: real-time, hourly, daily? Who consumes it — BI analysts, data scientists, operational apps? Any regulated data (PII/PHI)? What's the team's skill set? And what's driving the move — cost, a data-center exit, scale?" Say you learn: ~20 TB growing 1 TB/yr, daily freshness is fine, consumers are BI + a forecasting ML model, there's customer PII, the team knows SQL, and a Teradata lease is expiring in 9 months.

2 · Assumptions & constraints

"So: hard deadline (the lease), mixed BI+ML workload (points me at a lakehouse, not a pure warehouse), PII means governance is in scope, SQL-first team means dbt over hand-rolled Spark, daily batch is enough so I don't need streaming."

3 · Grain & source of truth

"The legacy EDW is the source of truth until cutover. I'll preserve the existing fact grains and reconcile against them — I'm not redesigning the model and migrating at the same time; that's two risks at once."

4 · The design

"Land raw extracts in object storage (Bronze), keep the data as an open Iceberg table so both the warehouse and the ML job read one copy. Transform with dbt staging→intermediate→marts (Silver→Gold). Orchestrate with Dagster. Serve BI from the Gold marts; the ML model reads the same Iceberg tables directly. Ingest the historical load in bulk, then a daily incremental sync keeps the target current during the project."

5 · Trade-offs & alternatives

"Why lakehouse over a pure warehouse? The ML workload and one-copy goal. Why Iceberg over copying into the warehouse's native format? No second copy to sync, no lock-in. Why daily batch over streaming? The requirement is daily — streaming would add cost and complexity for no benefit. Why not re-architect the model during the move? It multiplies risk; modernize the model after a clean migration."

6 · Failure & recovery

"Every load step is idempotent (overwrite partition / upsert on key) so retries and backfills are safe. Reconciliation runs each cycle — row counts and checksums on everything, row-level hash diff on the recent slice (Part 2). A failed quality gate halts the pipeline (circuit breaker) before bad data reaches BI. Cutover keeps a parallel run for N cycles and a written rollback to the legacy EDW until the new system is proven."

7 · Cost

"Storage is cheap (object storage). Compute is the lever: separate warehouses for ETL vs BI vs ML so they don't contend, auto-suspend idle compute, and partition the big facts by date so queries scan a slice, not the history. I'd put a budget alert and per-workload cost tags in from day one."

That is a senior answer: it earned its tools by reasoning from constraints, named what it rejected, and treated failure and cost as part of the design — not afterthoughts.

The eternal triangle — name what you're trading

Cost · Performance · FreshnessYou are almost always trading among these three, and you can't max all at once. Want fresher data? You pay in compute (more frequent runs) or performance (live queries instead of cached). Want it cheaper? You accept staleness (less frequent refresh) or slower queries (less pre-aggregation). Want it faster? You pay to pre-compute, or you let it go stale. Good architecture states which corner the client actually cares about and optimizes for that, instead of pretending the trade doesn't exist.

The other foundational trades you'll lean on are all taught in Foundations: OLTP vs OLAP, normalization vs denormalization, CAP/PACELC and strong-vs-eventual consistency, push vs pull, schema-on-read vs -write. When a prompt touches distributed data, that's where the consistency vocabulary earns its keep.

Check yourself
  1. What are the first four things you ask before proposing any design, and why ask before designing?
  2. A prompt says "the dashboard must always show the absolute latest number." Which corner of the triangle did the client just pick, and what are you now willing to spend?
  3. Why migrate the data first and redesign the model later, rather than both at once?
  4. An interviewer asks you to "design X." You immediately name a tool stack. What did you skip, and why does it read as junior?
Answers

1) Volume/growth, freshness/latency, consumers, and constraints (SLAs/budget/compliance). Asking first means the design is justified by requirements instead of guessed — and it's exactly what separates an architect from an implementer.

2) Freshness — so you're now willing to spend on cost (frequent/continuous refresh or live queries) and accept lower raw query performance or higher compute to keep it current.

3) Doing both at once stacks two independent risks (did the data move correctly and is the new model right?); separating them means a clean reconciliation against the old model first, then a modeling improvement you can validate on trusted data.

4) You skipped requirements, assumptions, and grain — you "earned" a tool choice you can't yet defend. Senior answers reason from constraints to the tools, not the reverse.

GGovernance, Security & Compliance

Who can see what, how you prove it, and how you stay legal — the architect-level layer, and your edge with regulated clients.

Learn it — trust has a legal and security half

The earlier chapters make data correct. This one makes it safe and lawful to use. For your target industries — healthcare, insurance, finance — this isn't optional polish; it's the thing that decides whether you can win the work at all. A consultant who can say "here's how we'll handle PII, prove who accessed what, and satisfy HIPAA" is in a different tier from one who only moves tables. Governance answers three questions: who can see what (access), can we prove it (lineage/audit), and are we legal (compliance).

Access control — RBAC vs ABAC

RBAC (role-based) grants permissions to roles, and users get roles: "the analyst role can read the marts." Simple and the default everywhere. ABAC (attribute-based) decides access from attributes at query time: "a user can see rows where the row's region = the user's region, and unmask PII only if their clearance = high." ABAC is more powerful and dynamic but more complex; most shops are RBAC with a few ABAC-style rules for row/column security.

Worked example — RLS, column security, masking, tokenization

These four protect data inside a table that different people are allowed to see differently. Using the shop customers (who have emails — PII):

Worked example · four ways to control what a viewer sees
-- 1) ROW-LEVEL SECURITY: each region manager sees only their state's rows.
CREATE VIEW customers_rls AS
SELECT c.* FROM customers c
JOIN user_region_map u ON c.state = u.state
WHERE u.username = current_user();

-- 2) COLUMN-LEVEL SECURITY: hide a whole column from a role (grant per-column).
GRANT SELECT (customer_id, name, state) ON customers TO ROLE analyst;
--    analyst simply cannot select `email`.

-- 3) MASKING: everyone sees the row, but PII is obscured unless privileged.
CREATE VIEW customers_masked AS
SELECT customer_id, name, state,
  CASE WHEN is_role('pii_reader') THEN email
       ELSE regexp_replace(email, '(^.).*(@.*$)', '\1***\2')  -- a***@x.com
  END AS email
FROM customers;

-- 4) TOKENIZATION: replace the real value with a meaningless token; the
--    real value lives in a separate, tightly-controlled vault. Used when
--    analysts must JOIN on a person but must never SEE the identifier.
--    email 'amy@x.com'  ->  token 'tok_9f3a'   (reversible only via the vault)

Masking vs tokenization — the distinction interviewers want

Masking usually shows a partial/altered value for display and is often irreversible. Tokenization swaps the value for a consistent token so you can still join/group on "the same person" without ever exposing who they are; the mapping back is locked in a vault. Reach for tokenization when analytics needs the identity relationship but not the identity.

You don't need to be a lawyer; you need to know what each regime makes you do to a pipeline. Know which ones your target industries care about.

Catalog, lineage & MDM — the "prove it" layer

A data catalog (DataHub, OpenMetadata, Unity Catalog, Collibra) is the searchable inventory of your tables, owners, and classifications — "where does customer PII live?" should be one query. Lineage (ideally column-level) shows how a field flows source→mart so you can do impact analysis ("if we change this source column, what breaks?") and root-cause ("this dashboard is wrong — trace it back"). MDM (Master Data Management) builds the golden record: when "customer" exists in five systems with conflicting spellings, MDM resolves them into one authoritative entity.

Worked example — a data contract

A data contract moves quality upstream: the team that produces data promises a schema, types, freshness, and semantics, enforced automatically so a breaking change fails at the source instead of silently corrupting everyone downstream.

Worked example · a contract + its enforcement
# contracts/orders.yml — the producer's promise
model: orders
owner: checkout-team
freshness: { max_lag: 1h }
columns:
  - { name: order_id,    type: int,     unique: true,  not_null: true }
  - { name: customer_id, type: int,     not_null: true }
  - { name: status,      type: string,  accepted: [paid, refunded, pending] }
  - { name: amount,      type: decimal, not_null: true }
classification: { customer_id: PII }
-- enforced as a build-blocking test (dbt/GE). If checkout-team ships a change
-- that renames a column or breaks a type, THIS fails and the pipeline stops
-- BEFORE bad data spreads — the producer feels the pain, not 30 downstream teams.

FinOps — cost as a governed constraint

Spend is a first-class design inputFinOps treats cloud cost like any other governed thing: tag workloads so you know who spends what, isolate workloads (separate compute for ETL/BI/ML), monitor with budget alerts, and optimize (partition to cut bytes scanned, auto-suspend idle compute, right-size warehouses). On a cloud warehouse a single un-partitioned query on a big table can cost real money; an architect designs so it can't.
Check yourself
  1. RBAC or ABAC: "every user may see only rows for their own department, decided at query time." Which, and why?
  2. Analysts must count repeat purchases per customer but must never see the customer's identity. Masking or tokenization? Why?
  3. A healthcare client asks what HIPAA changes about your pipeline. Give three concrete things.
  4. Why does a data contract put the failure on the producer, and why is that better than catching it downstream?
  5. Name the four capabilities almost every compliance regime reduces to.
Answers

2) Tokenization — analysts can still group/join on the consistent token to count repeat purchases ("same person"), but the real identity is locked in a vault. Masking would break the join or expose partial identity.

3) Encrypt PHI; restrict and log every access (audit trail); mask/de-identify PHI in non-production environments. (Also: data-handling agreements, retention limits.)

4) Because the producer's breaking change is what causes the corruption; failing at the source stops it before it fans out to every downstream consumer, and puts the fix where the control is. Catching it downstream means N teams debug someone else's change after bad data already spread.

5) Know where sensitive data is; control who touches it; prove who touched it; delete/retain on a schedule.

The engagement lifecycle

Most data engagements walk the same arc. Know it so you can tell a client where you are and what's next:

  1. Discovery — understand the business goal, the current state, the pain, the constraints. Listen more than you talk.
  2. Assessment — profile the data and systems; document findings, risks, and options. This is often a paid first phase on its own.
  3. Design — the target architecture and the plan to get there (run it through the Part 7 answer-shape).
  4. Build — implement in thin vertical slices that each deliver something demoable, not one big-bang at the end.
  5. Validation — prove it works and is correct (reconciliation, tests, UAT with real users).
  6. Handover — documentation, runbooks, training, so they can operate it without you. Then support/iterate.

Discovery — the questions that de-risk everything

Bad engagements almost always trace to a discovery that didn't ask enough. Walk in with a checklist:

A discovery question setBusiness: What decision or outcome should this enable? How will you know it worked? Consumers: Who uses the data, for what, how fresh does it need to be? Sources: What systems, what volumes, who owns them, how clean? Constraints: Budget, deadline (and what's driving it), compliance/PII, the team's skills and what they'll maintain after you leave. Pain: What's broken today that made you call us? Success & risks: What does "done" look like, and what could derail it?

Scoping & estimation — protecting yourself and the client

The most common way data projects fail is scope creep — "while you're in there, can you also…". You manage it with structure:

  • Phase the work. A paid assessment first means you scope the build with real knowledge instead of guessing. Then build in slices with checkpoints.
  • Write scope and non-scope. Explicitly list what's not included; it prevents the awkward later conversation.
  • Estimate from unknowns, not optimism. The big risks in data work are source data quality and access/permissions delays — pad those, and surface them as assumptions.
  • Pick the contract shape. Fixed-price when scope is genuinely clear (you carry the risk); time-and-materials when it's exploratory (they carry it). Misjudging this is how firms lose money.
  • Use a change-request process. New asks are welcome — as a logged change with its own time/cost, not a silent addition.

The cadence that builds trustDemo working software early and often. A client who sees a real slice every two weeks stays confident and corrects course cheaply; a client who waits three months for a big reveal has had three months to imagine it wrong. "No surprises" is the entire reputation game — raise a risk the moment you see it, with an option attached.

Deliverables — what you actually hand over

"Consulting" means leaving artifacts, not just a running system. A credible engagement produces:

  • Assessment / findings report — current state, risks, recommended approach (the output of the assessment phase).
  • Architecture & design doc — the diagram + the decisions and trade-offs behind it (your Part 7 reasoning, written down).
  • The built system — modeled, tested, orchestrated, in their environment.
  • Validation / reconciliation report — the proof it's correct (your Part 2 ladder, as a signed-off document). This is the trust artifact.
  • Runbook & handover docs — how to operate it, what breaks, how to recover; plus a walkthrough/training so the team owns it.
Check yourself
  1. A client says "migrate our warehouse, fixed price, start Monday." Why is that risky, and what do you propose instead?
  2. Three weeks into a build, the sponsor asks you to "also add three new dashboards." How do you respond without damaging the relationship?
  3. Which two unknowns most often blow up data-project estimates, and how do you handle them up front?
  4. You finished a flawless migration but the client feels uneasy and won't sign off. What deliverable most likely closes that gap, and why?
  5. Why demo a thin slice every two weeks instead of one big reveal at the end?
Answers

1) You can't fixed-price what you haven't assessed — source data quality is unknown, so you'd be pricing risk blind. Propose a paid assessment phase first, then a scoped (and more accurate) fixed-price build.

2) Welcome it as a change request: "Great — let me scope it; it's roughly X days / $Y and would shift the timeline by Z, or we can slot it into phase 2. Which do you prefer?" New value, logged and priced, not a silent scope add.

3) Source data quality (it's always messier than described) and access/permissions delays (waiting on IT). Pad both, list them as explicit assumptions, and request access on day one.

4) A validation/reconciliation report — concrete proof (row counts, checksums, row-level diffs) that the data moved correctly. Unease about correctness is cured by evidence, not assurances.

5) Early demos keep the client confident, surface misunderstandings while they're cheap to fix, and let users shape the result — a big-bang reveal risks discovering three months of wrong assumptions at once.

08How Design Interviews Are Scored

The meta-skill. Memorizing terms without this loop won't pass a senior round.

  1. Gather requirements first — volume, latency, consumers, SLAs, budget, sources — before designing.
  2. Declare assumptions & constraints out loud.
  3. Reason about trade-offs — name the alternatives and justify; there's rarely one right answer.
  4. Grain / source-of-truth discipline (the analytics-engineering signal).
  5. Failure handling — what breaks, how you detect, how you recover (the reliability signal).
  6. Cost awareness — cloud spend is a first-class constraint.
  7. Incremental delivery / migration safety — rollback, parallel run, phased cutover.
  8. Governance & quality baked in, not bolted on.
The reusable answer shaperequirements → assumptions → declare grain / source-of-truth → propose design → name trade-offs & alternatives → failure & recovery → cost. Run every prompt through this loop.

High-frequency prompts to rehearse

  • Migrate a legacy on-prem EDW to a cloud lakehouse — walk me through it.
  • Design the analytics layer for a company moving off spreadsheets.
  • Validate that a migration moved data correctly. (your sweet spot)
  • Design a data quality / observability framework for critical pipelines.
  • Implement SCD Type 2 in dbt — explain the mechanics.
  • Build a semantic layer so finance and ops agree on "revenue."
  • Design an incremental dbt model handling late-arriving data.
  • Reduce Snowflake/BigQuery cost without hurting SLAs.
