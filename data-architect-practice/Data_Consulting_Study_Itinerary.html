curl -LsSf https://astral.sh/uv/install.sh | sh

# 2 — create the lab project on Python 3.13 (safer than 3.14 for dbt et al.)
uv init eigenbasis-lab && cd eigenbasis-lab
uv venv --python 3.13
source .venv/bin/activate

# 3 — core stack, all local
uv pip install duckdb pyarrow dbt-core dbt-duckdb \
                 great_expectations dagster dagster-webserver \
                 dagster-dbt jupyter pandas

# 4 — verify

git init && gh repo create eigenbasis-lab --private --source=.
Python 3.14 caveatData tooling often trails the newest CPython by a release. Build this lab's venv on 3.13. Keep 3.14 around for general use; if a package refuses to install, the version is almost always why — recreate with uv venv --python 3.13.
No Power BI on MacPower BI Desktop is Windows-only. Use Metabase (Docker, cross-platform) for the BI modules. Tableau Public also runs on Mac if you want a second tool.

METHODHow to study every module

The same five-step loop, applied everywhere. If you can't draw it, you don't understand it yet.

  1. Concept pass (30–60 min): start with the relevant Foundations group (plain-English + "when to use which"), then read the matching deeper Part of Concepts & Readings + that module's one curated source. Goal: explain it out loud in 2 minutes.
  2. Mental-model sketch (10 min): draw it — a diagram or a 5-bullet what/why/trade-off note.
  3. Minimal rep (1–2 hrs): the smallest hands-on version — one table, one model, one test. Not a big project.
  4. Teach-back / interview rep: answer the module's design prompt out loud (record or write it). This is where it sticks.
  5. Commit the artifact: SQL file, dbt model, markdown writeup, or diagram → into your repo.

Cadence: ~6–8 focused hours/week → ~12 weeks (Weeks 1–10 build the working capability; Weeks 11–12 are the architect-level and regulated-industry layers). Compress or stretch freely; each module is self-contained.

DATASETUse one dataset throughout

Continuity beats variety. Reuse one set across every module so the capstone integrates naturally.

  • TPC-H (generate in DuckDB: INSTALL tpch; LOAD tpch; CALL dbgen(sf=1);) — clean, relational, perfect for dimensional modeling.
  • A messy retail set (Olist Brazilian e-commerce, or Northwind) — the realistic mess that makes data-quality and migration reps real.
  • NYC Taxi Parquet — big enough to feel "analytics scale" for DuckDB/partitioning practice.
RecommendationTPC-H for modeling clarity + Olist/Northwind for the messy quality & migration reps. Both are free and load straight into DuckDB.

WEEK 0.5Foundations primer — do this before Week 1

The fastest way to make every later module click: learn the load-bearing ideas from scratch first.

Work through the Foundations, Explained From Scratch section in Concepts. Every concept is a collapsible card — plain-English definition, an everyday analogy, a tiny example, a "going deeper" layer, and a "when to use which" decision rule. Five groups, each pairing with later weeks:

GroupCoversPairs with
A · Stored & queriedOLTP vs OLAP, row vs columnar, normalization, schema-on-read/writeWeeks 1–2
B · Cloud platformstorage÷compute, MPP, warehouse/lake/lakehouse, table formats, partitioningWeek 8
C · CorrectnessACID, CAP, PACELC, consistency models, idempotency, delivery guaranteesWeeks 6–7
D · Moving dataETL vs ELT, CDC, batch vs streamingWeeks 3, 6, 7
E · Modelinggrain, star schema, SCD2, surrogate keys, semantic layerWeeks 2, 4, 9
The testYou're done when, for each card, you can say the plain definition in one breath and state when you'd choose it over its alternative. Don't memorize — aim to be able to re-derive it. Re-open the relevant group at the start of each week.

WEEK 1SQL & Analytics Foundations

DuckDBSQLthe bedrock
Concepts

OLTP vs OLAP, row vs columnar, window functions, CTEs, aggregation grain, joins & join explosions. → Concepts Foundations A · Part 7.1

Practice
  • Load TPC-H into DuckDB. Write 10 queries escalating in difficulty: simple aggregates → multi-join → window functions (running totals, rank, lag/lead) → a query where a bad join explodes row counts, then fix it.
  • Read a raw Parquet/CSV file directly in DuckDB (SELECT * FROM 'file.parquet') — internalize "query files in place."
Artifact

week1_sql_drills.sql — committed, each query commented with what it demonstrates.

Interview

"Explain grain with a query that violates it." · "Row vs columnar — when each?"

Readings — Week 1
  • Mode — the SQL Tutorial, Intermediate + Advanced (window functions). The cleanest free window-function teaching; runnable in-browser.
  • DuckDB docs — "Querying Parquet/CSV" + the SQL intro. Teaches the "query files in place" pattern you'll lean on all program.
  • Concepts & Readings — Foundations A + Part 7.1 (OLTP vs OLAP). The from-scratch cards anchor why columnar exists and when to use each.

WEEK 2Dimensional Modeling (Kimball)

DuckDBSQLmodeling
Concepts

Star vs snowflake; fact types (transaction/periodic/accumulating); additive/semi/non-additive measures; conformed/role-playing/degenerate dimensions; surrogate keys; declaring grain. → Concepts Foundations E · Part 3.3

Practice
  • Take TPC-H's normalized schema and redesign it as a star schema by hand: declare the grain of a sales fact, identify dimensions (customer, part, supplier, date), build a date dimension, create surrogate keys.
  • Write SQL that populates fct_orders, dim_customer, dim_date.
  • Contrast paradigms (30 min sketch): on paper, model the same sales domain as a Data Vault (hubs/links/satellites) and as One Big Table. Write one sentence on when you'd pick each over a Kimball star. → Concepts Part 3 (modeling paradigms)
Artifact

A star-schema diagram (draw it) + the build SQL + one sentence: "The grain of fct_orders is one row per ___."

Interview

"Design a star schema for e-commerce / healthcare claims. What's your grain?"

Readings — Week 2
  • Kimball Group — the free Dimensional Modeling Techniques index (read SCD + "declare the grain" entries). Authoritative; the vocabulary interviewers use.
  • dbt Labs — blog: "Kimball dimensional modeling in the modern stack." Bridges classic Kimball to how it's done in dbt today.
  • Book (optional) — The Data Warehouse Toolkit, Ch. 1–3. If one modeling book, this.

  • dbt Labs — "How we structure our dbt projects." The exact staging→marts answer interviewers expect.
  • dbt-duckdb — the adapter README. Profile setup specifics for local runs.

WEEK 4Advanced dbt — Incremental & SCD Type 2

dbt-duckdbmost-tested advanced topic
Concepts

Incremental models (is_incremental(), unique_key, strategies: merge/append/delete+insert/insert_overwrite), late-arriving data, snapshots / SCD Type 2, Jinja/macros. → Concepts Foundations E (SCD2) + C (idempotency) · Part 3.2–3.3

Practice
  • Convert a large fact model to incremental; simulate new data arriving; prove only new rows process.
  • Build a dbt snapshot on a changing dimension (e.g., customer address) → watch Type 2 history rows with dbt_valid_from/dbt_valid_to.
  • Write one macro to DRY up repeated logic.
Artifact

Incremental model + working snapshot + a short note explaining the SCD2 mechanics in your own words.

Interview

"Implement SCD Type 2 in dbt — explain the mechanics." · "Handle late-arriving data in an incremental model."

Readings — Week 4
  • dbt docs — Incremental models, Snapshots, Materializations. Concise and exact on the most-tested mechanics.
  • dbt Labs — blog on incremental strategies & late-arriving data. The trade-offs between merge / insert_overwrite.

WEEK 5Data Quality, Testing & Observability

dbt testsGreat Expectationsmessy data
Concepts

7 quality dimensions, 5 observability pillars, dbt tests (generic + singular), GE (suites/checkpoints), assertion vs anomaly detection, data contracts. → Concepts Part 4 · Tools Guide §5

Practice

Use the messy Olist/Northwind set here.

  • Add dbt tests: unique, not_null, relationships, accepted_values; add dbt_expectations; add one singular test (custom SQL).
  • Build a Great Expectations suite on a raw source: profile, generate expectations, run a checkpoint, view Data Docs.
  • Add dbt source freshness checks.
  • Write one data contract: a small YAML declaring a source's schema, types, and freshness SLA + an owner; wire a dbt/GE test that fails the build if the contract is violated. This is the producer-side enforcement idea made concrete.
Artifact

A test suite (dbt + GE) catching ≥3 real issues, with a note mapping each to a pillar/dimension.

Interview

"Design a data quality framework for critical pipelines." · "5 pillars — a monitoring example for each."

Readings — Week 5
  • Monte Carlo — "5 Pillars of Data Observability." From the team that coined the framing.
  • Great Expectations docs — concepts tour + a profiling tutorial. Hands-on object model for your suite.
  • dbt docs — Tests + dbt_expectations README. Fastest path to model coverage.

WEEK 6Data Migration & Reconciliation

Postgres (Docker)DuckDBPythonyour differentiator
Concepts

Migration patterns (big-bang/phased/parallel-run/strangler), the lifecycle, reconciliation (count → checksum → row-diff), failure classes (truncation/timezone/collation/rounding), CDC mechanisms. → Concepts Foundations C–D (idempotency, delivery, CDC) · Part 2 (all) — incl. the new migration "when to use each" decision tables

Practice

The marquee project.

  • Stand up Postgres in Docker as the "legacy source"; load the messy dataset into it.
  • "Migrate" it to DuckDB (or BigQuery, Week 8): initial bulk load, then simulate incremental changes.
  • Build a reconciliation framework (Python + SQL): row counts, column checksums/hashes, row-level diff on PK, schema/type validation. Use PyArrow schemas to compare types. Deliberately inject a truncation and a timezone bug — prove your framework catches them.
  • Add timestamp-based incremental sync; read about log-based CDC (Debezium) conceptually.
  • Managed ingestion rep (optional, ~1 hr): stand up Airbyte (Docker) and replicate one Postgres table into your warehouse with a few clicks. Contrast it with your hand-written extract — when is buy (Fivetran/Airbyte) better than build? → Concepts Foundations D (ETL vs ELT)
Artifact

A reusable reconciliation toolkit (parameterized SQL/Python) + a one-page "migration validation methodology." A portfolio piece and an Eigenbasis sales asset.

Interview

"How do you validate a migration moved data correctly?" · "Walk me through migrating a legacy EDW to the cloud." (whiteboard the lifecycle)

Readings — Week 6
  • Datafold — posts on data diffing & validating migrations. Clearest practitioner reconciliation strategy.
  • Martin Fowler — StranglerFigApplication + ParallelChange. Migration-safety patterns; reframe for data.
  • Debezium docs — CDC overview + snapshot-vs-streaming FAQ. Log-based CDC vocabulary without running it.

WEEK 7Orchestration & Pipeline Design

DagsterAirflow (Docker, optional)
Concepts

DAGs, idempotency, retries, backfills, watermarks, batch vs streaming, Airflow (task-centric) vs Dagster (asset-centric). → Concepts Foundations C–D (idempotency, exactly-once, batch vs streaming) · Part 5 (+ Airflow vs Dagster decision note) · Tools Guide §7–9

Practice
  • Wrap your pipeline as Dagster software-defined assets: ingest → dbt build (via dagster-dbt) → quality checks. Run dagster dev and watch the asset graph materialize.
  • Make a step idempotent and re-runnable; simulate a failure and a backfill.
  • (Optional) Stand up Airflow via Docker compose and rebuild one simple DAG to feel the task-centric contrast.
  • Streaming micro-demo (optional, ~1 hr): run Kafka (Docker), produce a few events to a topic, and consume them — enough to feel topics/partitions/offsets and event-time. Goal is vocabulary, not a production stream. → Concepts Foundations D (batch vs streaming) · Part 5
Artifact

A working orchestrated pipeline (ingest → transform → test) with visible asset lineage.

Interview

"Airflow vs Dagster — contrast the models." · "Make a pipeline idempotent and handle backfills."

Readings — Week 7
  • Dagster docs — Software-Defined Assets tutorial + dagster-dbt guide. Your hands-on path; integrates your dbt project directly.
  • Astronomer / Airflow — Core Concepts + idempotency/backfill guides. Incumbent vocabulary for interviews.
  • Tools Guide §9 — Dagster vs Airflow. The senior framing to rehearse.

WEEK 8Cloud — Platform Modernization on GCP

BigQueryGCSdbt-bigquerythe one cloud module
Concepts

Separation of storage & compute, warehouse vs lakehouse vs lake, table formats, partitioning/clustering, cost control, MPP. → Concepts Foundations B (storage÷compute, MPP, warehouse/lake/lakehouse, table formats, partitioning) · Part 1

This is the one module local can't fully teach — you need a real serverless cloud warehouse.

Practice
  • Create a GCP project (free tier: 10GB storage, 1TB queries/mo). Load your data into GCS as Parquet, then into BigQuery.
  • Repoint your existing dbt project from dbt-duckdb to dbt-bigquery — same models, new adapter. Proves portability (and is a strong talking point).
  • Partition & cluster a table; run the same query partitioned vs not and compare bytes scanned (= cost). Internalize cost-as-a-constraint.
  • Query raw GCS files as an external table → feel the lakehouse pattern.
Artifact

The same dbt project running on BigQuery + a short cost/perf note (partitioned vs unpartitioned bytes scanned).

Interview

"Warehouse vs lakehouse — how do you choose?" · "Reduce warehouse cost without hurting SLAs."

Readings — Week 8
  • Google Cloud — "BigQuery explained" blog series (storage, query/slots, partitioning & clustering). Your target's internals, vendor-authored and clear.
  • dbt-bigquery — adapter setup docs. The profile change that proves portability.
  • GCP docs — BigQuery cost controls / best practices. Bytes-scanned thinking = the cost answer.

WEEK 11Architecture & System-Design Practice

whiteboarddesign repsthe architect signal
Concepts

The cross-cutting reasoning that separates an implementer from an architect: CAP/PACELC/ACID applied to real choices, strong vs eventual consistency, distributed-systems basics (partitioning, replication, skew, shuffle), the reference modern-data-stack diagram, and the scored answer-shape (requirements → assumptions → grain → design → trade-offs → failure → cost). → Concepts Foundations C · Part 7 (Architecture) · Part 8 (How interviews are scored)

Practice

This week is reps, not new tooling — reuse the capstone you've built.

  • Take four prompts and whiteboard each out loud with the answer-shape loop, timed to ~20 min: (1) migrate a legacy on-prem EDW to a cloud lakehouse; (2) design the analytics layer for a company moving off spreadsheets; (3) design a data-quality/observability framework for critical pipelines; (4) reduce warehouse cost without hurting SLAs.
  • Draw the reference modern-data-stack diagram from memory (sources → ingestion → lake/warehouse → dbt → semantic layer → BI; orchestration/observability/governance as cross-cutting). Redraw until fluent.
  • For two prompts, write a one-page design doc: assumptions, the design, the two trade-offs you weighed, the failure modes, and the cost lever.
  • Record yourself answering two prompts; watch it back and tighten.
Artifact

A design playbook: the answer-shape checklist, two written worked designs, and a clean hand-drawn MDS diagram. Your interview talk-track and a credibility asset on sales calls.

Interview

"Migrate a legacy EDW to a cloud lakehouse — walk me through it." · "Warehouse vs lakehouse — how do you choose?" · "Where would this design fail, and how do you recover?"

Readings — Week 11
  • Concepts & Readings — Part 7 (Architecture) + Part 8 (answer-shape) + Foundations C. Your primary text; rehearse the loop until automatic.
  • Book — Designing Data-Intensive Applications (Kleppmann), Ch. 1–6, 9. The reference for the consistency/partitioning/replication trade-offs you'll be pressed on.
  • Book — Fundamentals of Data Engineering (Reis & Housley), Ch. 3. The architecture-survey framing tailored to this consulting domain.

WEEK 12Governance, Security, Compliance & Lineage

DuckDB/BigQueryOpenMetadata (Docker)regulated-industry edge
Concepts

Access control (RBAC vs ABAC), row-level security, column masking & tokenization, the compliance regimes your target industries care about (GDPR, HIPAA, SOX, PCI), data catalogs & column-level lineage, master data management (MDM), data contracts, and FinOps (cost as a governed constraint). → Concepts Governance chapter · Part 7.3

Practice

Use the messy Olist/Northwind set (it has real PII: names, emails, addresses).

  • Classify the PII in your source: tag every column as public / internal / PII / sensitive-PII. This classification is the governance starting point.
  • Implement column masking (a masked view or policy that shows j***@x.com to non-privileged roles) and one row-level security rule (a region sees only its rows). Do it in DuckDB views or BigQuery policy tags.
  • Stand up column-level lineage: either dbt's exposures + docs DAG, or run OpenMetadata/DataHub via Docker and point it at your warehouse. Trace one PII column end to end.
  • Write one data contract (schema + freshness + ownership in YAML) and enforce it with a dbt/GE test that fails the build on violation.
  • Write a short compliance mapping: pick healthcare (HIPAA) or finance (SOX/PCI) and list what each requires of your pipeline (access logs, retention, masking, right-to-erasure).
Artifact

A governance mini-framework: a PII classification table, working masking + RLS, a lineage screenshot, one enforced data contract, and a one-page compliance checklist for a target industry. A direct Eigenbasis sales asset for regulated clients.

Interview

"How do you handle PII during a migration?" · "Design governance for a regulated (healthcare/finance) client." · "What's a data contract and where do you enforce it?"

Readings — Week 12
  • Concepts & Readings — the Governance, Security & Compliance chapter. Your primary text; the worked masking/lineage/contract examples.
  • DataHub / OpenMetadata docs — the "what is lineage / a data catalog" intros + local quickstart. Hands-on catalog & column lineage.
  • Chad Sanderson — the Data Contracts essays. The clearest case for producer-side enforcement.
  • Vendor primers — a plain-English GDPR + HIPAA overview (any cloud's compliance page). Enough to speak to obligations without legal depth.

WHY THIS ORDERSequencing logic

Foundations: W0.5 Concepts primer ─► W1 SQL ─► W2 Modeling ─► W3 dbt ─► W4 dbt advanced Quality/Trust: W5 Data Quality ─► W6 Migration & Reconciliation ◄─ your differentiator Production: W7 Orchestration ─► W8 GCP / Cloud Platform Delivery: W9 Semantic + BI ─► W10 Automation + Capstone Architect: W11 Architecture & System Design ─► W12 Governance & Compliance ◄─ the role-level layer

SQL → modeling → dbt is the non-negotiable spine; everything hangs off a working dbt project. Migration/reconciliation comes after you can build & test models (you need something to reconcile). Cloud comes after local mastery so you're swapping adapters, not learning two things at once. Weeks 11–12 come last on purpose: the architecture and governance reps are most valuable once you have a concrete capstone to reason about — you design and govern a system you've actually built.

SHORT ON TIMEThe minimum-viable path

If you can only do four things before going to market, do these — most interview surface, most billable work.

  1. Week 3 — dbt fundamentals (a working project)
  2. Week 4 — incremental + SCD2 (the advanced dbt signal)
  3. Week 6 — migration & reconciliation (your differentiator + a sales asset)
  4. Week 8 — the same project on BigQuery (proves cloud/platform competence)
