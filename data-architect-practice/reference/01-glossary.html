Master Glossary

BEDROCKFoundations underneath everything

OLTP Source — Online Transaction Processing: a database tuned for many small reads/writes (orders, logins). Row-oriented, normalized, built for writes, not analytics.

OLAP Store — Online Analytical Processing: a system tuned for large scans/aggregations over few columns. Columnar, denormalized. The warehouse side.

Row-oriented — stores all of a row's columns together. Fast to fetch/modify a whole record. Slow to scan one column across millions of rows.

Columnar — stores each column together. Reads only the columns a query needs and compresses them hard — why analytics warehouses are columnar.

Normalization — splitting data into many tables with no duplication (3rd normal form). Great for OLTP write-integrity; needs lots of joins to analyze.

Denormalization — deliberately duplicating data into wide tables to avoid joins at read time. The analytics trade-off.

Schema-on-write — enforce structure when data is written (classic warehouse). vs schema-on-read — store raw, impose structure when queried (data lake).

ACID — Atomicity, Consistency, Isolation, Durability — the transactional guarantees an OLTP database makes so a transfer can't half-happen.

CAP theorem Architecture — under a network partition a distributed system can keep only Consistency or Availability, not both. Names the trade-off you're forced into.

PACELC — CAP's extension: under a Partition choose A or C; Else (normal operation) choose Latency or Consistency. The everyday version of the trade-off.

Strong vs eventual consistency — strong = every reader sees the latest write immediately; eventual = readers may lag briefly but converge. Eventual buys availability/latency.

Idempotency Orchestration — running a step twice = same result as once. Achieved by loading on a key (merge). Without it, a retry double-counts rows.

Delivery guarantees — at-most-once (may lose), at-least-once (may duplicate), exactly-once (neither — usually = at-least-once + idempotency).

The grain Transform — "one row represents exactly ___." The first thing you declare for any table. Get it wrong and every downstream number lies.

Surrogate vs natural key — natural = the business id (email, order #); surrogate = a system-generated id (1,2,3…) used to join dimensions, stable across source changes and needed for SCD2.

STOREPlatform & warehouse modernization

Storage ÷ compute — cloud warehouses keep data in cheap object storage and spin compute up/down separately. You pay for each independently — the core modern-platform idea.

MPP — Massively Parallel Processing: split a query across many nodes, each scanning a slice, then combine. How warehouses scan huge tables fast.

Data lake — raw files (Parquet/JSON/CSV) in your cheap object storage with no table layer — read whatever files are there (schema-on-read). Cheap & flexible; untrustworthy alone (the "swamp").

Lakehouse — the same open files you own, plus an open table-format layer (Iceberg/Delta) tracking which files = a table, its schema & versions → ACID, schema, time-travel on files you still own. The modern default.

Data warehouse — data loaded into the vendor's system, stored in a proprietary internal format only its engine reads (BigQuery, Snowflake). Most polished/governed; lock-in & pricier. Note: both warehouse & lakehouse have strict tables — the split is open files you own vs proprietary vendor-managed storage.

Table format — a metadata layer (Iceberg, Delta, Hudi) over Parquet files that adds transactions, schema evolution, time-travel — what makes a lakehouse possible.

Parquet — the standard open columnar file format. Compressed, columnar, splittable — the physical bytes a lake/lakehouse stores.

Partitioning — physically splitting a table into separate folders by a column's value (often date); the value lives in the folder path, so a filtered query prunes whole folders unopened. Coarse, directory-level. The #1 lever to cut bytes scanned = cost.

Clustering — sorting/co-locating rows within files by a column so per-file min/max ranges stay tight and the engine skips files that can't match. Fine-grained complement to partitioning; cluster high-cardinality cols, partition low-cardinality ones.

Index (vs partition/cluster) — a separate structure of pointers to individual rows for OLTP point lookups. OLAP warehouses don't index to find rows — they partition/cluster to skip scanning data.

Bytes scanned — how much data a query reads — and on BigQuery, what you pay. Partition + cluster + select fewer columns to scan less. The cost language.

External table — a table defined over files left in the lake (not loaded in). Query files in place — the lakehouse access pattern.

Medallion (bronze/silver/gold) — layering convention: bronze = raw, silver = cleaned/conformed, gold = business-ready marts. Maps onto dbt staging→intermediate→marts.

INGEST / MIGRATEMoving data, and proving it moved

ETL — Extract → Transform → Load: transform before loading. Classic, used when storage was expensive or transforms must precede landing.

ELT — Extract → Load → Transform: land raw first, transform inside the warehouse with SQL/dbt. The modern default (cheap storage + powerful warehouses).

CDC — Change Data Capture: move only rows that changed. Query-based (watermark column) or log-based (read the write-ahead log; catches deletes).

Watermark — the high-water value (e.g. max updated_at) you remember from the last run, to fetch only newer rows next time.

Backfill — re-processing historical periods through a pipeline (after a fix or a new column). Only safe if the step is idempotent.

Big-bang migration — cut over all at once. Simple, high-risk, hard to roll back.

Phased migration — move in slices (by module/table/region). Lower risk, longer, needs both systems coexisting.

Parallel run — run old and new side by side, compare outputs until trust is earned, then cut over. Safest, most expensive.

Strangler fig — incrementally route functionality to the new system until the old one is dead. Fowler's pattern, reframed for data.

Reconciliation — proving landed data matches source. The ladder: counts → column aggregates/checksums → row-level diff on PK. Skip it and data loss goes silent.

Checksum / hash — a fingerprint of a column or row used to compare source vs target without shipping all the data.

Failure classes — the usual migration corruptions to name on sight: truncation, timezone shift, collation/encoding, numeric rounding, null handling.

TRANSFORMAnalytics engineering & dbt

dbt model — a SELECT in a .sql file that dbt turns into a view/table. The unit of transformation.

ref() / source() — ref() links a model to another (builds the DAG); source() points at raw input tables. Lineage comes for free.

Materialization — how a model is built: view (logic only), table (full rebuild), incremental (append/merge new rows), ephemeral (inlined CTE).

Staging → intermediate → marts — the standard layering: stg_ (1:1 cleaned/typed) → int_ (joins/logic) → fct_/dim_ (business-ready). The expected project answer.

Star schema — one central fact table surrounded by dimension tables. The Kimball default for BI; few joins, intuitive.

Snowflake schema — a star whose dimensions are further normalized into sub-tables. More joins, less duplication; usually avoided for BI.

Fact table — the measurements/events (one row per order line). Types: transaction, periodic snapshot, accumulating snapshot.

Dimension — the descriptive context you filter/group by (customer, product, date).

Additive / semi / non-additive — a measure you can sum across all dimensions (revenue) / only some (a balance, not over time) / none (a ratio). Governs how you aggregate.

Conformed dimension — one dimension shared identically across multiple facts, so "customer" means the same everywhere.

Role-playing dimension — one dimension used in several roles (date as order_date, ship_date, due_date).

Degenerate dimension — a dimension attribute that lives on the fact with no separate table (e.g. order_id itself).

SCD Type 1 — overwrite the changed attribute. Simple; history is lost.

SCD Type 2 — keep history: close the old row, open a new one with valid_from/valid_to. Point-in-time questions stay correct.

SCD Type 3 — add a column for the prior value (plan + prior_plan). Keeps exactly one previous value, not a full timeline.

Snapshot (dbt) — dbt's built-in mechanism that implements SCD2 for you over a changing source.

Incremental model — a model that processes only new/changed rows each run (via is_incremental() + unique_key) instead of full rebuild.

Macro / Jinja — dbt's templating: reusable SQL snippets (macros) written in Jinja to DRY up repeated logic.

Data Vault — an alternative modeling paradigm (hubs/links/satellites) for auditability/scale. Name it; know it's heavier than Kimball.

One Big Table (OBT) — flatten everything into one wide denormalized table. Simple for a single use case; duplicative across many.

QUALITYReliability & observability

7 quality dimensions — completeness, validity, accuracy, consistency, uniqueness, timeliness, integrity. The vocabulary for "what does good data mean."

5 observability pillars — freshness, volume, schema, lineage, distribution. What you monitor to know a pipeline is healthy (Monte Carlo's framing).

dbt test — generic — reusable column assertions: unique, not_null, relationships, accepted_values.

dbt test — singular — a one-off custom test: any SQL that returns failing rows.

Source freshness — a check that raw source data arrived recently enough (an SLA on lateness).

Data contract — a producer-side promise (schema + types + freshness + owner) that fails the build if violated — stops bad data at the source.

Assertion vs anomaly detection — assertion = a fixed rule you write (must be > 0); anomaly = learned-normal monitoring that flags deviations.

ORCHESTRATIONThe conductor

DAG — Directed Acyclic Graph: the dependency map of steps, no cycles. Defines what runs before what.

Retry — automatically re-running a failed step. Only safe if the step is idempotent.

Task-centric (Airflow) — you orchestrate tasks ("run this script"). The incumbent model; you track what ran.

Asset-centric (Dagster) — you declare the data assets you want to exist; the tool figures out the tasks. You track what data is fresh.

Software-defined asset — Dagster's unit: a declared table/file + the code that produces it, with lineage built in.

Sensor / schedule — triggers: a schedule fires on cron; a sensor fires on an event (a file landed).

GOVERNANCESecurity, compliance & lineage

RBAC — Role-Based Access Control: permissions attached to roles (analyst, admin); users get a role. Coarse, simple, the default.

ABAC — Attribute-Based Access Control: access decided by attributes (department=finance AND region=EU). Finer-grained, more flexible, more complex.

RLS (row-level security) — a policy that filters which rows a role sees (EU team sees only EU rows). Same query, different result per user.

Column masking — show a sensitive column obscured (a***@x.com) to unprivileged roles; real value stays for authorized ones.

Tokenization — replace a sensitive value with a reversible token stored in a separate vault. Stronger than masking; original is recoverable only via the vault.

PII — Personally Identifiable Information (name, email, address). Classifying it is the starting point of governance.

Data catalog — a searchable inventory of tables/columns with owners, descriptions, classifications (DataHub, OpenMetadata).

Column-level lineage — tracing a single column from source through every transform to the dashboard. Required to answer "where did this number come from / where does this PII flow."

MDM — Master Data Management: one authoritative version of core entities (the single "customer") across systems.

GDPR / HIPAA / SOX / PCI — compliance regimes: EU privacy / US health data / financial reporting controls / card data. Each dictates access logs, retention, masking, erasure.

FinOps — treating cloud cost as a governed, monitored constraint — not a surprise bill. Bytes scanned is its main lever here.

SERVE / BIThe layer the client sees

Semantic / metrics layer — define a metric (revenue) once, query it by any dimension. So finance and ops can't disagree on the number (dbt MetricFlow).

Governed dashboard — a certified dashboard built on trusted marts/metrics, with access controls — vs ad-hoc spreadsheets.

Live vs extract — (import vs DirectQuery) — query the warehouse on every view (fresh, slower/costlier) vs cache a copy in the BI tool (fast, stale-er).

Report rationalization — consolidating N sprawling legacy reports into a few certified dashboards. The reporting-modernization play.

ARCHITECTUREThe reasoning layer

Answer-shape — the loop to run every design prompt through: requirements → assumptions → grain → design → trade-offs → failure modes → cost.

The eternal triangle — name what you're trading (consistency / availability / latency; or cost / speed / flexibility). Architecture = defensible trade-offs, not tool-picking.

Partitioning / replication (distributed) — split data across nodes (partition) and copy it for safety/reads (replicate). The two distributed-systems primitives.

Skew / shuffle — skew = uneven data per node (one slow worker); shuffle = moving data between nodes for a join/group (the expensive step).

Modern data stack (MDS) — the reference architecture you draw: sources → ingestion → lake/warehouse → dbt → semantic layer → BI, with orchestration/observability/governance cross-cutting.
