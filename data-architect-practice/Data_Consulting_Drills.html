00How to drill

Recall beats re-reading. Use these the way an athlete uses reps.

  • Say it out loud first. Answer before revealing — retrieval is what builds memory; recognition ("yeah I knew that") is a trap.
  • Climb the tiers. Warm-up = recall a definition. Core = apply it to a scenario. Stretch = a design/interview prompt you'd answer for 2–3 minutes.
  • Space it. Run a theme's warm-up the day you study it, the core a few days later, the stretch the week after. Re-run flashcards weekly.
  • Trace failures back. Miss one? Open the linked Concepts chapter, re-learn, and re-drill next session.
PairingEach set links to its Concepts chapter for the teaching and to the Itinerary week for the hands-on build. Drills check understanding; the Itinerary builds the muscle.

01Foundations & analytics reasoning

You know SQL — these drill the reasoning, not the syntax. → Concepts Foundations A

Warm-uprecall
  1. In one sentence each: OLTP vs OLAP. Why must analytics run on a copy?
    OLTP = many tiny live reads/writes (run the business); OLAP = few big scans (analyze it). You copy out because heavy OLAP scans would starve the live app of resources.
  2. Why is columnar storage cheap for SELECT AVG(x) over 100M rows?
    It reads only the x column (column pruning) + compresses well + skips non-matching blocks (predicate pushdown). A row store reads every full row.
  3. State the rule: normalize for ___, denormalize for ___.
    Normalize for writes (OLTP source systems); denormalize for reads (BI-facing marts).
Coreapply
  1. Write a query whose result is wrong because of a join explosion, then describe the fix.
    Joining orders to a one-to-many child (e.g. order_items) then SUM(order_total) double-counts the header value per child row. Fix: aggregate the child to the order grain first, or sum a line-level measure — i.e. respect the grain.
  2. A report on the production Postgres app DB keeps timing out the checkout flow at peak. Diagnose and fix in one move.
    It's running OLAP on the OLTP store, contending with live traffic. Move the report to a warehouse fed by a copy/CDC of the data; never analyze on the transactional DB.
Stretchdesign / interview
  1. "Walk me through what physically makes a cloud warehouse fast and cheap for analytics." (2 min: columnar + compression/pruning + separation of storage & compute + MPP parallelism.)

02Platform, storage & table types

→ Concepts Foundations B + Part 1

Warm-uprecall
  1. Warehouse vs lake vs lakehouse, one line each.
    Warehouse = structured, strict, SQL/BI. Lake = raw files, cheap, weak guarantees. Lakehouse = lake storage + a table format that re-adds ACID/schema/time-travel.
  2. Three things a table format gives you over raw Parquet.
    ACID transactions, schema evolution, time travel (also: multi-engine access, decoupled storage/engine).
  3. Managed table vs external table vs view — who stores the bytes, and is data materialized?
    Managed: warehouse stores bytes, materialized. External: your bucket stores bytes, materialized (files). View: nothing stored, computed on read.
Coreapply
  1. Decode this: "Our data's in Snowflake but there's a BigQuery copy too." Why does the copy exist, and what removes it?
    A native Snowflake table is readable only by Snowflake, so the data was copied into BigQuery's native format for BQ users — two copies, two formats, a sync job that can drift. One open Iceberg table in object storage, read by both engines, removes the copy and the drift.
  2. Pick the table type: (a) a costly 5-join aggregate hit 10k×/day, 15-min lag OK; (b) partner drops daily Parquet in your bucket, query in place, no ACID; (c) an instant full prod copy to test a migration without doubling storage.
    (a) materialized view (or dynamic table); (b) external table; (c) zero-copy clone.
  3. A 2 TB table query filters WHERE event_date = today but scans it all. First fix and rough saving?
    Partition by event_date → partition pruning reads one day, ~100–1000× fewer bytes scanned (= cost) on a multi-year table.
Stretchdesign / interview
  1. "Why would a client move from native warehouse tables to Iceberg?" (lock-in, one copy across engines, you own the files, no cross-warehouse sync).
  2. "Warehouse vs lakehouse — how do you choose for this client?" Run it through requirements (structured-only + governance → warehouse; +ML/semi-structured/large raw → lakehouse).

03Modeling & dbt

→ Concepts Part 3 · Itinerary W2–W4

Warm-uprecall
  1. What is "grain," and why declare it first?
    What one row represents. Declare it first so joins/aggregations stay correct; skipping it invites fan-out (silently multiplied metrics).
  2. view / table / incremental / ephemeral / snapshot — one use each.
    view = default cheap/fresh; table = expensive query or many readers; incremental = huge append-mostly fact; ephemeral = inlined CTE; snapshot = capture SCD2 history.
  3. SCD Type 1 vs Type 2 in a sentence.
    Type 1 overwrites (current only, no history); Type 2 adds a new dated row and keeps the old (point-in-time history).
Coreapply
  1. "What % of last year's revenue came from customers who were in Texas at time of purchase?" Which technique, and why does SCD1 answer wrong?
    SCD Type 2 on the customer dim preserves state-at-purchase. SCD1 overwrites to current state, mis-attributing every past order to where the customer lives now.
  2. A model is read by 40 dashboards and takes 90s. Which materialization?
    table — compute once, store, let dashboards read the result (incremental only if also large/append-mostly).
  3. Why surrogate keys on dimensions? Name two reasons + the one that's mandatory for SCD2.
    Decouple from messy source IDs; compact/fast joins; survive source re-keying. Mandatory for SCD2 because one natural key maps to many versioned rows.
Stretchdesign / interview
  1. "Structure a dbt project for an e-commerce client." (sources → staging 1:1 cleaned → intermediate joins/logic → marts fct_/dim_, by domain; tests + docs; declare grain on each fact.)
  2. "Implement SCD Type 2 in dbt and explain the mechanics." (snapshot with unique_key + check strategy → dbt_valid_from/to; surrogate per version; join facts on the validity window.)

04Data quality & observability

→ Concepts Part 4 · Itinerary W5

Warm-uprecall
  1. The 5 observability pillars.
    Freshness, Volume, Schema, Distribution, Lineage.
  2. Assertion-based vs anomaly-detection testing.
    Assertion = you state the explicit rule (dbt tests/GE). Anomaly = a learned statistical baseline alerts on deviation (Monte Carlo). Precision vs coverage trade.
Coreapply
  1. Give a concrete monitor for each of the 5 pillars on a daily payments table.
    Freshness: updated within 24h SLA. Volume: row count within trailing-7d band. Schema: no column drop/type change. Distribution: null-rate/mean of amount stable. Lineage: upstream sources fresh/passing.
  2. Which dbt test enforces "every FK in the fact exists in the dim," and which quality dimension is it?
    relationships; the integrity (referential) dimension.
  3. Where do you place a circuit breaker in an ELT flow, and where do bad rows go?
    Between transform stages (run tests after each model, before dependents). dbt build halts the branch; failing rows go to a quarantine/dead-letter table, not the marts.
Stretchdesign / interview
  1. "Design a data-quality framework for a client's critical pipelines." (classify criticality → assertion tests on known rules + anomaly detection on the rest → 5-pillar monitors with SLAs → circuit breakers + dead-letter → data contracts at producer boundary → incident runbook.)

05Migration & reconciliation — your differentiator, drill hardest

→ Concepts Part 2 · Itinerary W6

Warm-uprecall
  1. The reconciliation ladder, four rungs in order.
    Row counts → column aggregates/checksums → row-level hash diff on PK → schema/type validation.
  2. Name five failure classes.
    Truncation, rounding/precision, timezone shift, charset/collation, NULL-vs-empty, duplicate explosion, late/out-of-order (any five).
  3. Rehost vs re-platform vs re-architect.
    Rehost = lift-and-shift, minimal change. Re-platform = some cloud-native wins, no rewrite. Re-architect = redesign the model/system.
Coreapply
  1. Source and target both have 40,000,000 rows. Are you done? Why not?
    No — equal counts only rule out gross loss/dup. Values could be entirely wrong. You still need checksums + a row-level hash diff.
  2. Why COALESCE every column before hashing a row, and why a delimiter?
    A NULL makes the concat NULL → the whole row-hash is NULL → corruption hides. COALESCE to a sentinel keeps each fingerprint meaningful. A delimiter stops '1'+'23' colliding with '12'+'3'.
  3. Daily SUM(amount) matches but per-row cents are off. Which failure class, and why did the sum hide it?
    Rounding/precision — per-row sub-cent errors average out in the dollar sum but show per row. Aggregates mask per-row drift; that's why Rung 3 exists.
  4. 5B-row table, can't fully hash-diff nightly. Your tiered strategy?
    Counts + column checksums on the whole table every cycle; full row-hash diff on the recent partition + a random historical sample; full historical diff once at cutover.
Stretchdesign / interview
  1. "How do you validate a migration moved data correctly?" Walk the full ladder + name two failure classes you'd deliberately test for on an international + money dataset (charset/collation, rounding, timezone, truncation).
  2. "Walk me through migrating a legacy Teradata EDW to a cloud lakehouse." Run the lifecycle (discovery→schema map→bulk load→CDC sync→reconcile→parallel-run cutover→rollback→decommission) and pick a cutover strategy with justification.

06Orchestration & pipelines

→ Concepts Part 5 · Itinerary W7

Warm-uprecall
  1. Airflow vs Dagster in one line each.
    Airflow = task-centric (define steps + order). Dagster = asset-centric (define data products; steps inferred).
  2. Define idempotency and why pipelines need it.
    Running twice = same result as once. Needed because retries/backfills + at-least-once delivery mean tasks re-run; non-idempotent ones duplicate data.
Coreapply
  1. A job does INSERT … WHERE order_date = today and got retried after a blip. What broke, how to fix?
    The retry inserted today's rows twice (not idempotent). Fix: upsert/MERGE on order_id, or delete-today's-partition-then-insert, so re-runs land identically.
  2. You fixed a transform bug and must recompute last quarter. Name it and the required property.
    A backfill; tasks must be idempotent so historical re-runs overwrite cleanly instead of duplicating.
Stretchdesign / interview
  1. "Do you actually need streaming here?" Argue the batch-first default and the narrow cases (per-event decision within seconds: fraud, alerting) where streaming earns its complexity.

07Architecture & system design

→ Concepts Part 7 + Foundations C · Itinerary W11

Warm-uprecall
  1. The seven beats of the answer-shape.
    Requirements → assumptions → grain/source-of-truth → design → trade-offs/alternatives → failure & recovery → cost.
  2. CAP in one line; what it really reduces to.
    During a network partition you keep either consistency or availability, not both. P is mandatory, so it's really C-vs-A during failures.
  3. PACELC — the part CAP misses.
    Else (no partition), you trade Latency vs Consistency — keeping replicas in sync costs latency even when nothing's broken.
  4. The eternal triangle.
    Cost vs Performance vs Freshness — you optimize the corner the client cares about and trade the others.
Coreapply
  1. Client: "money/inventory system, wrong value is unacceptable." CP or AP? And a globally-distributed feed that tolerates staleness?
    CP for money/inventory (consistency over availability); AP for the feed (availability + low latency, eventual consistency).
  2. "Dashboard must always show the absolute latest number." Which triangle corner, and what are you now spending on?
    Freshness — so you spend on cost (frequent/live refresh) and accept lower query performance or higher compute.
Stretchdesign / interview
  1. Full rep: "Design the analytics layer for a company moving off spreadsheets." Run all seven beats out loud in ~3 minutes; record and critique.
  2. "Reduce Snowflake/BigQuery cost without hurting SLAs." (partition/cluster to cut bytes; separate + auto-suspend compute; materialize hot aggregates; right-size; kill un-pruned scans.)
  3. Draw the reference modern-data-stack diagram from memory; explain each layer and the cross-cutting concerns.

08Governance, security & compliance

→ Concepts Governance chapter · Itinerary W12

Warm-uprecall
  1. RBAC vs ABAC.
    RBAC grants to roles users hold; ABAC decides from attributes at query time (per-row/clearance). RBAC simple/default; ABAC dynamic/powerful.
  2. Masking vs tokenization.
    Masking obscures a value for display (often irreversible). Tokenization swaps it for a consistent token (join-able) with the real value vaulted — use when you need the identity relationship but not the identity.
  3. The four capabilities most compliance regimes reduce to.
    Know where sensitive data is; control who touches it; prove who touched it; delete/retain on schedule.
Coreapply
  1. Analysts must count repeat purchases per customer but never see identity. Masking or tokenization?
    Tokenization — they group/join on the consistent token (same person) while identity stays vaulted; masking would break the join or leak partial identity.
  2. Name three concrete things HIPAA changes about your pipeline.
    Encrypt PHI; restrict and log every access (audit trail); mask/de-identify in non-prod. (Also retention limits, handling agreements.)
  3. Why does a data contract put failure on the producer, and why is that better?
    The producer's breaking change causes the corruption; failing at the source stops it before it fans out to N downstream teams and puts the fix where the control is.
Stretchdesign / interview
  1. "Design governance for a regulated healthcare client." (PII/PHI classification + catalog → RBAC + masking/RLS → access logging + column lineage → encryption + non-prod de-identification → retention/erasure → data contracts.)

09Consulting delivery

→ Concepts Consulting Delivery chapter

Warm-uprecall
  1. The engagement lifecycle, six stages.
    Discovery → assessment → design → build (thin slices) → validation → handover.
  2. The five core deliverables.
    Assessment/findings report; architecture & design doc; the built system; validation/reconciliation report; runbook & handover docs.
Coreapply
  1. "Migrate our warehouse, fixed price, start Monday." Why risky, what do you propose?
    You can't fixed-price un-assessed work (source quality unknown = pricing risk blind). Propose a paid assessment first, then a scoped fixed-price build.
  2. Mid-build, sponsor asks for "three more dashboards." Response that protects the relationship?
    Welcome it as a logged change request with its own time/cost and a choice (shift timeline now, or phase 2) — never a silent scope add.
  3. Flawless migration but client won't sign off. Which deliverable closes the gap?
    The validation/reconciliation report — concrete proof (counts, checksums, row diffs). Unease about correctness is cured by evidence.
Stretchdesign / interview
  1. Run a mock discovery: list the questions you'd ask a new client in the first meeting, grouped business / consumers / sources / constraints / pain / success.

10Rapid-fire flashcards

Term on the front, one-sentence definition on the back. Run weekly; if any takes more than a beat, re-read its chapter.

Grain
What one row of a table represents; declared before building.
Fan-out
A one-to-many join inflating rows and silently multiplying metrics.
SCD Type 2
Keep history by adding a dated new row instead of overwriting.
Surrogate key
A meaningless system ID for a dim row; enables SCD2, decouples from source IDs.
ELT
Load raw into the warehouse, then transform in SQL (dbt). The modern default.
Table format
Open metadata over Parquet (Iceberg/Delta/Hudi) giving ACID + time travel + multi-engine.
Managed vs external table
Warehouse stores the bytes vs your bucket stores them; warehouse just reads.
Materialized view
A view whose result is stored and auto-refreshed; speed for repeated aggregates.
Zero-copy clone
Metadata-only copy sharing files until changed; instant cheap prod copies.
Partition pruning
A filter on the partition column skips other partitions → fewer bytes scanned.
ACID
Atomicity, Consistency, Isolation, Durability — the guarantees of a transaction.
CAP
In a partition, choose consistency or availability, not both.
PACELC
CAP, plus: else (normal) trade latency vs consistency.
Idempotency
Running twice = same result as once; makes retries/backfills safe.
CDC
Capture only changed rows (log/timestamp/trigger) to keep a copy fresh.
Reconciliation ladder
Counts → checksums → row-level hash diff on PK → schema validation.
Circuit breaker
Halt the pipeline on a failed quality gate so bad data can't spread.
Data contract
Producer-promised schema/types/freshness, enforced to fail at the source.
5 pillars
Freshness, Volume, Schema, Distribution, Lineage.
Semantic layer
Define each metric once so every BI tool computes it identically.
RBAC vs ABAC
Permissions to roles vs permissions from attributes at query time.
Tokenization
Swap a value for a join-able token; real value locked in a vault.
MDM
Resolve an entity across systems into one golden record.
FinOps
Treat cloud cost as governed: tag, isolate, monitor, optimize.
Answer-shape
Requirements→assumptions→grain→design→trade-offs→failure→cost.
Eternal triangle
Cost vs performance vs freshness — optimize the one that matters.
Scope creep
Silent additions; managed via assessment + change requests.
Strangler pattern
Incrementally route consumers off the legacy system as the new one proves out.
