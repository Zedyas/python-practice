00The big picture

Every tool here plays exactly one role. Learn the role first; the syntax follows.

The modern data stack is a relay. Data lands, gets stored, gets transformed, gets checked, and something has to run all of that on a schedule and tell you when it breaks. Each tool below owns one leg of that relay:

ToolOne-line roleLeg of the relay
uvInstalls Python & packages, manages venvs — fastWorkbench setup
DuckDBAn analytics database that runs inside your processStore + compute (local)
PyArrowThe in-memory columnar format everything speaksThe data interchange layer
dbtTransforms data with SQL, version-controlledTransform
Great ExpectationsAsserts data is what you expectQuality / validation
DockerRuns any tool in an isolated, reproducible boxPackaging / environment
Dagster / AirflowSchedules & orchestrates everything aboveThe conductor
Mental modelDuckDB is the kitchen, dbt is the recipe book, Great Expectations is the health inspector, Docker is a identical pop-up kitchen you can ship anywhere, and Dagster/Airflow is the head chef calling the order of service. PyArrow is the standard-size container every station passes food in.

01uv workbench

A single Rust-based tool that replaces pip, venv, and pyenv — and is dramatically faster.

What it is

uv is a Python package and environment manager. It creates virtual environments, installs packages, and can even install Python versions — all from one binary, resolving dependencies in seconds where pip takes minutes.

The mental model

Mental modelThink of uv as pip + venv + pyenv fused and sped up ~10–100×. Anywhere a guide says pip install X, you say uv pip install X (inside an active venv) or uv add X (in a uv-managed project).

Usage — the commands you'll actually use

# install uv itself (Mac)
curl -LsSf https://astral.sh/uv/install.sh | sh

# you already have 3.13 and 3.14 — point a project at one explicitly
uv venv --python 3.13           # creates .venv using Python 3.13
source .venv/bin/activate         # Mac/Linux activation

# install into the active venv (pip-compatible interface)
uv pip install duckdb dbt-core dbt-duckdb great_expectations dagster

# OR: a managed project with a lockfile (reproducible — better for real work)
uv init eigenbasis-lab && cd eigenbasis-lab
uv add duckdb pyarrow dbt-duckdb        # writes pyproject.toml + uv.lock
uv run python script.py                # runs inside the env, no manual activate
On a Mac with 3.13 & 3.14Some data tools lag the newest Python. If a package won't install on 3.14, create the venv with --python 3.13. dbt in particular is often a release behind the very latest CPython — 3.13 is the safer default for this lab.

02DuckDB store + compute, local

The load-bearing tool of local learning. An OLAP database that runs inside your Python process — no server, no cluster.

What it is

DuckDB is an in-process analytical (OLAP) database — "SQLite for analytics." It's a single dependency, stores a whole database in one file (or in memory), runs columnar/vectorized queries fast on a laptop, and reads Parquet/CSV/JSON directly without an import step.

Why it matters for you

It's your local stand-in for Snowflake/BigQuery. You learn real analytics SQL, run dbt against it, and practice modeling — at zero cost, offline. The skills transfer because dbt + SQL are the same; only the adapter changes when you move to BigQuery.

Mental modelSQLite is to transactional apps what DuckDB is to analytics. "A warehouse you import duckdb into existence." No connection strings, no infra — the database is a file or just RAM.

Usage

# CLI
duckdb mylab.duckdb              # open/create a persistent DB file

# query a file directly — no loading step
SELECT count(*) FROM 'taxi.parquet';
SELECT * FROM read_csv_auto('messy.csv') LIMIT 5;

# generate a realistic relational dataset to practice on
INSTALL tpch; LOAD tpch; CALL dbgen(sf=1);
SELECT * FROM lineitem LIMIT 5;
# From Python — the common pattern
import duckdb
con = duckdb.connect("mylab.duckdb")
con.sql("SELECT region, sum(sales) FROM orders GROUP BY 1").show()
df = con.sql("SELECT * FROM orders").df()      # → pandas
arrow_tbl = con.sql("SELECT * FROM orders").arrow()  # → PyArrow, zero-copy
The "aha"DuckDB queries pandas/Arrow/Parquet in place. You can point SQL straight at a DataFrame variable or a folder of Parquet files — that's the whole "query files where they live" lakehouse intuition, on your laptop.

03PyArrow interchange layer

The invisible plumbing. You rarely "use" it directly, but knowing what it is explains why the modern stack is fast.

What it is

Apache Arrow is a standardized columnar in-memory format; PyArrow is its Python library. Its purpose is to let different tools (DuckDB, pandas, Polars, Spark, BigQuery clients) share data without copying or re-serializing it — "zero-copy." It's also the engine for reading/writing Parquet files.

The mental model

Mental modelArrow is a shared shipping-container standard for data in RAM. Before Arrow, every tool packed data its own way, so handing off meant unpack-and-repack (slow). With Arrow, the container is universal — DuckDB can hand a table to pandas instantly because they already agree on the layout.

Where you'll touch it

  • Reading/writing Parquet — Parquet is on-disk columnar; Arrow is in-memory columnar. PyArrow bridges them.
  • Zero-copy handoffs — duckdb_result.arrow() → feed to another tool with no serialization cost.
  • Schemas & types — when you need precise control over column types (decimal precision, timestamps with timezone) during migration/validation, Arrow's type system is what you reach for.

Usage

import pyarrow as pa
import pyarrow.parquet as pq

table = pq.read_table("orders.parquet")        # read Parquet → Arrow Table
print(table.schema)                            # inspect types precisely
pq.write_table(table, "orders_out.parquet", compression="zstd")

# round-trip with pandas / duckdb is near-instant (zero-copy)
df = table.to_pandas()
For migration work specificallyType fidelity is where migrations break (decimal→float, naive vs tz-aware timestamps). Reading source and target into Arrow and comparing .schema is a clean, language-agnostic way to catch schema drift during reconciliation.

04dbt transform

You've used it on a team. Here's how to reason about it as a system you own end to end.

What it is

dbt turns SQL SELECT statements into managed, tested, documented tables and views inside your warehouse. You write SELECTs; dbt handles the CREATE/INSERT, figures out build order from dependencies, runs tests, and generates lineage docs. It's the "T" in ELT — transformation as version-controlled, modular code.

The mental model

Mental modeldbt is "software engineering practices for SQL transformations." Each model is a function whose output is a table; ref() is how functions call each other; dbt reads those refs to build a dependency DAG and runs models in the right order. You never write CREATE TABLE — you write the query and declare how it should be materialized.

The core objects, plainly

ObjectWhat it does
model (a .sql file)A SELECT dbt materializes as a view/table
ref('other_model')Declares a dependency → builds the DAG & build order
source()Points at raw loaded tables; enables freshness checks
materializationview / table / incremental / ephemeral
snapshotCaptures history → SCD Type 2 automatically
testsunique, not_null, relationships, custom SQL
macros (Jinja)Reusable, parameterized SQL — DRY logic

Usage — the daily loop

uv pip install dbt-core dbt-duckdb     # adapter = which warehouse
dbt init eigenbasis_lab                # scaffold a project

dbt debug      # check the connection (profiles.yml → your .duckdb file)
dbt run        # build all models
dbt test       # run all tests
dbt build      # run + test + snapshot + seed, in DAG order (use this)
dbt docs generate && dbt docs serve    # browse the lineage graph

Project layout you'll standardize on: models/staging/ (stg_, 1:1 with sources) → models/intermediate/ (int_, joins/logic) → models/marts/ (fct_/dim_, business-facing). Same SQL, same project — to go to the cloud you swap dbt-duckdb for dbt-bigquery and change the profile. Nothing else.

The portability pointBecause only the adapter is warehouse-specific, the dbt project you build locally on DuckDB runs on BigQuery with a profile change. That portability is itself a strong interview and sales talking point.

05Great Expectations quality / validation

The health inspector. Turns "I think the data's fine" into explicit, runnable, documented assertions.

What it is

Great Expectations (GE) is a framework for declaring what valid data looks like and checking it automatically. You write Expectations ("this column is never null," "values are in this set," "row count is within range"), group them into a Suite, and run a Checkpoint against a dataset. It produces human-readable Data Docs reporting pass/fail.

The mental model

Mental modelGE is unit tests for data. Where dbt tests live inside your transformation project, GE is a standalone validator you can point at anything — a raw source file, a vendor feed, both sides of a migration. Assertion-based: you state the rule, it checks it.

dbt tests vs Great Expectations — when each

dbt testsGreat Expectations
LivesInside the dbt project, on modelsStandalone; any dataset, any stage
Best forTransform-layer integrity (uniqueness, FK, accepted values)Source validation, profiling, migration source↔target checks
OutputPass/fail in the dbt runRich Data Docs (shareable with clients)
Use both?Yes — dbt tests for the warehouse models, GE at the ingestion/migration boundary.

Usage — the flow

uv pip install great_expectations
great_expectations init        # scaffolds the project
# Python (modern GE, >=0.18 style)
import great_expectations as gx
context = gx.get_context()

batch = context.sources.add_pandas("src").read_csv("orders.csv")
batch.expect_column_values_to_not_be_null("order_id")
batch.expect_column_values_to_be_between("amount", min_value=0)
batch.expect_column_values_to_be_in_set("status", ["open","shipped","closed"])
results = batch.validate()        # → pass/fail, feeds Data Docs
For your migration specialtyRun the same Suite against the legacy source and the migrated target; matching pass profiles is part of your reconciliation evidence. The Data Docs HTML is a client-ready artifact.

06Docker packaging / environment

How you run the "real" versions of tools (Postgres, Airflow, Kafka) without polluting your Mac.

What it is

Docker packages an application plus everything it needs to run (OS libs, dependencies, config) into an image. A running instance of an image is a container — an isolated, disposable box. "Works on my machine" stops being a problem because the box is the same everywhere.

The mental model

Mental modelAn image is a recipe + frozen ingredients; a container is the cooked meal running right now. Containers are cheap, isolated, and disposable — spin one up, use it, throw it away, and your Mac stays clean. docker-compose is how you run several containers together (e.g., Airflow + its Postgres) with one command.

Why you need it in this lab

DuckDB and dbt run natively, but some concepts need real services: a Postgres source DB for migration/CDC practice, the full Airflow, Metabase for BI, or Kafka for streaming vocabulary. Docker runs all of them locally, identically, with no manual installs.

Usage — the commands that cover 90%

# run a throwaway Postgres "legacy source" for migration practice
docker run -d --name pg -p 5432:5432 -e POSTGRES_PASSWORD=pw postgres:16

docker ps                 # what's running
docker logs pg            # see its output
docker exec -it pg psql -U postgres   # shell into it
docker stop pg && docker rm pg        # tear down — Mac stays clean

# multi-service stacks (Airflow, Metabase…) via a compose file
docker compose up -d
docker compose down
Mac noteInstall Docker Desktop (Apple Silicon build). Some images are amd64-only; if one warns about platform, add --platform linux/amd64 — it runs under emulation, slower but fine for learning.

07Dagster orchestration · asset-centric

The conductor. Your recommended orchestrator for independent study — it runs cleanly on a Mac and its model matches how data people actually think.

What it is

Dagster schedules and runs your pipelines, tracks what ran, and surfaces failures in a UI. Its defining idea is the software-defined asset: instead of describing tasks ("run this, then that"), you declare the data assets you want to exist ("a customers table, a daily_revenue table") and how each is computed. Dagster figures out the order from the dependencies between assets.

The mental model

Mental modelAirflow asks "what tasks should run, in what order?" Dagster asks "what data should exist, and what does each piece depend on?" You declare the nouns (assets) and Dagster derives the verbs (the run order). This maps almost 1:1 onto how you already think in dbt — every dbt model is an asset.

Core concepts

  • Asset — a persistent object you want to exist (a table, a file, an ML model). Defined with @asset.
  • Dependencies — an asset names the assets it reads; Dagster builds the graph.
  • Materialize — "run the code that produces this asset."
  • Schedules & sensors — run on a cron, or when an event happens (a file lands).
  • Partitions — slice an asset by date/key for incremental + backfills.
  • I/O managers — pluggable "where does this asset get stored/read" (DuckDB, Parquet, BigQuery).
  • Resources — shared connections (a DuckDB handle, a warehouse client) injected into assets.

Usage

uv pip install dagster dagster-webserver
dagster dev          # launches the local UI at localhost:3000
# a minimal two-asset pipeline
from dagster import asset
import duckdb

@asset
def raw_orders():
    con = duckdb.connect("lab.duckdb")
    con.sql("CREATE OR REPLACE TABLE raw_orders AS SELECT * FROM 'orders.csv'")

@asset(deps=[raw_orders])          # depends on raw_orders → Dagster orders the run
def daily_revenue():
    con = duckdb.connect("lab.duckdb")
    con.sql("CREATE OR REPLACE TABLE daily_revenue AS "
            "SELECT date, sum(amount) rev FROM raw_orders GROUP BY 1")

In the UI you see the asset graph, click Materialize, and watch it run in dependency order. There's a first-party dagster-dbt integration that turns every dbt model into a Dagster asset automatically — so your dbt DAG and your orchestration graph become one picture.

Why Dagster for solo studyIt runs with one command (dagster dev), needs no database or scheduler daemon to start, has a clean local UI, and its asset model reinforces the lineage/grain thinking you're learning anyway. Airflow's full setup is heavier and task-centric — great to know, harder to learn cold.

08Airflow orchestration · task-centric

The incumbent. Most established shops run it, so you must speak it — even if you prototype in Dagster.

What it is

Apache Airflow is the long-standing orchestrator. You author a DAG in Python made of tasks; Airflow schedules them, handles retries, and tracks runs in a web UI. It's task-centric: you describe the steps and their order explicitly.

The mental model

Mental modelAirflow is a programmable cron with dependencies, retries, and a UI. You wire up tasks ("extract → transform → load") and their order with >>. It cares about operations; whether the data is correct afterward is on you (hence pairing it with dbt tests / GE).

Core concepts

DAGTask / OperatorSensor (wait for X)Hook (connection)XCom (pass small values)Executor (Local/Celery/K8s)schedule / catchupbackfillretries

Usage

# run the real thing locally via Docker compose (recommended on Mac)
curl -LfO 'https://airflow.apache.org/docs/apache-airflow/stable/docker-compose.yaml'
docker compose up -d        # UI at localhost:8080
# a minimal DAG
from airflow import DAG
from airflow.operators.bash import BashOperator
import pendulum

with DAG("lab", schedule="@daily",
         start_date=pendulum.datetime(2024,1,1), catchup=False) as dag:
    extract = BashOperator(task_id="extract", bash_command="echo extract")
    transform = BashOperator(task_id="transform", bash_command="dbt build")
    extract >> transform        # defines the order
Learning tipRun Airflow via the official Docker compose rather than a local install — it has several moving parts (scheduler, webserver, metadata DB) that compose wires up for you. Build one simple DAG to feel the task-centric model, then move on; you don't need deep Airflow to interview well, just fluency in its vocabulary and the contrast with Dagster.

09Dagster vs Airflow the comparison you'll be asked

A near-guaranteed interview question and a real architecture decision. Know the axis, not just a winner.

AxisAirflowDagster
Core abstractionTasks (do this, then that)Assets (this data should exist)
Question it answers"What runs, in what order?""What data exists, from what?"
Data awarenessOperations-focused; data-blind by defaultData-aware; lineage & freshness built in
Local devHeavier (scheduler + webserver + DB)dagster dev, one command
TestingBolt-onFirst-class (type checks, asset checks)
dbt integrationVia operatorsNative — dbt models become assets
Maturity / ubiquityIndustry incumbent, huge ecosystemNewer, fast-growing, modern
Best whenExisting Airflow shop; arbitrary task DAGs; ops orchestrationGreenfield analytics; data-asset lineage matters; dbt-centric stack
The senior framing"They solve the same problem from opposite ends. Airflow orchestrates tasks and is the safe, ubiquitous incumbent — if a client already runs it, I work in it. Dagster orchestrates data assets, which maps directly onto dbt lineage and makes data quality and backfills first-class — for a greenfield analytics stack I'd lean Dagster. The choice is existing-investment and team-model, not one being strictly better."

That answer scores because it leads with the abstraction difference, acknowledges context (existing investment), and ties back to data quality / lineage — the things senior interviewers actually probe.

10How they fit together

One picture of the whole local-first lab.

┌─────────────────────── Dagster (or Airflow) ───────────────────────┐ │ conductor: schedules, runs in dependency order, shows failures │ └──────────────────────────────┬──────────────────────────────────────┘ │ materializes assets ↓ Postgres (Docker) ──ingest──► DuckDB (local warehouse) ──swap adapter──► BigQuery (GCP) "legacy source" │ "the cloud" │ transform ↓ dbt (staging → intermediate → marts) │ validate ↓ dbt tests + Great Expectations (source & target) │ PyArrow / Parquet move data between all stages Docker hosts the services that don't run natively

Read it as the relay from the top of this guide, now wired up: Docker supplies the source DB and any heavyweight services, PyArrow/Parquet is the format passing between stages, DuckDB is the local warehouse (BigQuery in the cloud), dbt transforms, dbt-tests + GE validate, and Dagster conducts. Build this once as your capstone and you've used every tool in context — which is exactly what turns "familiar from work" into "can stand it up alone."
