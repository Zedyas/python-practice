Database design · 25% of the interview

Core vocabulary

These are the terms you weave into a design walkthrough to sound fluent. Memorize the one-line definition for each so you can drop it naturally.

Entity memorize

A real-world thing you store data about, represented as a table. Rows are instances; columns are attributes. In a trading system: Trade, Account, Instrument. In healthcare: Patient, Encounter, Provider.

Worded example (trader activity): "A Trade is an entity — each row is one execution. A Trader is another entity — each row is one person placing trades."

Worded example (fund data): "An InvestmentFund is an entity — each row is one fund. A Holding is an entity — each row is one position a fund holds in one security on one date."

Attribute

A property of an entity, stored as a column. A Trade has attributes like trade_id, price, quantity, executed_at.

Worded example (fund data): "The InvestmentFund entity has attributes fund_id, fund_name, strategy (e.g. equity, fixed-income), inception_date, and aum (assets under management). Each is a column describing the fund."

Primary key (PK) memorize

A column (or set of columns) that uniquely identifies each row. Never null, never duplicated. Best practice: a surrogate key — a system-generated meaningless integer/UUID (e.g. trade_id) — rather than a natural key that could change.

Worded example (fund data): "fund_id is the primary key of InvestmentFund — one fund, one ID, forever. I'd use a surrogate fund_id rather than the fund's ticker, because a fund can be renamed or re-tickered but its identity shouldn't change."

Foreign key (FK) memorize

A column that references the primary key of another table, enforcing a relationship. Trade.account_id is an FK pointing to Account.account_id. This is referential integrity: you can't insert a trade for an account that doesn't exist.

Worded example (fund data): "In the Holding table, fund_id is a foreign key back to InvestmentFund, and security_id is a foreign key to Security. Referential integrity means I can't record a holding for a fund that doesn't exist — the database rejects it."

Cardinality

How many rows on one side relate to rows on the other. Three kinds: one-to-one (1:1), one-to-many (1:N), many-to-many (M:N). An Account has many Trades (1:N). A Trade can have many Instruments and an instrument appears in many trades (M:N).

Worded example (fund data): "One InvestmentFund holds many securities, and one security is held by many funds — that's many-to-many. One fund has many daily Holding snapshots over time — that's one-to-many. One fund has exactly one current strategy classification — that's effectively one-to-one if I split it out."

Schema

The overall structure: the tables, their columns and types, and the relationships (keys/constraints) between them. The "data model" they ask you to design is the schema.

Worded example (fund data): "The fund-analytics schema is InvestmentFund, Security, Holding, and PriceHistory, plus the FK relationships linking holdings to funds and securities. That collection of tables and their links is the schema."

Say this in the interview I'd start by identifying the core entities, then their attributes, then the relationships between them — paying special attention to any many-to-many links, which I'll resolve with junction tables. Then I'll normalize, add keys and indexes, and finally think about how this OLTP schema would feed downstream analytics.

That sentence alone is a roadmap that signals competence. Say it first; it buys you time and structures everything after.

Normalization

Normalization memorize

The process of organizing tables to eliminate redundancy and prevent update anomalies, by ensuring each fact is stored in exactly one place. You normalize up to a "normal form."

Worded example (fund data): "If I stored fund_name and fund_strategy on every single holding row, renaming a fund would mean updating thousands of holding rows — an update anomaly. Normalizing means the fund's name lives once in InvestmentFund, and holdings just reference it by fund_id."

FormRule (memorize the gist)Plain meaning
1NFAtomic values, no repeating groupsEach cell holds one value. No comma-separated lists in a column.
2NF1NF + no partial dependency on part of a composite keyEvery non-key column depends on the whole primary key.
3NF2NF + no transitive dependenciesNon-key columns depend on the key, the whole key, and nothing but the key.
The line that wins points "I'd design to 3NF for the transactional side — every non-key column depends on the key, the whole key, and nothing but the key — and then deliberately denormalize for the analytics layer where read performance matters more than write integrity."

Example: an un-normalized table and its fix

Imagine a flat trades table that repeats account and instrument details on every row:

-- UN-NORMALIZED: account_name and ticker repeat on every trade
-- Update anomaly: rename an account -> must update thousands of rows
trades_flat(trade_id, account_name, account_tier, ticker,
            instrument_name, price, quantity, executed_at)

Normalized to 3NF, each fact lives once:

CREATE TABLE account (
  account_id   BIGINT PRIMARY KEY,
  account_name VARCHAR,
  account_tier VARCHAR
);
CREATE TABLE instrument (
  instrument_id   BIGINT PRIMARY KEY,
  ticker          VARCHAR UNIQUE,
  instrument_name VARCHAR
);
CREATE TABLE trade (
  trade_id      BIGINT PRIMARY KEY,
  account_id    BIGINT REFERENCES account(account_id),   -- FK
  instrument_id BIGINT REFERENCES instrument(instrument_id), -- FK
  price         NUMERIC(18,4),
  quantity      INTEGER,
  executed_at   TIMESTAMP
);
Tradeoff to name out loud Normalization reduces redundancy and keeps writes safe, but it means more JOINs at read time. For analytics you often denormalize (pre-join) to make queries faster and simpler — that's the OLTP-vs-OLAP tension, and naming it is exactly the consulting instinct DAS42 wants.

Relationships & the many-to-many fix

Junction table (a.k.a. associative / bridge table) memorize

The standard way to model a many-to-many relationship: a third table holding two foreign keys, one to each side. It turns one M:N into two 1:N relationships.

Worded example (fund data): "Funds and securities are many-to-many — a fund holds many securities, a security sits in many funds. I resolve it with a fund_holding junction table holding fund_id + security_id + the facts that describe that pairing: shares_held, market_value, as_of_date. Those facts belong on the junction because '500 shares' only means something for that specific fund-and-security pair."

Healthcare example — a patient can have many diagnoses and a diagnosis applies to many patients:

CREATE TABLE patient (
  patient_id BIGINT PRIMARY KEY,
  full_name  VARCHAR
);
CREATE TABLE diagnosis (
  diagnosis_id BIGINT PRIMARY KEY,
  icd10_code   VARCHAR,
  description  VARCHAR
);
-- junction table resolves the many-to-many
CREATE TABLE patient_diagnosis (
  patient_id   BIGINT REFERENCES patient(patient_id),
  diagnosis_id BIGINT REFERENCES diagnosis(diagnosis_id),
  diagnosed_at DATE,
  PRIMARY KEY (patient_id, diagnosis_id, diagnosed_at)  -- composite PK
);
Pattern to recognize instantly Any time you hear "a ___ can have many ___ and vice versa" — recipes & ingredients, songs & playlists, orders & products, patients & diagnoses — the answer is a junction table with two FKs. Interviewers plant these on purpose to see if you spot them.

Indexes

Index memorize

A separate data structure (usually a B-tree) that lets the database find rows by a column's value without scanning the whole table — like the index at the back of a book. Speeds up reads (WHERE, JOIN, ORDER BY) dramatically.

Worded example (fund data): "If analysts constantly query holdings WHERE fund_id = ? AND as_of_date = ?, I'd index those columns so the database jumps straight to the relevant rows instead of scanning every holding ever recorded."

The index tradeoff

Indexes make reads faster but writes slower (every INSERT/UPDATE must also update the index) and consume storage. So you index columns you filter or join on frequently, not every column.

What to index, said simply

  • Primary keys (indexed automatically).
  • Foreign keys you JOIN on — e.g. trade.account_id.
  • Columns in frequent WHERE/ORDER BY — e.g. trade.executed_at for date-range queries.
CREATE INDEX idx_trade_account  ON trade(account_id);
CREATE INDEX idx_trade_exec_time ON trade(executed_at);
-- composite index: order matters, leftmost column usable alone
CREATE INDEX idx_trade_acct_time ON trade(account_id, executed_at);
Interview soundbite "I'd index the foreign keys and the timestamp we filter on, but I'd be careful not to over-index a write-heavy trade table, since every index adds write overhead."

The design walkthrough method (your script)

Whatever they ask you to design, follow these six steps out loud, in order. This is the single most important thing to internalize — it turns a scary open-ended question into a checklist.

  1. Clarify scope. Ask 2–3 questions. "Are we modeling the transactional storefront or the analytics behind it?" "What are the main things users do?" This shows consulting instinct and buys thinking time.
  2. List entities. Name the core nouns. Say them aloud as you write boxes.
  3. Add attributes & keys. Give each entity a surrogate PK and its key columns.
  4. Draw relationships. State cardinality for each ("an account has many trades"). Resolve every M:N with a junction table.
  5. Normalize & index. Note you're at 3NF; call out which columns you'd index and why.
  6. Bridge to analytics. Describe how this OLTP schema would feed a star schema for reporting. This is the differentiator at a data consultancy.
Opening move, verbatim "Before I start drawing — am I designing the operational database that runs the app, or the warehouse model behind the analytics? I'll assume operational first and then show how it feeds analytics. A few clarifying questions: what are the main actions a user takes, and what's the most important thing the business needs to report on?"

Worked example: design AllRecipes

Rehearse this aloud. The structure is what matters, not memorizing exact columns.

1. Scope. "It's a recipe-sharing site: users post recipes, recipes have ingredients and steps, other users rate and review them, and save favorites. I'll model the transactional side."

2–4. Entities, keys, relationships:

user (user_id PK, username, email, created_at) recipe (recipe_id PK, user_id FK, title, prep_minutes, cook_minutes, created_at) ingredient (ingredient_id PK, name, default_unit) recipe_ingredient (recipe_id FK, ingredient_id FK, quantity, unit) <- junction (M:N) step (step_id PK, recipe_id FK, step_number, instruction) review (review_id PK, recipe_id FK, user_id FK, rating, body, created_at) favorite (user_id FK, recipe_id FK, saved_at) <- junction (M:N)

Relationships, said aloud:

  • A user authors many recipes; a recipe has one author → 1:N (recipe.user_id).
  • A recipe has many ingredients and an ingredient appears in many recipes → M:N, resolved by recipe_ingredient, which also carries quantity/unit (those belong on the relationship, not on the ingredient).
  • A recipe has many ordered steps → 1:N, with a step_number to preserve order.
  • Users review recipes, users favorite recipes → two more relationships, with favorite as a junction.

5. Normalize & index: "This is 3NF — ingredient names live once in ingredient, not repeated per recipe. I'd index recipe.user_id, review.recipe_id, and recipe_ingredient's FKs since those drive the common reads."

6. Bridge to analytics: "For reporting — say, most-saved recipes by month — I'd build a star schema with a fact_review or fact_favorite table and dimensions for recipe, user, and date."

The detail that impresses Putting quantity and unit on the junction table (not on ingredient) shows you understand that those facts describe the relationship between a recipe and an ingredient — "2 cups of flour" only means something for that pairing.

Worked example: design a data model for an online store (the confirmed question)

This is the version a real DAS42 candidate reported. Make it your primary rehearsal.

1. Scope. "A retailer selling physical products to consumers. Core flows: browse products, place an order with multiple items, pay, ship. I'll model the operational database, then the analytics."

customer (customer_id PK, email, name, created_at) address (address_id PK, customer_id FK, line1, city, region, postal_code, type) product (product_id PK, sku, name, description, unit_price, category_id FK) category (category_id PK, name, parent_category_id FK) <- self-reference for hierarchy inventory (product_id FK PK, quantity_on_hand, warehouse_id) "order" (order_id PK, customer_id FK, order_status, ordered_at, ship_address_id FK) order_item (order_id FK, product_id FK, quantity, unit_price_at_purchase) <- junction (M:N) payment (payment_id PK, order_id FK, amount, method, status, paid_at)

Key relationships and the things interviewers probe:

  • An order contains many products; a product appears in many orders → M:N via order_item.
  • Critical detail: store unit_price_at_purchase on order_item, not just a reference to product.unit_price. Prices change; the order must remember what the customer actually paid. This is a classic "gotcha" they look for.
  • order_status ("pending → paid → shipped → delivered") models the lifecycle; for full history you'd add an order_status_history table.
  • Category self-references via parent_category_id to allow nested categories.

5. Normalize & index: "3NF. Index order.customer_id, order_item FKs, and product.sku. The order table is write-heavy, so I'd keep its indexes lean."

6. Bridge to analytics: "For revenue reporting I'd build fact_order_item (grain = one line item) with dim_customer, dim_product, dim_date. That answers 'revenue by category by month' with a simple star-schema query."

If they push: "what about scale?" "I'd partition the order/order_item tables by date, archive old orders to cold storage, and for the analytics side push everything into a columnar warehouse like Snowflake where the star schema lives."

Dimensional modeling · the consulting differentiator

OLTP vs OLAP

Knowing both worlds — and the bridge between them — is what separates an analyst from a data engineer. DAS42 lives in this space.

OLTP (transactional)OLAP (analytical)
PurposeRun the applicationAnswer business questions
WorkloadMany small reads/writesFew large aggregating reads
DesignNormalized (3NF)Denormalized (star schema)
Example queryInsert one tradeTotal volume by desk by quarter
TechPostgres, SQL ServerSnowflake, BigQuery, Redshift
Data warehouse memorize

A central analytical repository optimized for reading and aggregating large volumes of historical data from many sources. It's where OLAP lives. Snowflake, BigQuery, and Redshift are cloud data warehouses.

Star schema, facts & dimensions

Fact table memorize

The center of a star schema. Holds measurable, numeric events (the "facts") plus foreign keys to dimensions. Each row is one event at a defined grain. Example: fact_trade with price, quantity, commission and FKs to date, account, instrument.

Worded example (fund data): "fact_holding has one row per fund per security per day, with measures market_value and shares_held, plus FKs fund_key, security_key, date_key. The numbers I sum live here; the descriptions live in the dimensions."

Dimension table memorize

Descriptive context you slice and filter by — the "who, what, where, when." dim_account, dim_instrument, dim_date. Wide and denormalized on purpose.

Worded example (fund data): "dim_fund holds fund_name, strategy, manager, inception_date — the descriptive context. So 'total market value by strategy by quarter' filters on dim_fund.strategy and dim_date.quarter while summing fact_holding.market_value."

Grain memorize

What one row of the fact table represents. "One row per trade" or "one row per order line item." Defining the grain first is the cardinal rule of dimensional modeling — say this and you sound like you've done it.

Worded example (fund data): "I'd declare the grain up front: 'one row per fund, per security, per day.' That single sentence pins down everything — it tells you the FKs (fund, security, date) and prevents double-counting. If someone later asks for intraday data, I know the grain has to change."

dim_date | dim_account --- fact_trade --- dim_instrument | dim_desk fact_trade grain = one executed trade measures: price, quantity, gross_value, commission FKs: date_key, account_key, instrument_key, desk_key
-- A star-schema query reads naturally: measures from fact, filters from dims
SELECT d.year, d.quarter, i.asset_class,
       SUM(f.gross_value) AS total_value
FROM fact_trade f
JOIN dim_date d       ON f.date_key = d.date_key
JOIN dim_instrument i ON f.instrument_key = i.instrument_key
WHERE d.year = 2025
GROUP BY d.year, d.quarter, i.asset_class
ORDER BY total_value DESC;
Star vs snowflake schema A star schema keeps dimensions denormalized (one table each). A snowflake schema normalizes dimensions into sub-tables. Star is usually preferred for simplicity and query speed. (Yes — name-drop that the company you're interviewing at is named after this concept if it feels natural; it shows you get it.)

Slowly changing dimensions (SCD)

Slowly changing dimension memorize

How you handle dimension attributes that change over time (e.g. a customer moves, an account changes tier). The two you must know:

Type 1 — overwrite the old value. Simple, no history.
Type 2 — add a new row with effective/expiry dates and a "current" flag, preserving full history. This is the one interviewers care about.

-- SCD Type 2: account tier history preserved
dim_account(account_key, account_id, tier,
            valid_from, valid_to, is_current)
-- when tier changes: expire old row (set valid_to, is_current=false),
-- insert new row with is_current=true

Migration & pipelines · your resume's core

ETL vs ELT

ETL memorize

Extract, Transform, Load. Pull data from sources, transform it (clean, join, reshape) in a staging engine, then load the finished result into the warehouse. Classic on-prem pattern.

Worded example (fund data): "Old way: pull raw fund-holdings files from a vendor, run a separate process to clean and aggregate them into fund-level summaries, then load only the finished summary into the warehouse. The transform happens before the data lands."

ELT memorize

Extract, Load, Transform. Load raw data into the warehouse first, then transform it in-place using the warehouse's compute. The modern cloud pattern — it's what dbt + Snowflake do, and what you did in your migration.

Worded example (fund data): "Modern way: dump the raw vendor holdings file straight into the warehouse as-is, then use dbt to build cleaned and aggregated models on top. The raw data stays available, so if I find a bug in my aggregation logic I just rerun the transform — no re-extracting."

Why the shift matters (say this) "We moved toward ELT because cloud warehouses like Snowflake have cheap, elastic compute — so it's faster and simpler to load raw data and transform it in-warehouse with dbt than to run a separate transformation tier. It also keeps the raw data available for reprocessing."

Your migration walkthrough (the script)

You will be asked to walk through a project. Use this skeleton with your SQL-Server-to-Snowflake work. Aim for 10–15 minutes; practice hitting each beat.

  1. Context / problem. "We had analytical datasets on on-prem SQL Server. It didn't scale for the analytics workload, and we wanted a modern cloud warehouse, so we migrated to Snowflake."
  2. Your role. Be specific and honest about what you built — the ETL/ELT pipelines, the dbt transformations, the validation. Interviewers ask "what did you specifically do?"
  3. Approach. "We extracted from SQL Server, landed raw data in Snowflake, then rebuilt the business logic as modular dbt models — an ELT pattern."
  4. Hardest part. Have one concrete challenge ready (data-type mismatches, reconciling row counts, a transformation that didn't match the legacy report). This is the most common follow-up.
  5. How you validated. Reconciliation, schema checks, business-rule tests (next section). This is gold at a consultancy.
  6. Outcome. Quantify if you can — faster queries, fewer incidents, reliable reporting.
Prepare the "what broke" answer Pick ONE real challenge and rehearse it as a mini-STAR: what went wrong, how you diagnosed it, what you did, what you learned. "A dbt model's totals didn't match the legacy SQL Server report; I traced it to a NULL-handling difference in a SUM, added a not-null test and a reconciliation check, and it caught two more cases before production." That single story answers half their follow-ups.

Validation strategies (say these by name)

Source-to-target reconciliation

Compare row counts and key aggregates (sums, distinct counts) between source and target to confirm nothing was lost or duplicated in transit.

Schema validation

Confirm column names, data types, and nullability match the contract after migration. Catches silent type coercions.

Business-rule testing

Assert domain rules hold: no negative quantities, every trade has a valid account, dates are in range.

Reconciliation by aggregate

Beyond counts: compare SUM(amount), MIN/MAX dates, distinct counts per key to catch subtle drift.

-- Reconciliation check: row counts + a key aggregate must match
SELECT 'source' AS side, COUNT(*) AS rows, SUM(amount) AS total
FROM source.trades
UNION ALL
SELECT 'target', COUNT(*), SUM(amount)
FROM snowflake.trade;
-- the two rows should be identical; any diff is a red flag

dbt concepts (be ready to defend this)

dbt (data build tool) memorize

A transformation framework for the "T" in ELT. You write modular SELECT statements as "models"; dbt handles dependencies, materialization, testing, and documentation. It brings software engineering practices (version control, testing, modularity) to SQL transformations.

TermWhat it is
ModelA single .sql file = one SELECT that becomes a table or view.
MaterializationHow a model is built: view, table, incremental (only new rows), or ephemeral.
ref(){{ ref('stg_trades') }} — links models so dbt builds them in dependency order (a DAG).
TestBuilt-in assertions: not_null, unique, relationships (FK integrity), accepted_values.
SourceDeclared raw input tables; lets dbt track lineage from raw to final.
Staging / mart layersConvention: stg_ models clean raw sources; mart/fct/dim models hold business logic.
-- models/staging/stg_trades.sql  (a dbt model)
SELECT
  trade_id,
  account_id,
  CAST(price AS NUMERIC(18,4)) AS price,
  quantity,
  executed_at
FROM {{ source('raw', 'trades') }}
WHERE executed_at IS NOT NULL
# schema.yml  — tests that run on every build
models:
  - name: stg_trades
    columns:
      - name: trade_id
        tests: [unique, not_null]
      - name: account_id
        tests:
          - relationships:
              to: ref('stg_accounts')
              field: account_id
If asked "what challenges did you face with dbt?" Have a real one ready: managing model dependencies as the DAG grew, getting incremental models to handle late-arriving data correctly, or standardizing tests across the team. Concrete beats generic.

Snowflake · named explicitly in candidate reports

Architecture basics

Separation of storage and compute memorize

Snowflake's defining feature. Storage (your data) and compute (the engines that query it) scale independently. You can spin up multiple compute clusters against the same data without copying it.

Virtual warehouse

A compute cluster (named T-shirt sizes: XS, S, M…). You size it for the workload and can run several in parallel. Pay only while it runs; auto-suspend when idle.

Micro-partitions

Snowflake automatically stores table data in compressed columnar chunks (~50–500MB) with per-column metadata (min/max). This metadata lets it skip irrelevant partitions at query time.

Pruning

Skipping micro-partitions that can't contain matching rows, using the metadata. This is why filtering on a well-clustered column is fast — Snowflake reads only the relevant partitions.

LayerWhat it does
StorageColumnar, compressed micro-partitions in cloud object storage.
ComputeVirtual warehouses that execute queries, scaled independently.
Cloud servicesMetadata, query optimization, security, transaction management.

Snowflake performance levers

  • Clustering keys — for very large tables, define a clustering key so related data sits in the same micro-partitions, improving pruning.
  • Warehouse sizing — bigger warehouse = more compute for heavy queries; scale up for performance, scale out (multi-cluster) for concurrency.
  • Result caching — identical queries return cached results instantly within the retention window.
  • Avoid SELECT * — columnar storage means selecting only needed columns reads far less data.
Soundbite "Because Snowflake is columnar with micro-partition pruning, the biggest wins come from filtering on clustered columns and selecting only the columns you need — that minimizes the data scanned, which is what you pay for in both time and credits."

SQL · 20% + the OA gate

Window functions (the most-tested topic)

Window function memorize

A function that computes across a set of rows related to the current row (the "window") without collapsing them like GROUP BY does. You keep every row but add a computed column. Syntax: func() OVER (PARTITION BY ... ORDER BY ...).

Worded example (fund data): "To compute each fund's day-over-day change in market value, I partition by fund_id, order by as_of_date, and use LAG() to grab the prior day's value — every daily row stays, but now each one knows yesterday's number too."

FunctionReturns
ROW_NUMBER()Sequential number per partition (1,2,3…), no ties.
RANK()Rank with gaps after ties (1,2,2,4).
DENSE_RANK()Rank without gaps (1,2,2,3).
LAG()/LEAD()Value from the previous/next row — great for deltas.
SUM() OVERRunning total / windowed aggregate.
-- Daily price change per instrument using LAG
SELECT
  instrument_id,
  trade_date,
  close_price,
  close_price - LAG(close_price) OVER (
      PARTITION BY instrument_id ORDER BY trade_date
  ) AS daily_change
FROM daily_prices;
-- Top trade per account by value (classic "top-N-per-group")
WITH ranked AS (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY account_id ORDER BY gross_value DESC
    ) AS rn
  FROM trade
)
SELECT account_id, trade_id, gross_value
FROM ranked
WHERE rn = 1;
Know the difference cold "GROUP BY collapses rows into one per group; a window function keeps every row and adds a computed column over a partition. Use window functions for running totals, rankings, and row-to-row comparisons like period-over-period change."

Joins & deduplication

JoinReturns
INNEROnly matching rows in both tables.
LEFTAll left rows + matches; NULLs where no match.
FULL OUTERAll rows from both, matched where possible.
SELFA table joined to itself (e.g. category → parent_category).
-- Deduplicate: keep the latest record per patient
WITH ranked AS (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY patient_id ORDER BY updated_at DESC
    ) AS rn
  FROM patient_records
)
SELECT * FROM ranked WHERE rn = 1;
-- Find accounts with no trades (anti-join via LEFT + IS NULL)
SELECT a.account_id
FROM account a
LEFT JOIN trade t ON a.account_id = t.account_id
WHERE t.account_id IS NULL;

Query optimization (have a story)

  • Filter early — push WHERE conditions down so you process fewer rows.
  • Index the join/filter columns — covered above.
  • Avoid SELECT * — especially in columnar warehouses.
  • Read the query plan — EXPLAIN shows full table scans vs index use; look for the expensive step.
  • Avoid functions on indexed columns in WHERE — WHERE YEAR(dt)=2025 can't use an index; use dt >= '2025-01-01' AND dt < '2026-01-01'.
Optimization story template "A reporting query was doing a full scan on a large trade table. I checked the plan with EXPLAIN, saw it wasn't using the date filter efficiently, rewrote a function-wrapped WHERE into a sargable range predicate, and added a composite index on (account_id, executed_at). Runtime dropped from minutes to seconds."

SQL — the recurring OA shapes

  • Join + aggregate: GROUP BY with SUM/COUNT/AVG, often after a join.
  • Top-N-per-group: ROW_NUMBER() OVER (PARTITION BY g ORDER BY x DESC) in a CTE, then WHERE rn <= N.
  • Running total / period-over-period: SUM() OVER (... ORDER BY ...) or LAG().
  • Dedup: ROW_NUMBER() ... = 1 to keep one row per key.
  • Self-join / anti-join: LEFT JOIN + IS NULL to find non-matches.

One-page cheat sheet

The 6-step design script

1) Clarify scope · 2) List entities · 3) Attributes + keys · 4) Relationships (resolve M:N with junction tables) · 5) Normalize to 3NF + index FKs/filters · 6) Bridge to a star schema for analytics.

One-liners to have ready

3NF: every non-key column depends on the key, the whole key, and nothing but the key.
M:N → junction table with two FKs.
Index tradeoff: faster reads, slower writes — index FKs and filtered columns.
OLTP vs OLAP: normalized transactional vs denormalized analytical.
Star schema: fact (measures + FKs) surrounded by dimensions (context); define the grain first.
ELT: load raw to the warehouse, transform in-place with dbt.
Snowflake: storage/compute separated; micro-partitions + pruning.
Window fn: compute over a partition without collapsing rows.
