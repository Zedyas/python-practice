Design Round — Framework & Practice ❄

The interview's make-or-break round is a "design X" question (your referral got Reddit; the sample answer was AllRecipes). The prompt changes but the rubric never does. This file gives you the repeatable thinking process, then prompts to drill it — one fully worked (answer hidden so you try first), four to practice. The shape is always the same as Reddit/AllRecipes: entities, relationships, the tricky three. Nothing more exotic.

Start here

The mindset

A "design X" question is not a memory test. There's no single right schema. The interviewer is watching how you think — can you take a messy real-world thing, break it into entities and relationships out loud, and handle curveballs calmly. So the win condition is a calm, repeatable process, not a memorized diagram.

The reframe that removes the fear You are not recalling "the Spotify schema." You are running the same six steps on whatever noun they hand you, and asking one question over and over: "how many of A relate to how many of B?" That question routes every relationship. Master the process and any prompt is approachable.

The 6-step script

Memorize this, not tables. Run these in order, out loud, for any prompt.

  1. Clarify scope (30 sec, 2–3 questions). "Are we modeling the operational app or the analytics behind it? What are the main things a user does?" Buys thinking time, shows consulting instinct, stops you building the wrong thing. State your assumption, then move.
  2. List the core entities (the nouns). Say them aloud as you draw boxes. Don't worry about relationships yet — just get the nouns down.
  3. Add keys + 2–3 attributes each. Every entity gets a surrogate primary key (user_id). Name a couple real attributes so it feels concrete; don't list them all.
  4. Draw relationships — say the cardinality aloud for each pair. This is the heart. For every pair ask "how many of A to how many of B?" and name it: "an artist has many albums, an album has one artist — one-to-many."
  5. Resolve the tricky relationships. Many-to-many → junction table. Self-referencing → nullable parent FK. Attributes that describe the pairing → put them on the junction. (This is where points are won.)
  6. Gesture at scale / analytics. One or two sentences: index the FKs, cache hot counts, denormalize for reads, feed a star schema for analytics. Separates you from someone who only knows OLTP — and it's the consulting differentiator at DAS42.
Opening line, verbatim "Before I start drawing — am I designing the operational database that runs the app, or the analytics warehouse behind it? I'll assume operational first. Quick questions: what are the main actions a user takes, and what does the business most need to report on?" Then go.

The relationship decision tree

Steps 4–5 are where every design lives or dies. There are only four patterns. Recognizing which one applies is the entire skill.

1 · One-to-Many (1:N)
Cue: "A has many B, but each B belongs to one A."
Fix: foreign key on the "many" side.   e.g. albums.artist_id, posts.user_id
2 · Many-to-Many (M:N)
Cue: "A has many B AND B has many A."
Fix: junction table with two FKs + a unique constraint to stop duplicates.   e.g. playlist_songs, favorites
3 · Self-Referencing (threaded / hierarchy)
Cue: "B can relate to other B" — replies, org charts, nested categories.
Fix: nullable FK on the table pointing back at itself. comments.parent_comment_id → comments.comment_id (null = top level). This is the "adjacency list."
4 · Attributes ON the relationship
Cue: a fact describes the pairing, not either entity alone.
Fix: put those columns on the junction table.   e.g. recipe_ingredients(recipe_id, ingredient_id, quantity, unit), a song's sort_order within a playlist.
The one question that unlocks everything For every pair of entities: "How many of A relate to how many of B?" The answer points you straight at pattern 1, 2, 3, or 4. Drill that reflex and the relationships fall out automatically.

The "what happens when…" follow-ups

These aren't random — each maps to a specific pattern. So when you hear the cue, you already know the answer they're fishing for. Pre-empt them in your first pass and they have less to ask.

The follow-upWhat they're fishing for
"…lots of child items (steps, ingredients, photos)?"Separate 1:N table + an explicit sort_order column (DB insertion order is not guaranteed).
"…a user favorites / likes / follows / saves?"M:N junction table + unique constraint so the same action can't be stored twice.
"…replies to replies (nesting / threading)?"Self-referencing parent_id (adjacency list). Name a recursive CTE for reads; mention closure table / materialized path if it gets very deep.
"…this gets huge / queries get slow?"Index the FKs, cache hot counts (don't COUNT a billion rows each load), denormalize for reads, partition by date, feed a star schema for analytics.
"…the same thing can be liked/tagged in multiple places (posts AND comments)?"Polymorphic association: a type + id pair. Name the tradeoff vs. separate tables (loses strict FK integrity, gains flexibility).
"…you need history / things change over time (price, status)?"Don't overwrite — add a history/status table, or SCD Type 2 with valid_from / valid_to / is_current.

Worked example — try before revealing

Design Spotify

Do this one for real before opening anything. Read the hints, run the 6-step script out loud or on paper, sketch the tables, then reveal to compare. The goal isn't matching exactly — it's hitting the same structure.

“Design Spotify”
Users stream songs, organize them into playlists, follow artists, like songs. Model the core operational app (not the recommendation engine).
Hint 1 — the entities (peek if stuck on step 2)

The core nouns: users, artists, albums, songs, playlists. That's enough to build the whole thing. Don't over-add — you can mention genres or labels as "I'd extend with…" later.

Hint 2 — the relationships to work out (peek if stuck on step 4)

Walk these pairs and name the cardinality for each: artist↔album, album↔song, playlist↔song, user↔song (likes), user↔artist (follows). Two of those are one-to-many; three are many-to-many. Which is which? And for playlist↔song, is there a fact that belongs on the relationship rather than the song?

Reveal — the full walkthrough

Scope: "Core app, not recommendations. Main actions: play a song, build a playlist, follow an artist, like a song."

Entities + keys: users (user_id), artists (artist_id), albums (album_id), songs (song_id, title, duration_seconds), playlists (playlist_id, user_id).

Relationships, cardinality said aloud:

  • Artist → albums: one artist, many albums — 1:N → albums.artist_id FK.
  • Album → songs: one album, many songs — 1:N → songs.album_id FK.
  • Playlist ↔ songs: many–many → junction playlist_songs. And sort_order goes on the junction (pattern 4) — a song's position is a fact about that pairing.
  • User ↔ songs (likes): M:N → liked_songs(user_id, song_id, liked_at), unique on (user_id, song_id).
  • User ↔ artists (follows): M:N → follows(user_id, artist_id).
users(user_id PK, username, email) artists(artist_id PK, name) albums(album_id PK, artist_id FK, title, release_date) songs(song_id PK, album_id FK, title, duration_seconds) playlists(playlist_id PK, user_id FK, name) playlist_songs(playlist_id FK, song_id FK, sort_order) <- M:N + position liked_songs(user_id FK, song_id FK, liked_at) unique(user_id,song_id) follows(user_id FK, artist_id FK)

Scale gesture: "Operationally 3NF. Play counts would hammer the DB if counted live, so I'd keep a denormalized play_count on songs updated async. For analytics — top songs by region by week — a star schema with fct_play and date/user/song/artist dimensions."

Likely follow-ups & answers: "What about a song appearing on multiple albums (single + compilation)?" → that flips album↔song to M:N, so a song_albums junction. "How do you store the order of a playlist?" → the sort_order you already put on the junction. "Following gets huge?" → index both FK columns, cache follower counts.

Practice prompts — drill these

1 · Design Twitter / X

Same shape as Reddit. Run the script. Hints collapsed; sketch your answer before peeking.

“Design Twitter”
Users post tweets, follow other users, like and retweet, and reply to tweets (threaded). Model the core app.
Hint — where the four patterns show up

Entities: users, tweets. That's nearly it. Now the patterns: a user posts many tweets (1:N). A user follows many users and is followed by many — that's M:N between users and themselves (a follows(follower_id, followee_id) junction). Likes and retweets are M:N user↔tweet junctions. Replies are the threaded case — which pattern? (Self-referencing parent_tweet_id on tweets.) The "follow" relationship being user-to-user is the curveball here.

Reveal — the structure
users(user_id PK, handle, display_name, created_at) tweets(tweet_id PK, user_id FK, body, created_at, parent_tweet_id FK -> tweets) <- self-ref: reply (null = top-level) follows(follower_id FK->users, followee_id FK->users) <- M:N user to user unique(follower_id, followee_id) likes(user_id FK, tweet_id FK, liked_at) unique(user_id, tweet_id) retweets(user_id FK, tweet_id FK, created_at)

Scale gesture: "The home timeline is the hard part at scale — you don't query all followees' tweets live, you fan-out on write into a per-user timeline cache. For analytics, fct_tweet / fct_engagement with date and user dimensions."

The teaching point: the follow relationship is many-to-many from a table to itself — same junction pattern, both FKs point at users. If you saw that, you've internalized the pattern rather than memorizing examples.

2 · Design Uber

“Design Uber”
Riders request trips, drivers accept them, trips have a fare and a rating afterward. Model the core app.
Hint — the entities and the central fact

Entities: users (or split riders/drivers), trips, vehicles. The interesting one is trips — it links a rider and a driver, so it carries two FKs to users (like Twitter's follows, but with trip data attached). A trip has one rider, one driver: that makes trips a 1:N child of each. Ratings — does a rating belong on the trip, or its own table? (Both riders and drivers rate each other, so a small ratings table tied to the trip is cleaner.) Watch the "price at time of trip" detail — store the fare on the trip, don't recompute from current rates.

Reveal — the structure
users(user_id PK, name, role, rating_avg) <- role: rider/driver vehicles(vehicle_id PK, driver_id FK->users, make, model, plate) trips(trip_id PK, rider_id FK->users, driver_id FK->users, vehicle_id FK, start_loc, end_loc, requested_at, started_at, ended_at, fare, status) ratings(rating_id PK, trip_id FK, rater_id FK->users, ratee_id FK->users, stars, comment)

Key details to say: store fare and the locations on the trip (a snapshot — surge pricing means today's rate isn't yesterday's). status models the lifecycle (requested → accepted → in progress → completed); for full history add a trip_status_events table.

Scale gesture: "Live driver locations are high-write, low-history — I'd keep those in a fast store (Redis), not the main trips table. Trips feed a fct_trip star schema for analytics on revenue, utilization, etc."

3 · Design a Library system

“Design a library”
Members borrow books; a book can have multiple physical copies; loans have due dates and return dates. Model it.
Hint — the copy-vs-title distinction is the whole point

The trap and the test: separate the book title (the abstract work — "Dune", one ISBN, one author) from the physical copy (the specific item on the shelf you can actually loan). One title has many copies (1:N). A loan is about a copy, not a title. Authors↔books can be M:N (a book has multiple authors, an author writes multiple books). A loan links a member and a copy with dates — that's the junction-with-attributes pattern.

Reveal — the structure
members(member_id PK, name, email, joined_at) books(book_id PK, title, isbn, publisher) <- the title/work book_copies(copy_id PK, book_id FK, shelf_location, condition) <- 1:N authors(author_id PK, name) book_authors(book_id FK, author_id FK) <- M:N loans(loan_id PK, copy_id FK, member_id FK, borrowed_at, due_date, returned_at) <- returned_at null = still out

The detail that scores: loans reference copy_id, not book_id — you lend a physical copy, and the library needs to know which copy is out. A null returned_at means it's still on loan. Overdue = returned_at IS NULL AND due_date < today.

Follow-up & answer: "How do you handle reservations / holds?" → a reservations(member_id, book_id, requested_at, status) table at the title level (you reserve any copy, not a specific one).

4 · Design an Event Ticketing system

“Design Ticketmaster”
Venues host events; users buy tickets for specific seats; an event can sell many tickets but each seat sells once. Model it.
Hint — the uniqueness constraint is the test

Entities: venues, events, seats, users, tickets. A venue has many events (1:N). A venue has many seats (1:N). The crux: a ticket is a specific seat at a specific event sold to one user — and the same seat at the same event must never sell twice. Where does that uniqueness live? (A unique constraint on (event_id, seat_id) in the tickets table.) That constraint is exactly what they're probing — the favorites-unique-constraint pattern, but load-bearing here.

Reveal — the structure
venues(venue_id PK, name, address) seats(seat_id PK, venue_id FK, section, row, number) <- physical seats events(event_id PK, venue_id FK, name, event_datetime) users(user_id PK, name, email) tickets(ticket_id PK, event_id FK, seat_id FK, user_id FK, price, purchased_at) UNIQUE(event_id, seat_id) <- a seat sells once per event

The detail that scores: the UNIQUE(event_id, seat_id) constraint is the database guaranteeing no double-sold seat — you don't rely on application logic. Store price on the ticket (prices vary by event/tier and change over time).

Follow-up & answer: "Two people try to buy the same seat at once?" → the unique constraint makes one insert fail at the DB level; you'd wrap purchase in a transaction and handle the conflict. "General admission, no assigned seats?" → drop the seat link, track a sold-count against event capacity instead.

Reference

Pre-interview checklist

Say these, every time:

  • Open by clarifying scope (operational vs analytics) + 2 questions.
  • Name entities aloud, give each a surrogate PK.
  • For every pair: "how many of A to how many of B?" → pick the pattern.
  • Every M:N gets a junction table; add a unique constraint where duplicates would be wrong.
  • Put pairing-facts (quantity, sort_order, price-at-time) on the junction.
  • Threaded/nested → self-referencing nullable parent FK; name "adjacency list."
  • Close with a scale/analytics sentence (index FKs, cache counts, star schema).
  • When they ask "what happens when…", match it to a pattern — don't improvise from scratch.
Remember Think out loud the whole time — the interviewer is grading your reasoning, not your silence. A clear wrong turn you catch and explain beats a correct answer delivered mutely. And there's no single right schema; "here are two options and here's when I'd pick each" is the strongest thing you can say.
