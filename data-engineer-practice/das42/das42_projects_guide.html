Guided · fully step-by-step

Project 1 — a sales data mart with dbt + DuckDB (no Airflow)

Hand-held ~2 hours Teaches: ELT, staging→mart, star schema, dbt tests, lineage

Goal: turn raw sales data into a clean star schema you can query, using the exact staging→mart pattern DAS42 uses. You run dbt by hand (no orchestration yet — that's Project 2). Every command and file is given.

Project layout you'll end up with

sales_mart/ ├── warehouse.duckdb ← your local "Snowflake" ├── load_raw.py ← the extract+load script ├── dbt_project.yml ├── profiles.yml └── models/ ├── staging/ │ ├── sources.yml │ ├── stg_orders.sql │ └── stg_customers.sql └── marts/ ├── dim_customer.sql ├── dim_date.sql └── fct_orders.sql
1
Generate raw data into DuckDB

Create load_raw.py. This is your hand-written "Fivetran" — it lands raw tables in a raw schema. We use TPC-H so there's nothing to download.

# load_raw.py
import duckdb

con = duckdb.connect("warehouse.duckdb")
con.execute("INSTALL tpch; LOAD tpch;")
con.execute("CALL dbgen(sf=0.1);")       # generates orders, customer, etc.
con.execute("CREATE SCHEMA IF NOT EXISTS raw;")

# copy the two tables we'll model into our raw schema
con.execute("CREATE OR REPLACE TABLE raw.orders    AS SELECT * FROM orders;")
con.execute("CREATE OR REPLACE TABLE raw.customer  AS SELECT * FROM customer;")

print("orders:", con.execute("SELECT COUNT(*) FROM raw.orders").fetchone()[0])
print("customers:", con.execute("SELECT COUNT(*) FROM raw.customer").fetchone()[0])
con.close()

Run it: python load_raw.py. You should see row counts print. The E and L are done — by hand, exactly what a managed connector automates for you.

2
Configure dbt to talk to DuckDB

Two small files. profiles.yml is the only thing that would change to point at Snowflake instead.

# profiles.yml  (put it in the project folder)
sales_mart:
  target: dev
  outputs:
    dev:
      type: duckdb
      path: warehouse.duckdb
      schema: analytics
# dbt_project.yml
name: 'sales_mart'
profile: 'sales_mart'
version: '1.0'
models:
  sales_mart:
    staging:
      +materialized: view
    marts:
      +materialized: table
Concept: materialization Staging models are view (cheap, always-fresh logical layers); marts are table (physically built for fast reads). This staging-as-view, marts-as-table convention is standard at dbt shops and maps directly to Snowflake object types.

Declare your sources + tests
# models/staging/sources.yml
version: 2
sources:
  - name: raw
    schema: raw
    tables:
      - name: orders
      - name: customer
4
Build the staging models (clean + rename)

Staging models do one job: take a raw source and make it tidy — sensible names, correct types, light filtering. No business logic yet.

-- models/staging/stg_orders.sql
SELECT
  o_orderkey      AS order_id,
  o_custkey       AS customer_id,
  o_orderstatus   AS status,
  o_totalprice    AS order_total,
  o_orderdate     AS order_date
FROM {{ source('raw', 'orders') }}
-- models/staging/stg_customers.sql
SELECT
  c_custkey   AS customer_id,
  c_name      AS customer_name,
  c_mktsegment AS market_segment,
  c_acctbal   AS account_balance
FROM {{ source('raw', 'customer') }}
5
Build the marts (the star schema)

Now the dimensional model: dimensions for context, a fact for the measurable events. Grain of the fact = one row per order.

-- models/marts/dim_customer.sql
SELECT
  customer_id,
  customer_name,
  market_segment,
  account_balance
FROM {{ ref('stg_customers') }}
-- models/marts/dim_date.sql  — a date dimension built from the orders
SELECT DISTINCT
  order_date                       AS date_day,
  EXTRACT(year  FROM order_date) AS year,
  EXTRACT(month FROM order_date) AS month,
  EXTRACT(quarter FROM order_date) AS quarter
FROM {{ ref('stg_orders') }}
-- models/marts/fct_orders.sql  — grain: one row per order
SELECT
  o.order_id,
  o.customer_id,          -- FK to dim_customer
  o.order_date,           -- FK to dim_date
  o.status,
  o.order_total           -- the measure
FROM {{ ref('stg_orders') }} o
6
Add tests, then run the whole thing
# models/marts/marts.yml — tests that run on every build
version: 2
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests: [unique, not_null]
      - name: customer_id
        tests:
          - relationships:                 # FK integrity!
              to: ref('dim_customer')
              field: customer_id
  - name: dim_customer
    columns:
      - name: customer_id
        tests: [unique, not_null]
dbt run      # builds staging (views) then marts (tables), in dependency order
dbt test     # runs unique / not_null / relationships checks
dbt docs generate && dbt docs serve   # open the lineage graph in a browser
7
See your warehouse work

Query the finished mart with a real star-schema question. The duckdb command-line tool is a separate one-line install (brew install duckdb, or download from duckdb.org); if you'd rather not install it, use the Python snippet just below instead.

# option A: the DuckDB CLI (needs the separate `duckdb` binary)
duckdb warehouse.duckdb "
  SELECT c.market_segment, d.year,
         ROUND(SUM(f.order_total),0) AS revenue
  FROM analytics.fct_orders f
  JOIN analytics.dim_customer c USING (customer_id)
  JOIN analytics.dim_date d ON f.order_date = d.date_day
  GROUP BY 1,2 ORDER BY revenue DESC LIMIT 10;
"
# option B: same query, pure Python (only needs the duckdb you already pip-installed)
import duckdb
con = duckdb.connect("warehouse.duckdb")
print(con.execute("""
  SELECT c.market_segment, d.year,
         ROUND(SUM(f.order_total),0) AS revenue
  FROM analytics.fct_orders f
  JOIN analytics.dim_customer c USING (customer_id)
  JOIN analytics.dim_date d ON f.order_date = d.date_day
  GROUP BY 1,2 ORDER BY revenue DESC LIMIT 10
""").fetchall())
What you can now say in the interview "I built an ELT pipeline: a Python load step landing raw data, dbt staging models to clean and rename, then a star schema — dimensions plus a fct_orders fact at one-row-per-order grain. dbt resolved build order via ref(), and I added unique, not_null, and a relationships test enforcing FK integrity between fact and dimension. The lineage graph documents the whole flow." That's a complete, defensible project.

Guided · step-by-step

Project 2 — orchestrate Project 1 with Airflow

Hand-held ~2-3 hours Teaches: DAGs, tasks, dependencies, scheduling, retries

Goal: take Project 1's two steps (load, then dbt) and let Airflow run them on a schedule, in the right order, with automatic retries. This is exactly what Airflow does at DAS42: orchestrate ingestion and dbt. Keep your Project 1 folder; you're just adding a DAG that calls it.

Version note that matters Airflow 3.x (current) moved BashOperator out of core into the standard provider package, and the DAG class now comes from airflow.sdk. The imports below are the modern 3.x ones. If you ever see tutorials using from airflow.operators.bash import BashOperator or from airflow import DAG, that's the older 2.x style. Note this difference in your REAL_STACK_NOTES.md.
1
Install Airflow in standalone mode

Standalone runs the scheduler, the API/UI, and a local executor in one process — perfect for learning. It also creates the dags folder and prints a login.

# pick an AIRFLOW_HOME so everything lives in one place
export AIRFLOW_HOME=~/airflow

# install core + the standard provider (which now holds BashOperator)
pip install "apache-airflow" "apache-airflow-providers-standard"

# first run: spins up the UI at http://localhost:8080 and prints an admin password
airflow standalone

Leave it running. In a browser, open localhost:8080 and log in with the printed credentials. You'll create the DAG file next; Airflow auto-discovers files dropped in ~/airflow/dags/.

2
Write the DAG

Create ~/airflow/dags/sales_pipeline.py. Use absolute paths to your Project 1 folder so Airflow finds the script and dbt project regardless of where it runs from.

# ~/airflow/dags/sales_pipeline.py
from datetime import datetime
from airflow.sdk import DAG                                  # 3.x: DAG comes from the SDK
from airflow.providers.standard.operators.bash import BashOperator  # 3.x path

# EDIT these two to your real absolute paths
PROJECT = "/Users/you/sales_mart"
VENV    = "/Users/you/sales_mart/venv/bin/activate"

with DAG(
    dag_id="sales_pipeline",
    start_date=datetime(2024, 1, 1),
    schedule="@daily",          # run once a day
    catchup=False,               # don't backfill every missed day
    default_args={"retries": 2},   # auto-retry a failed task twice
    tags=["sales", "dbt"],
) as dag:

    extract_load = BashOperator(
        task_id="extract_load",
        bash_command=f"source {VENV} && cd {PROJECT} && python load_raw.py",
    )

    dbt_build = BashOperator(
        task_id="dbt_build",
        # dbt build = run + test in one command; DBT_PROFILES_DIR points at our profiles.yml
        bash_command=f"source {VENV} && cd {PROJECT} && DBT_PROFILES_DIR={PROJECT} dbt build",
    )

    extract_load >> dbt_build      # THE dependency: load must finish before transform
Concept: the dependency operator extract_load >> dbt_build is the whole point. It tells Airflow "don't start the transform until the load succeeds." If extract_load fails, dbt_build never runs — which is exactly the data-integrity guarantee you want in a pipeline.
3
Run it and watch the graph
  • Refresh the UI; within ~30s sales_pipeline appears (Airflow scans the dags folder periodically).
  • Unpause it with the toggle, then hit the play button to trigger a manual run.
  • Open the Graph view: watch extract_load go green, then dbt_build start and go green. That ordering is the DAG working.
  • Click any task → Logs to see the actual stdout from your script and dbt — this is the observability story.
4
Break it on purpose (learn the failure path)

Edit load_raw.py to raise an error (add raise Exception("boom") at the top). Trigger the DAG again. Watch extract_load retry twice (your retries: 2), then go red — and notice dbt_build never runs because its upstream failed. Then fix it and re-run. You've now seen retries, failure propagation, and recovery.

What you can now say "I orchestrated the pipeline as an Airflow DAG — a directed graph of tasks with dependencies. extract_load >> dbt_build guarantees ingestion finishes before transformation, the retry policy handles transient failures, and the UI gives run history and per-task logs for monitoring." That single explanation covers most orchestration questions an interviewer will ask.
Real-stack difference to log Locally you run airflow standalone (one process, SQLite, LocalExecutor). In production / at DAS42, Airflow runs a separate scheduler + workers on Postgres, often in Docker or Kubernetes, with connections and secrets managed centrally. The DAG code is identical; only the deployment differs.

Rough edges · you build it (no full code)

Project 3 — a daily weather pipeline (Airflow + incremental dbt)

No copy-paste code ~1 day Teaches: API ingestion, incremental models, idempotency

Goal: build a pipeline that pulls weather readings from a free API every day, lands them in DuckDB, and transforms them with an incremental dbt model so reruns only process new data. You get the spec and hints — not the code. Messy Python is fine. The dataset is trivial to model so you can focus on the engineering.

Why weather, and why it's easy to model

The Open-Meteo API is free, needs no API key, and returns hourly temperature for a location. Your grain is dead simple: one row per city per hour. No tricky relationships — all your effort goes into ingestion and incremental logic, which is the point.

The spec (what to build, not how)

  • A Python script that calls Open-Meteo for 3-4 cities and appends the readings to a raw.weather table in DuckDB (don't overwrite — append, so history accumulates).
  • A dbt staging model that cleans/types the raw readings.
  • A dbt incremental model fct_weather that, on each run, only inserts readings newer than what it already has.
  • An Airflow DAG that runs the ingestion then the dbt build, on an @hourly or @daily schedule.
  • At least two dbt tests (e.g. not_null on the timestamp, a freshness or range check on temperature).

Hints to get unstuck (not solutions)

You'll wonder…Hint
How do I call the API?It's a GET request; requests.get(url).json(). The endpoint is api.open-meteo.com/v1/forecast with latitude, longitude, and hourly=temperature_2m params. Look at the JSON shape before modeling it.
How does an incremental model work?Wrap the new-rows filter in {% if is_incremental() %} WHERE reading_time > (SELECT MAX(reading_time) FROM {{ this }}) {% endif %}. {{ this }} is the model's own existing table.
Config for incremental?Top of the model: {{ config(materialized='incremental', unique_key='...') }}.
How do I avoid duplicate rows on rerun?That's exactly what unique_key + the incremental filter solve. Test it by running twice and checking the count doesn't double.

The lesson hiding in here

Idempotency The real skill is making the pipeline idempotent — running it twice produces the same correct result, not duplicates. Getting the incremental filter and unique_key right is the single most transferable production-pipeline concept, and you'll only truly get it by fighting through it once here.
The interview payoff When asked "walk me through a pipeline you built," you'll describe real API ingestion, an incremental model that handles new data without reprocessing history, and Airflow scheduling — with specific gotchas you hit (duplicate rows, a timestamp comparison that didn't work at first). That specificity is impossible to fake.

Project 4 — MovieLens star schema (Airflow-orchestrated)

No copy-paste code ~1 day Teaches: dimensional modeling, multi-step DAG, reconciliation

Goal: take the small MovieLens dataset (movies + ratings + users), build a clean star schema with dbt, and orchestrate the whole load-and-transform with Airflow — plus a reconciliation check. Again: spec and hints only. MovieLens is a classic teaching set precisely because the model is obvious.

The dataset

Grab MovieLens "ml-latest-small" from grouplens.org (~1MB zip). It has movies.csv (movieId, title, genres), ratings.csv (userId, movieId, rating, timestamp), and tags.csv. The star schema almost designs itself.

The spec

  • A Python step that reads the CSVs and loads them into raw.movies, raw.ratings in DuckDB.
  • Staging models that clean each (split genres, cast the timestamp to a real datetime).
  • A star schema: fct_rating at grain one row per user per movie rating, with dim_movie and dim_date.
  • A relationships test enforcing every rating's movie_id exists in dim_movie.
  • A reconciliation model or test: raw ratings row count must equal fct_rating row count, and average rating should match between raw and fact. If they differ, you dropped or duplicated rows somewhere.
  • An Airflow DAG chaining load → dbt build → (optionally) a final reconciliation task that fails loudly if counts mismatch.

Hints

ChallengeHint
Defining the fact grainState it in one sentence before writing SQL: "one row per (userId, movieId, timestamp)." That sentence prevents double-counting and tells you the FKs.
Genres are a pipe-delimited stringDecide: keep as-is in dim_movie, or split into a bridge table (movie↔genre many-to-many). Either is defensible — say which and why.
Reconciliation that fails the DAGWrite a dbt test (or a tiny SQL check in a final BashOperator) comparing counts; a non-zero exit makes Airflow mark the task failed.
The lesson This is the closest project to the actual DAS42 work: messy source files → clean dimensional model → validated against the source. The reconciliation step is the "source-to-target validation" bullet on your resume, made real. Expect your counts to mismatch the first time — finding out why is the entire point.

Level up

Project 5 — re-do Projects 1 & 2 on real Snowflake

Partly guided ~half a day Teaches: what's actually different on the real platform

Goal: take your working sales mart from Projects 1-2 and point it at a real Snowflake account. The dbt models barely change — the value is feeling the differences (staging/COPY INTO, warehouses, the cost meter) so you can speak to them with first-hand authority. Read the Snowflake guide alongside this.

1
Get a free Snowflake trial

Snowflake offers a 30-day trial with $400 in credits and no credit card. Sign up at signup.snowflake.com, choose any cloud/region, and pick Enterprise edition (it includes longer Time Travel and multi-cluster, useful to see). Critically: set your warehouse AUTO_SUSPEND to 60s and keep it XS so $400 lasts the whole month.

Protect your credits Credits burn only while a warehouse runs. Use an XS warehouse, aggressive auto-suspend, and don't leave queries looping. Light learning easily fits inside 30 days of the $400 balance.
2
Create the objects + load data the Snowflake way

This is the part that's genuinely different from DuckDB. Instead of generating data in-process, you stage a file and COPY INTO.

-- run in a Snowflake worksheet
CREATE WAREHOUSE IF NOT EXISTS learn_wh
  WAREHOUSE_SIZE='XSMALL' AUTO_SUSPEND=60 AUTO_RESUME=TRUE;
CREATE DATABASE IF NOT EXISTS sales_mart;
CREATE SCHEMA IF NOT EXISTS sales_mart.raw;
USE WAREHOUSE learn_wh;
USE SCHEMA sales_mart.raw;

-- export your DuckDB raw tables to CSV first (locally):
--   COPY raw.orders   TO 'orders.csv'   (HEADER, DELIMITER ',');
--   COPY raw.customer TO 'customer.csv' (HEADER, DELIMITER ',');

CREATE OR REPLACE FILE FORMAT csv_ff
  TYPE=csv SKIP_HEADER=1 FIELD_OPTIONALLY_ENCLOSED_BY='"';
CREATE OR REPLACE STAGE load_stage FILE_FORMAT=csv_ff;

-- from SnowSQL CLI (or the Snowsight upload UI): push the files to the stage
--   PUT file://orders.csv   @load_stage;
--   PUT file://customer.csv @load_stage;

CREATE OR REPLACE TABLE orders(
  o_orderkey NUMBER, o_custkey NUMBER, o_orderstatus STRING,
  o_totalprice NUMBER(18,2), o_orderdate DATE);
COPY INTO orders FROM @load_stage/orders.csv ON_ERROR='ABORT_STATEMENT';
-- repeat the CREATE TABLE + COPY INTO for customer
3
Point dbt at Snowflake — the only project change

Install the Snowflake adapter and swap your profile. Your models/, tests, and sources don't change at all. That's the whole lesson: the transformation layer is portable.

pip install dbt-snowflake

4
Explore what DuckDB couldn't do

Now play with the Snowflake-only features from the Snowflake guide, against your own data:

  • Run SELECT COUNT(*) FROM fct_orders AT(OFFSET => -300); — Time Travel.
  • CREATE DATABASE sales_dev CLONE sales_mart; — an instant zero-copy clone; check it took seconds and no extra storage.
  • Open Query History in Snowsight and look at a query profile — see partitions scanned vs pruned.
  • Watch your credit balance tick in the UI as the warehouse runs, then drop to zero usage after auto-suspend.
The payoff — now your resume is fully authentic After this you can say, truthfully: "I've run the same dbt project against both DuckDB and Snowflake. On Snowflake I loaded via stages and COPY INTO, watched micro-partition pruning in the query profile, used zero-copy cloning to make a safe test copy, and managed warehouse auto-suspend for cost. The transformation layer was identical — only the profile and the loading mechanism changed." That's the migration narrative, lived.

After the five

Where to go next

If you've done all five, you've touched every concept the DAS42 technical round tests. If you want more reps, these extend naturally without new modeling headaches.

  • Add a BI layer. Point Metabase (free, runs in Docker) at your DuckDB or Snowflake marts and build a dashboard — closes the loop to the "BI tool" part of their stack.
  • Add CI. Put your dbt project in a GitHub repo and add a GitHub Action that runs dbt build on every push — the CI/CD bullet, made real.
  • Add a snapshot. Use dbt snapshots to track slowly-changing dimensions (e.g. a customer's market segment over time) — ties back to the SCD Type 2 concept in the study guide.
  • Volume test. Re-run Project 3 or 4 on a full NYC Taxi month (millions of rows) to feel where incremental models and pruning actually earn their keep.
Keep the differences log current By now your REAL_STACK_NOTES.md should have a dozen DuckDB-vs-Snowflake and standalone-vs-production-Airflow notes. Skim it the night before the interview — it's the most honest, specific prep you can bring, and it directly answers "how does your local practice translate to our stack?"
