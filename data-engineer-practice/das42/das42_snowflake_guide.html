Foundations

The three-layer architecture

This is the one Snowflake concept you must be able to explain in plain words. Everything else follows from it.

Separation of storage and compute core idea

Snowflake splits the system into three independent layers. The key consequence: your data (storage) and the engines that query it (compute) scale separately and don't fight each other. Ten teams can query the same data with ten separate compute clusters, none slowing the others.

LayerIn plain wordsWhat it does
Cloud servicesThe brainAuthentication, query planning/optimization, metadata, transactions. You never manage it.
Compute (warehouses)The muscleVirtual warehouses execute queries. You start, size, and stop them; you pay while they run.
StorageThe filing cabinetYour data, held as compressed columnar micro-partitions in cloud object storage. Cheap and effectively unlimited.
Say it like this "Snowflake's defining feature is the separation of storage and compute. Storage sits in cloud object storage as columnar micro-partitions; compute is virtual warehouses you spin up independently. That's why a heavy load job and a BI dashboard can run on separate warehouses against the same data without contending for resources — and why you scale each for its own need."

Contrast that lands well In traditional databases (and SQL Server, where your data came from), storage and compute are coupled on the same box — so a big query can starve everything else, and scaling means a bigger server. Naming that contrast shows you understand why the migration was worth doing.

Virtual warehouses

Virtual warehouse memorize

A virtual warehouse is a cluster of compute resources (CPU + memory) that executes your queries and data loads. It is not where data lives — it's the engine that processes it. Named by T-shirt size (XS, S, M, L…), each step up roughly doubles both power and credit cost. It can auto-suspend when idle and auto-resume on the next query.

-- create, size, and configure a warehouse
CREATE WAREHOUSE etl_wh
  WAREHOUSE_SIZE = 'SMALL'
  AUTO_SUSPEND = 60      -- pause after 60s idle (stops the cost meter)
  AUTO_RESUME = TRUE;   -- wake automatically on next query

USE WAREHOUSE etl_wh;   -- pick which engine runs your session
Scale up vs scale out memorize

Scale up = use a bigger warehouse so one heavy query finishes faster (more memory/CPU). Scale out = add clusters (multi-cluster warehouse) so many concurrent users each get resources, avoiding queuing. Up is for query size; out is for user concurrency.

Worded example (fund analytics) "The nightly job that rebuilds all fund-holding aggregates is one big query, so I'd scale up to a Medium warehouse for it. But the analysts hitting dashboards all morning are a concurrency problem, so I'd scale out with a small multi-cluster warehouse that adds clusters under load and shrinks after."

Storage & micro-partitions

Micro-partition memorize

Snowflake automatically chops every table into compressed, columnar chunks of ~50–500MB called micro-partitions. You don't manage them. Each one stores per-column metadata — crucially the min and max value of every column in that chunk. That metadata is what makes Snowflake fast.

Columnar storage

Data is stored column-by-column, not row-by-row. So a query that needs only market_value reads just that column's data, skipping the rest. This is why SELECT * is wasteful on Snowflake and selecting only needed columns is a real optimization.

Worded example "If I query holdings WHERE as_of_date = '2025-06-01', Snowflake checks each micro-partition's min/max for as_of_date and skips every chunk whose range can't contain that date. On a table with years of history, it might read a handful of partitions instead of all of them — that's pruning, and it falls out of the micro-partition design for free."

Working with data

Databases, schemas, tables

The object hierarchy is conventional — the same mental model as SQL Server, with Snowflake-specific table flavors worth knowing.

Object hierarchy

Account → Database → Schema → Table/View. You fully-qualify objects as database.schema.table. A common convention is a RAW schema for loaded source data and an ANALYTICS schema for dbt-built models.

CREATE DATABASE trading;
CREATE SCHEMA trading.raw;
CREATE SCHEMA trading.analytics;

CREATE TABLE trading.raw.trades (
  trade_id    NUMBER,
  fund_id     NUMBER,
  security_id NUMBER,
  price       NUMBER(18,4),
  quantity    NUMBER,
  executed_at TIMESTAMP_NTZ
);
Table types worth knowing

Permanent (default) — full Time Travel + fail-safe. Transient — no fail-safe, cheaper, good for staging/intermediate data you can rebuild. Temporary — exists only for your session. External — queries files sitting in cloud storage without loading them in.

Practical note Staging and intermediate ELT tables are often made TRANSIENT — you don't need fail-safe recovery on data you can simply re-derive, and it saves storage cost. A small detail that signals real-world awareness.

Loading data: stages & COPY INTO

This is how a migration actually moves data. Be ready to narrate it.

Stage memorize

A stage is a location where data files live before loading. Internal stages are managed by Snowflake; external stages point at your own cloud bucket (S3, GCS, Azure Blob). You stage files, then bulk-load them into tables.

COPY INTO memorize

The bulk-load command. It reads files from a stage and inserts them into a table, applying a file format (CSV, Parquet, JSON). It's idempotent-aware (tracks loaded files) and handles errors per your settings.

-- 1) define how files are shaped
CREATE FILE FORMAT my_csv
  TYPE = csv FIELD_OPTIONALLY_ENCLOSED_BY = '"' SKIP_HEADER = 1;

-- 2) a place for files to land
CREATE STAGE trade_stage FILE_FORMAT = my_csv;

-- 3) put a local file into the stage (run from SnowSQL CLI)
PUT file://./trades.csv @trade_stage;

-- 4) bulk-load into the raw table
COPY INTO trading.raw.trades
  FROM @trade_stage/trades.csv
  ON_ERROR = 'ABORT_STATEMENT';   -- or CONTINUE to skip bad rows
The migration narration "We extracted SQL Server tables to Parquet, landed them in an external stage on S3, and used COPY INTO to bulk-load raw Snowflake tables — that's the E and L. Then dbt handled the T, building cleaned and modeled tables on top. Keeping raw data intact meant any transformation bug was just a dbt rerun, not a re-extract."

Semi-structured data: VARIANT

VARIANT memorize

A column type that stores semi-structured data (JSON, Avro, Parquet) natively — no predefined columns needed. You query nested fields with : for paths and :: for casts. This matters because source data (API payloads, vendor feeds) often arrives as JSON, and the ELT way is to land it raw then shred it in-warehouse.

-- raw vendor payload landed as JSON in a VARIANT column
SELECT
  payload:fund_id::NUMBER            AS fund_id,
  payload:strategy::STRING          AS strategy,
  holding.value:ticker::STRING     AS ticker,
  holding.value:shares::NUMBER      AS shares
FROM trading.raw.fund_payloads,
     LATERAL FLATTEN(input => payload:holdings) holding;   -- expand a JSON array into rows
FLATTEN

A table function that turns a nested array inside a VARIANT into one row per element — the standard way to normalize JSON arrays into relational rows. Above, a fund's holdings array becomes one row per holding.

Views & materialized views

View

A saved query. It stores no data — it re-runs against the underlying tables each time you select from it. dbt models materialized as view are exactly this.

Materialized view

A view whose results are physically stored and kept up to date automatically as base data changes. Faster to read, but costs storage and maintenance compute. Use for expensive aggregations queried often; skip for data that changes constantly.

dbt connection In dbt you choose this per model with +materialized: view or table (and incremental for big append-only data). So "materialization" in dbt maps directly to these Snowflake object choices — a clean thing to say if asked how dbt and Snowflake fit together.

The standout features

Time Travel

Time Travel memorize

Snowflake retains previous versions of your data for a retention window (1 day on Standard, up to 90 on Enterprise). You can query a table as it existed at a past time, or restore tables/schemas/databases you dropped. It exists because storage is cheap and versioned under the hood.

-- query the table as it was 1 hour ago (did a bad load corrupt it?)
SELECT COUNT(*) FROM trades AT(OFFSET => -3600);

-- or as of a specific statement, or timestamp
SELECT * FROM trades BEFORE(STATEMENT => '01a2...');

-- recover a table dropped by mistake
UNDROP TABLE trades;
Why a data engineer loves this "During a migration, Time Travel is a safety net: if a load goes wrong I can compare current vs prior state, diff row counts against an hour ago, or undrop something deleted by accident — without restoring from a backup. It turns 'oh no' into a one-line query."

Zero-copy cloning

Zero-copy clone memorize

Instantly create a copy of a table, schema, or whole database that shares the same underlying storage until you change it (copy-on-write). The clone is created in seconds and costs no extra storage until modified — only the changed micro-partitions diverge.

-- spin up a full test copy of production, instantly and free
CREATE DATABASE trading_dev CLONE trading;

-- now run risky migration changes against trading_dev safely
Worded example (migration testing) "Before promoting new dbt models, I'd zero-copy clone the production database to a dev copy, run the models there, validate the numbers, and only then ship to prod. The clone is instant and costs nothing until I modify it — so I get a true prod-scale test environment without doubling storage."

Streams & tasks (change tracking + scheduling)

Stream

An object that tracks row-level changes (inserts/updates/deletes) on a table since you last read it — change data capture (CDC), built in. You consume the changes and the stream advances. The basis for incremental processing inside Snowflake.

Task

A scheduled piece of SQL (or a chain of them) that Snowflake runs on a cron or interval. Combined with streams, you get native "when new data arrives, transform it" pipelines without an external scheduler.

-- a stream captures new rows landing in raw.trades
CREATE STREAM trades_stream ON TABLE trading.raw.trades;

-- a task processes only the new rows, on a schedule
CREATE TASK load_new_trades
  WAREHOUSE = etl_wh
  SCHEDULE = '5 MINUTE'
AS
  INSERT INTO trading.analytics.fct_trade
  SELECT ... FROM trades_stream;   -- only the unconsumed changes
Competence note, not mastery You don't need to build streams/tasks from memory. Just know what they're for: streams = built-in change capture, tasks = built-in scheduling. At DAS42, Airflow usually orchestrates dbt instead — but knowing Snowflake can do lightweight scheduling natively is the kind of context that reads as "I understand the platform."

Secure data sharing

Secure data sharing

Share live data with another Snowflake account without copying or moving it — the consumer queries your data directly (using their own compute) and always sees the latest. Powered by the storage/compute split. The Marketplace is this concept productized.

Why it's worth a sentence "Because storage is separate from compute, Snowflake can grant another account read access to the same micro-partitions — so a client or partner queries fresh data with their own warehouse, no ETL, no stale copies. It's a frequent selling point in consulting engagements." A clean DAS42-flavored point.

Performance & cost

Pruning & clustering

Partition pruning memorize

Using micro-partition min/max metadata to skip chunks that can't match a query's filter, so Snowflake scans far less data. It happens automatically. Your job is to write queries (and lay out data) so pruning can work — mainly by filtering on columns whose values are clustered together in storage.

Clustering key

For very large tables, you can define a clustering key so Snowflake co-locates rows with similar values in the same micro-partitions, improving pruning on that column. Usually only needed on multi-terabyte tables; small/medium tables prune fine without it.

-- cluster a huge holdings table by date so date-range queries prune well
ALTER TABLE trading.analytics.fct_holding
  CLUSTER BY (as_of_date);
Don't over-claim Clustering keys are a tuning lever for big tables, not a default. Saying "I'd add a clustering key only after confirming a large table prunes poorly on its main filter column" shows judgment. Adding them everywhere is a known anti-pattern (they cost maintenance credits).

The cost model

Credits memorize

Compute is billed in credits, consumed per-second while a warehouse runs, scaled by size (XS≈1 credit/hr, S≈2, M≈4, doubling up). Storage is billed separately by the terabyte. The headline: you pay for compute time, so idle warehouses and oversized warehouses both waste money.

The levers that control cost (and what to say)

  • Auto-suspend — stop the meter the moment work ends. Short suspend = less waste.
  • Right-sizing — match warehouse size to the workload; don't run an L for a query an S handles.
  • Workload separation — separate warehouses for ETL vs BI so one doesn't bloat the other's cost, and so you can attribute spend.
  • Scan less — pruning + selecting only needed columns means less data read, which is less time, which is fewer credits.
The consulting-grade answer "At a consultancy, cost is a deliverable. I'd right-size warehouses, set aggressive auto-suspend, separate ETL compute from analyst-facing compute so a load job can't inflate dashboard cost, and watch the query profile for tables that aren't pruning. The lever is always 'minimize warehouse time and data scanned.'"

Caching

Result cache

If you run the exact same query and the underlying data hasn't changed, Snowflake returns the stored result instantly — no warehouse needed, no credits used. Lasts ~24 hours.

Warehouse (local) cache

A running warehouse caches recently read micro-partitions in its local SSD, so repeat queries over the same data are faster. Suspending the warehouse clears it — a reason not to set suspend too aggressively if a workload is bursty.

The nuance to mention Very short auto-suspend saves idle cost but throws away the warehouse cache, so the next query is cold. There's a small tradeoff between cost (suspend fast) and warm-cache performance (stay up). Naming that tradeoff is exactly the "competent, thinking engineer" signal.

In context

How Snowflake and dbt fit together

At DAS42 these two are joined at the hip, so be able to describe the division of labor cleanly.

dbt doesSnowflake does
Defines transformations as SQL modelsStores the data and runs the SQL
Manages model dependencies (the DAG)Provides the compute (warehouses) that executes each model
Chooses materialization (view/table/incremental)Creates the actual view/table object
Runs tests (not_null, unique, relationships)Executes those tests as SQL queries
One-sentence version "dbt is the transformation logic and orchestration of what to build and in what order; Snowflake is the engine that stores the data and runs the SQL dbt generates. dbt compiles a model into a CREATE TABLE AS or CREATE VIEW that a Snowflake warehouse executes."
